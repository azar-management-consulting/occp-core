Alembic Migrations: Managing Schema Evolution in AI Systems

AI platforms that maintain audit trails, policy evaluations, and agent registrations generate complex relational data. When the schema evolves — and it always does — you need migration infrastructure that handles async engines, dual backends, and zero-downtime upgrades. Here is how OCCP uses Alembic to manage schema evolution across SQLite development and PostgreSQL production.

Why Schema Migration Matters for AI Platforms

Most AI orchestration tools store data in flat files or ephemeral memory. OCCP takes a fundamentally different approach: every pipeline execution, every policy evaluation, every audit entry is persisted in a relational database. This data is not optional — it is the evidence chain that proves your AI system operated within defined boundaries. When regulations like the EU AI Act require record-keeping for high-risk AI systems, that data must survive schema changes without corruption or loss.

The challenge is that AI platforms evolve rapidly. In four months, OCCP went from a simple task table to a schema with users, agents, capabilities, audit entries, policy evaluations, settings, skills, and MCP connections. Each version added tables, modified columns, and introduced new relationships. Without migration infrastructure, each schema change would require manual database recreation and data loss.

Schema migration is not a database concern. It is a compliance concern. If your AI platform cannot evolve its schema without losing audit data, you cannot satisfy Article 12 of the EU AI Act.

Alembic Configuration for Async Engines

OCCP uses SQLAlchemy 2.0 with async mapped models. Alembic, SQLAlchemy's migration tool, supports async engines through its run_async configuration. The key configuration lives in alembic/env.py, where we configure the migration environment to work with both synchronous SQLite (development) and asynchronous PostgreSQL (production).

# alembic/env.py — async engine configuration
from sqlalchemy.ext.asyncio import create_async_engine
from alembic import context
from occp.database import Base

target_metadata = Base.metadata

def run_migrations_offline():
    """Run migrations in 'offline' mode — generates SQL scripts."""
    url = config.get_main_option("sqlalchemy.url")
    context.configure(
        url=url,
        target_metadata=target_metadata,
        literal_binds=True,
        dialect_opts={"paramstyle": "named"},
    )
    with context.begin_transaction():
        context.run_migrations()

async def run_async_migrations():
    """Run migrations with an async engine."""
    connectable = create_async_engine(
        config.get_main_option("sqlalchemy.url"),
        poolclass=pool.NullPool,
    )
    async with connectable.connect() as connection:
        await connection.run_sync(do_run_migrations)
    await connectable.dispose()

def do_run_migrations(connection):
    context.configure(
        connection=connection,
        target_metadata=target_metadata,
    )
    with context.begin_transaction():
        context.run_migrations()

The dual-mode configuration is critical. In development, we use sqlite+aiosqlite:///./occp.db for fast local iteration. In production, postgresql+asyncpg://... connects to the managed PostgreSQL instance. Alembic handles the dialect differences transparently, though some operations — like ALTER COLUMN — require dialect-specific handling that we encode in the migration files themselves.

Migration File Structure

Each migration file in migrations/versions/ follows a strict structure: a revision ID, a reference to the parent revision, and two functions — upgrade() and downgrade(). The revision ID is a short hex string generated by Alembic, and the parent reference creates a linked list of migrations that must be applied in order.

# migrations/versions/a3f8b2c1d4e5_add_agent_capabilities.py
"""Add agent capabilities table

Revision ID: a3f8b2c1d4e5
Revises: 9c7d6e5f4a3b
Create Date: 2026-02-10 14:30:00.000000
"""
from alembic import op
import sqlalchemy as sa

revision = 'a3f8b2c1d4e5'
down_revision = '9c7d6e5f4a3b'
branch_labels = None
depends_on = None

def upgrade():
    op.create_table(
        'agent_capabilities',
        sa.Column('id', sa.Integer(), primary_key=True),
        sa.Column('agent_id', sa.Integer(), sa.ForeignKey('agents.id')),
        sa.Column('capability', sa.String(100), nullable=False),
        sa.Column('granted_at', sa.DateTime(), server_default=sa.func.now()),
        sa.Column('granted_by', sa.Integer(), sa.ForeignKey('users.id')),
    )
    op.create_index('ix_agent_capabilities_agent_id', 'agent_capabilities', ['agent_id'])

def downgrade():
    op.drop_index('ix_agent_capabilities_agent_id')
    op.drop_table('agent_capabilities')

Auto-Generating Migrations from Models

Alembic can compare the current database state against the SQLAlchemy model definitions and auto-generate the migration diff. This is invoked with alembic revision --autogenerate -m "description". The autogenerate feature inspects Base.metadata, compares it to the database schema, and produces upgrade() and downgrade() functions that reconcile the differences.

We do not blindly trust autogenerated migrations. Every generated file is reviewed for correctness, especially around these areas:

  • Column type changes — SQLite does not support ALTER COLUMN TYPE. We use batch operations (op.batch_alter_table) for SQLite compatibility.
  • Default values — Server defaults versus client defaults have different behaviors across dialects. We enforce server defaults for audit timestamps.
  • Foreign key constraints — SQLite foreign keys require PRAGMA foreign_keys = ON. Our test fixtures enable this explicitly.
  • Index names — Autogenerated index names can conflict across backends. We use explicit naming conventions.

Testing Migrations in CI

Every pull request that includes a migration file triggers a dedicated CI job that applies the full migration chain from scratch on a clean SQLite database. The job runs alembic upgrade head followed by alembic downgrade base followed by alembic upgrade head again. This round-trip test catches irreversible migrations, missing downgrade logic, and ordering conflicts.

# .github/workflows/ci.yml — migration test step
- name: Test migrations
  run: |
    alembic upgrade head
    alembic downgrade base
    alembic upgrade head
    python -c "from occp.database import Base; print(f'Tables: {len(Base.metadata.tables)}')"

The final Python check verifies that the model metadata matches the migrated schema. If a migration creates a table that is not reflected in the models, or if a model references a table that no migration creates, the check fails. This keeps models and migrations in lockstep across all three Python versions we test against: 3.11, 3.12, and 3.13.

Rollback Strategies

Production rollback is the scenario nobody wants but everyone must plan for. Our rollback strategy operates at three levels:

  1. Alembic downgrade — Each migration has a downgrade() function. For additive changes (new tables, new columns with defaults), downgrade is safe. For destructive changes (column removal, type changes), the downgrade function must restore the original schema without data loss.
  2. Database snapshots — Before any production migration, the deployment script creates a PostgreSQL snapshot. If the migration fails or the application behaves incorrectly after migration, we restore the snapshot and redeploy the previous version.
  3. Feature flags — New columns that drive new features are read behind feature flags. The migration can be applied in advance without the application code referencing the new columns until the feature is enabled.

The Additive-Only Rule

We follow a strict rule: migrations should be additive whenever possible. Adding a new table is always safe. Adding a nullable column with a default is always safe. Renaming a column or changing a type is never safe without a multi-step migration: add the new column, backfill data, update the application to use the new column, then drop the old column in a subsequent migration.

Never remove a column in the same release that stops writing to it. The rollback window must include at least one release where both columns exist.

Dual Backend Considerations

Running SQLite in development and PostgreSQL in production introduces dialect-specific challenges. SQLite lacks support for several PostgreSQL features: ALTER COLUMN, concurrent index creation, and schema-qualified table names. Our migration files handle this with dialect-aware branching:

def upgrade():
    bind = op.get_bind()
    if bind.dialect.name == 'sqlite':
        with op.batch_alter_table('audit_entries') as batch_op:
            batch_op.add_column(sa.Column('retention_days', sa.Integer(), default=2555))
    else:
        op.add_column('audit_entries',
            sa.Column('retention_days', sa.Integer(), server_default='2555'))

The batch_alter_table context manager is essential for SQLite. It creates a temporary table with the new schema, copies data from the original table, drops the original, and renames the temporary table. This is transparent to the migration author but critical for correctness on SQLite, where ALTER TABLE support is limited to adding columns and renaming tables.

Migration as Compliance Infrastructure

For an AI governance platform, migrations are not merely a development convenience. They are compliance infrastructure. The audit_entries table, for example, must retain records for seven years under EU AI Act Article 19. A migration that alters this table must preserve every existing record with its original timestamps and hash chain references. Our migration review checklist includes a specific item: "Does this migration preserve audit trail integrity?" If the answer is no, the migration is rejected regardless of its technical correctness.

The migration chain itself serves as a record of schema evolution. Each migration file is committed to the repository with a descriptive message. The revision history forms a timeline of how the platform's data model evolved, providing the kind of technical documentation that auditors require when evaluating AI system record-keeping practices.

// Key Takeaways

  • Alembic async configuration enables migrations against both SQLite (dev) and PostgreSQL (prod) backends
  • Auto-generated migrations are reviewed for dialect-specific issues: batch operations for SQLite, server defaults for PostgreSQL
  • CI runs a full upgrade-downgrade-upgrade cycle to catch irreversible migrations
  • Additive-only rule: never remove a column in the same release that stops writing to it
  • Migration infrastructure is compliance infrastructure — audit data must survive every schema change
