10-Step Wizard: Designing Zero-Friction AI Platform Onboarding

The gap between "I installed this" and "I am using this" is where most infrastructure software loses its users. OCCP's 10-step onboarding wizard bridges that gap by guiding developers from the landing page to their first autonomous, governed task — without treating them like beginners.

The Design Constraint

OCCP is developer infrastructure, not a consumer application. The target user knows what API keys are, understands what agents do, and does not need tooltips explaining what "JSON" means. The wizard's design constraint is respect: guide the user through the required configuration sequence without condescending, without over-explaining, and without hiding complexity behind abstractions that will confuse them later.

The wizard is not a tutorial. It does not teach concepts. It produces a working system. When a user completes step 10, they have a registered agent executing governed tasks through the Verified Autonomy Pipeline. They have not "learned about" OCCP — they have configured and operated OCCP.

Step-by-Step Architecture

Step 1: Landing Page

The landing page presents the platform's value proposition in three lines: what OCCP is (an AI orchestration control plane), what it does (governs agent execution through a verified pipeline), and why it matters (audit trails, policy enforcement, compliance). A single "Get Started" button initiates the wizard. No registration wall — the user sees the platform's purpose before creating an account.

Step 2: Account Creation

Username, email, password. Password strength validation enforces minimum entropy without arbitrary character class rules. The form submits to POST /api/v1/auth/register, which creates the user and returns a JWT token. The wizard stores the token and advances to step 3 automatically. No email verification at this stage — that is a production configuration, not an onboarding blocker.

Step 3: LLM Provider Configuration

A selector presents supported providers: Anthropic (Claude), OpenAI (GPT-4), Google (Gemini), and a "Local Model" option for self-hosted inference. Each provider card shows the models available, pricing tier, and capability summary. The selection is stored in the user's settings and determines which API key format is required in step 4.

// Wizard step 3 — provider configuration component
const providers = [
  { id: "anthropic", name: "Anthropic", models: ["claude-3-opus", "claude-3-sonnet"] },
  { id: "openai", name: "OpenAI", models: ["gpt-4-turbo", "gpt-4o"] },
  { id: "google", name: "Google", models: ["gemini-1.5-pro", "gemini-1.5-flash"] },
  { id: "local", name: "Local Model", models: ["ollama", "vllm", "custom"] },
];

function ProviderStep({ onSelect }) {
  return (
    <div className="provider-grid">
      {providers.map(p => (
        <ProviderCard key={p.id} provider={p} onSelect={onSelect} />
      ))}
    </div>
  );
}

Step 4: API Key Entry

The API key input validates format against the selected provider. Anthropic keys start with sk-ant-, OpenAI keys with sk-, Google keys have a different format. After format validation, the wizard makes a lightweight API call to the provider to verify the key works — a simple model list request that costs nothing but confirms connectivity. The key is encrypted with AES-256-GCM before storage and is never displayed again after submission.

Step 5: Register First Agent

Agent registration requires a name and a description. The wizard provides sensible defaults ("My First Agent" / "General-purpose assistant for testing the OCCP pipeline") but encourages customization. The registration creates an agent record in the database with a unique identifier, status set to "active", and creation timestamp. The agent is assigned to the user who created it.

Step 6: Define Agent Capabilities

This step presents available skills as a checkbox grid. Each skill shows its name, description, required tools, and permission scope. The wizard pre-selects a minimal skill set appropriate for a first-time setup: basic text generation and code analysis. Advanced skills (repository management, database access, deployment automation) are available but not pre-selected. The selected skills are validated against the agent's tool access — if a skill requires a tool that is not connected, the wizard flags the dependency.

// Prerequisite validation for skill selection
function validateSkillSelection(selectedSkills, connectedTools) {
  const issues = [];
  for (const skill of selectedSkills) {
    for (const requiredTool of skill.required_tools) {
      if (!connectedTools.includes(requiredTool)) {
        issues.push({
          skill: skill.name,
          missing_tool: requiredTool,
          message: `Skill "${skill.name}" requires "${requiredTool}" — connect it in Settings > Tools`
        });
      }
    }
  }
  return { valid: issues.length === 0, issues };
}

Step 7: Create First Task

The task creation form requires a title and description. The wizard provides a suggested task: "Analyze the OCCP README and summarize the platform's key features." This task is designed to exercise the pipeline without requiring external tool access — it uses only the LLM provider configured in steps 3-4. The task is assigned to the agent created in step 5 and enters the pipeline in "pending" status.

Step 8: Pipeline Execution

This is the wizard's centerpiece. Clicking "Execute" triggers the full Verified Autonomy Pipeline. The wizard displays a real-time visualization of the five stages:

  1. Plan — The agent generates a task plan with step breakdown and risk assessment. The plan appears in the wizard as a collapsible JSON document.
  2. Gate — The policy engine evaluates the plan. PII detection, injection scanning, and resource limit checks run against the plan content. A green checkmark appears for each passed check.
  3. Execute — The LLM processes the task within the sandboxed environment. Execution output streams into the wizard in real time.
  4. Validate — Output validation verifies the result against task constraints. Output sanitization strips any inadvertently leaked credentials or PII.
  5. Ship — The result is finalized, the audit entry is created with SHA-256 hash chain linkage, and the task status updates to "completed".

If any stage fails, the wizard displays the failure reason and suggests remediation. A policy gate rejection, for example, shows which policy was violated and how to modify the task to comply.

Step 9: Audit Trail Review

After successful execution, the wizard navigates to the audit trail view for the completed task. The user sees every pipeline stage with timestamps, durations, inputs, outputs, and the SHA-256 hash that links each entry to the previous one. This step demonstrates the governance value: every action the agent took is recorded, timestamped, and cryptographically chained. The wizard highlights the hash chain verification — showing that the audit trail is tamper-evident.

Step 10: Production Deployment Path

The final step presents the production deployment configuration: a Docker Compose file with environment variables, SSL certificate instructions, and the recommended Apache2 reverse proxy configuration. This step does not execute the deployment — it provides the artifacts needed for the user to deploy on their own infrastructure. A "Download Configuration" button exports the deployment files as a zip archive.

Step 10 is the exit from the wizard and the entrance to production. The user leaves with a working configuration, not a promise of future documentation.

Prerequisite Validation

Each step validates that its prerequisites are satisfied before allowing progression. Step 4 (API key) cannot be reached without step 3 (provider selection). Step 7 (create task) cannot be reached without step 5 (register agent). The validation is enforced both client-side (disabled buttons, progress indicators) and server-side (the API rejects task creation if no agent exists, skill assignment if no agent is registered).

However, the wizard allows backward navigation without penalty. A user who reaches step 6 and realizes they want a different LLM provider can return to step 3, change the provider, and the wizard will re-validate all dependent steps. The progress indicator shows which steps need re-validation after a backward change.

State Persistence

Wizard progress is stored in two locations: browser localStorage for the current step index and UI state, and the server-side user profile for completed configuration (provider, API key, agent, skills). If the user closes the browser and returns, the wizard resumes from the last incomplete step. If the user completes the wizard and later wants to reconfigure, each step is accessible independently through the settings pages — the wizard is an onboarding convenience, not a configuration prison.

Respecting Developer Intelligence

The wizard does not explain what an API key is. It does not define "agent" or "task." It does not include tooltip popups or animated tutorials. Every step presents a form with clear labels, a submit action, and an optional "Learn more" link to the documentation. The wizard assumes the user is a capable engineer who needs a clear sequence, not an explanation of basic concepts. This design choice is deliberate: condescending onboarding is the fastest way to lose a technical user's trust.

// Key Takeaways

  • The wizard produces a working system, not a learning experience — step 10 outputs deployment-ready configuration
  • Prerequisite validation enforces step ordering without blocking backward navigation
  • Step 8 executes the full Verified Autonomy Pipeline with real-time stage visualization
  • State persists across sessions: localStorage for UI state, server-side for configuration
  • Developer-respectful design: clear forms, no tooltips, no concept explanations
  • Each wizard step maps to a settings page for post-wizard reconfiguration
