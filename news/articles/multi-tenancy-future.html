Multi-Tenancy and the Future of Shared AI Infrastructure

As AI orchestration moves from single-organization deployments to shared infrastructure, multi-tenancy becomes the defining architectural challenge. Tenant data must be completely isolated. Execution environments must be segregated. Audit chains must be tenant-scoped. And all of this must happen without sacrificing the cost efficiency, update velocity, and operational simplicity that make shared infrastructure attractive in the first place.

Why Multi-Tenancy Matters for AI Orchestration

Single-tenant deployments are straightforward: one organization, one database, one execution environment, one audit chain. Every resource belongs to one owner. Isolation is trivial because there is nothing to isolate from. But single-tenant deployments do not scale economically. Each new customer requires a new deployment: new servers, new databases, new monitoring, new maintenance. The operational cost per customer is constant, not amortized.

Multi-tenancy amortizes infrastructure across customers. One deployment serves many tenants, sharing compute, storage, and operational overhead. The cost per tenant decreases as the number of tenants increases. For SaaS-model AI orchestration, multi-tenancy is the economic foundation. But in AI orchestration, the isolation requirements are more demanding than in traditional SaaS applications because the data flowing through the system includes prompts, API tokens, model outputs, and audit records — each with its own sensitivity classification.

Multi-tenancy is an economic requirement that creates a security engineering challenge. The challenge is not whether to share infrastructure, but how to share infrastructure without sharing trust boundaries.

Data Isolation: Three Approaches

Database-Per-Tenant

The strongest isolation model gives each tenant its own database instance. Tenant A's data physically cannot be queried from Tenant B's connection. This model provides the simplest security guarantee but the worst operational scaling: managing thousands of database instances creates significant operational overhead, and cross-tenant operations (platform-wide analytics, system health monitoring) require querying every database individually.

# Database-per-tenant connection routing
def get_tenant_db(tenant_id: str) -> Database:
    """Route to tenant-specific database instance."""
    db_url = tenant_config.get_database_url(tenant_id)
    if db_url is None:
        raise TenantNotFoundError(tenant_id)
    return create_connection(db_url)

# Pros: strongest isolation, simple security model
# Cons: operational overhead, no cross-tenant queries

Schema-Per-Tenant

A middle-ground approach uses a shared database instance with per-tenant schemas (namespaces). Each tenant's tables are in a separate schema, and database connections are scoped to the tenant's schema. This provides logical isolation within a shared physical database. Cross-tenant data access requires explicit schema switching, which can be controlled at the connection pool level. This model balances isolation strength with operational simplicity.

# Schema-per-tenant with connection pool scoping
async def get_tenant_session(tenant_id: str) -> AsyncSession:
    """Create a database session scoped to tenant's schema."""
    session = await get_session()
    schema = f"tenant_{tenant_id}"
    await session.execute(
        text(f"SET search_path TO {schema}, public")
    )
    return session

Row-Level Security

The most operationally efficient model uses shared tables with row-level security (RLS) policies. Every table includes a tenant_id column, and database-level RLS policies automatically filter queries to the current tenant's rows. This model allows the simplest schema management (one set of tables, one migration path) but requires absolute trust in the RLS implementation. A bug in the RLS policy is a cross-tenant data breach.

# PostgreSQL row-level security for multi-tenancy
-- Enable RLS on the audit_log table
ALTER TABLE audit_log ENABLE ROW LEVEL SECURITY;

-- Create policy: tenants can only see their own rows
CREATE POLICY tenant_isolation ON audit_log
    USING (tenant_id = current_setting('app.current_tenant'));

-- Set tenant context at connection time
SET app.current_tenant = 'tenant_abc123';

OCCP's Approach

OCCP's multi-tenancy roadmap uses schema-per-tenant as the default isolation model, with database-per-tenant available as an upgrade for tenants with strict regulatory requirements. The schema model provides strong logical isolation, allows per-tenant schema migrations, and supports platform-wide operations through cross-schema queries when needed. Row-level security is used as an additional defense-in-depth layer within each schema, not as the primary isolation mechanism.

Execution Isolation

Per-Tenant Sandboxes

Data isolation protects stored data. Execution isolation protects running processes. When Tenant A's agent is executing a task, it must not be able to observe, influence, or interfere with Tenant B's agent execution. In OCCP's architecture, each tenant's tasks execute in dedicated sandbox instances. The sandbox provides filesystem isolation (each tenant sees only its own working directory), process isolation (each tenant's processes cannot signal or inspect other tenants' processes), and network isolation (each tenant's execution environment has its own network namespace).

Resource Quotas

Execution isolation extends to resource allocation. Each tenant has configurable quotas for CPU time, memory, concurrent tasks, LLM API token consumption, and storage. These quotas prevent the "noisy neighbor" problem where one tenant's workload degrades performance for others. Resource enforcement is implemented at the orchestrator level (task scheduling) and at the sandbox level (cgroup limits for process-level enforcement).

# Per-tenant resource quota configuration
{
    "tenant_id": "tenant_abc123",
    "quotas": {
        "max_concurrent_tasks": 20,
        "max_tokens_per_hour": 500000,
        "max_cpu_seconds_per_task": 120,
        "max_memory_mb_per_task": 512,
        "max_storage_gb": 10,
        "max_agents": 50
    }
}

Credential Isolation

Each tenant's API tokens and encryption keys are isolated from other tenants. The envelope encryption system (described in the AES-256-GCM article) uses per-tenant master keys, ensuring that even a complete key compromise for one tenant does not expose another tenant's credentials. The key version registry is tenant-scoped: each tenant can rotate their keys independently on their own schedule.

Audit Isolation

Tenant-Scoped Audit Chains

Each tenant has its own SHA-256 hash chain. Tenant A's audit chain is independent of Tenant B's chain: they have separate genesis entries, separate hash sequences, and separate retention policies. This design ensures that one tenant's audit history cannot be tampered with by affecting another tenant's chain, and that audit exports for regulatory compliance contain only the requesting tenant's data.

# Tenant-scoped audit chain
class TenantAuditChain:
    def __init__(self, tenant_id: str):
        self.tenant_id = tenant_id
        self.schema = f"tenant_{tenant_id}"

    async def append(self, entry: AuditEntry) -> str:
        """Append entry to tenant-specific chain."""
        prev = await self.get_latest_hash()
        entry.prev_hash = prev
        entry.tenant_id = self.tenant_id
        entry.entry_hash = compute_hash(entry, prev)
        await self.store(entry)
        return entry.entry_hash

    async def verify_integrity(self) -> bool:
        """Verify this tenant's chain independently."""
        entries = await self.get_all_entries()
        return verify_chain(entries)

Cross-Tenant Visibility

Platform operators need cross-tenant visibility for system health monitoring, capacity planning, and incident response. This visibility is carefully scoped: operators can see aggregated metrics (total tasks per tenant, error rates, resource utilization) but cannot see individual tenant data (prompt contents, model outputs, audit details). The aggregation happens at the query level, not by granting operators access to tenant schemas. Tenant data never leaves the tenant's isolation boundary, even for platform operations.

The Challenge: Security and Performance

Isolation Overhead

Every isolation mechanism has a performance cost. Schema-per-tenant requires connection pool management per schema. Sandbox-per-tenant requires process creation overhead for each task. Tenant-scoped audit chains require per-tenant hash computation. The engineering challenge is minimizing this overhead while maintaining isolation guarantees. Connection pooling per schema is amortized across requests. Sandbox creation is parallelized with task queuing. Hash computation is asynchronous and non-blocking.

Migration Complexity

Schema migrations in a multi-tenant system must be applied to every tenant's schema. A migration that takes 100ms for one schema takes 100ms times N tenants. Migrations must be backward-compatible (old code must work with the new schema) because tenants cannot all be migrated simultaneously in a rolling deployment. OCCP's migration system applies changes per schema with error handling: if a migration fails for one tenant, the other tenants are not affected, and the failed migration can be retried independently.

Shared Infrastructure Benefits

Despite the complexity, shared infrastructure provides compelling benefits. Cost efficiency: infrastructure costs are amortized across tenants, making the per-tenant cost lower than single-tenant deployments. Faster updates: platform improvements (security patches, new features, performance optimizations) are deployed once and benefit all tenants simultaneously. Unified monitoring: a single monitoring stack covers all tenants, enabling platform-wide alerting and capacity planning. Operational simplicity: one deployment pipeline, one incident response process, one on-call rotation.

The multi-tenancy roadmap for OCCP is designed to deliver these benefits while maintaining the isolation properties that enterprise customers require. The architecture is being developed incrementally: single-tenant foundations first (the current release), tenant-scoped data isolation next (v0.9.0), and full execution and audit isolation in the production-ready v1.0.0 release.

// Key Takeaways

  • Multi-tenancy requires three isolation dimensions: data, execution, and audit chain separation
  • OCCP uses schema-per-tenant as the default, with database-per-tenant available for strict regulatory requirements
  • Execution isolation provides per-tenant sandboxes with cgroup-level resource quotas to prevent noisy-neighbor effects
  • Tenant-scoped SHA-256 audit chains ensure independent integrity verification and compliance-ready exports
  • Platform operators see aggregated metrics but never individual tenant data, preserving isolation even for operations
  • Shared infrastructure delivers cost efficiency, faster updates, and unified monitoring across the tenant base
