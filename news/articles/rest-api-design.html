REST API Design for Autonomous Workflow Engines

Designing RESTful endpoints for task lifecycle management, agent coordination, and real-time pipeline execution. How FastAPI's type system and automatic schema generation enabled OCCP to ship dual SDKs from a single source of truth.

The Problem: Orchestrating Autonomous Agents via HTTP

An AI orchestration platform is fundamentally a state machine exposed over HTTP. Tasks are created, queued, assigned to agents, executed through a multi-stage pipeline, and eventually completed or failed. Every state transition must be atomic, auditable, and accessible to both human operators and automated SDKs. The REST API is not a convenience layer -- it is the platform's primary interface.

When we started designing the OCCP API surface for v0.2.0, we had three constraints: the API must support the full Verified Autonomy Pipeline (Plan, Gate, Execute, Validate, Ship), it must generate accurate OpenAPI schemas for SDK generation, and it must remain backward-compatible as the platform evolves. These constraints shaped every endpoint design decision.

Task Lifecycle Endpoints

The core resource in OCCP is the task. A task represents a unit of work submitted to the orchestration engine. The task lifecycle maps directly to HTTP methods:

# Create a new task
POST /api/v1/tasks
Content-Type: application/json
Authorization: Bearer <jwt-token>

{
  "description": "Analyze repository for security vulnerabilities",
  "agent_id": "agent-sec-01",
  "priority": "high",
  "metadata": {
    "repository": "occp-core",
    "scan_type": "full"
  }
}

# Response: 201 Created
{
  "id": "task-a1b2c3d4",
  "status": "pending",
  "created_at": "2025-12-12T10:30:00Z",
  "pipeline_stage": null,
  "audit_hash": "sha256:e3b0c44..."
}

The response includes the audit_hash from the first moment of creation. Every subsequent state change appends to the SHA-256 hash chain, creating a tamper-evident record from task birth to completion.

Retrieving Task State

Task retrieval supports both individual lookup and filtered listing. The individual endpoint returns the full task object including pipeline stage history:

# Get task by ID
GET /api/v1/tasks/task-a1b2c3d4

# List tasks with filters
GET /api/v1/tasks?status=running&agent_id=agent-sec-01&limit=20&offset=0

We chose query parameters over request body filtering for GET requests. This keeps the API cacheable and allows browser-based debugging. The limit and offset parameters follow standard pagination patterns, with a maximum page size of 100 enforced server-side.

Partial Updates with PATCH

Task updates use PATCH rather than PUT. In an orchestration system, clients rarely need to replace the entire task object -- they need to update specific fields like priority, metadata, or cancel a running task:

# Update task priority
PATCH /api/v1/tasks/task-a1b2c3d4
{
  "priority": "critical"
}

# Cancel a task
PATCH /api/v1/tasks/task-a1b2c3d4
{
  "status": "cancelled",
  "reason": "Superseded by task-e5f6g7h8"
}
Not every state transition is allowed via PATCH. The policy engine validates transitions -- you cannot move a task from "completed" back to "running", and cancellation of tasks in the "ship" stage requires elevated permissions.

Pipeline Execution Endpoints

The Verified Autonomy Pipeline is the core workflow engine. Each stage (Plan, Gate, Execute, Validate, Ship) is represented as a sub-resource of the task:

# Trigger pipeline execution for a task
POST /api/v1/tasks/task-a1b2c3d4/pipeline/execute

# Get current pipeline stage
GET /api/v1/tasks/task-a1b2c3d4/pipeline

# Response
{
  "task_id": "task-a1b2c3d4",
  "current_stage": "gate",
  "stages": {
    "plan": {"status": "completed", "duration_ms": 342},
    "gate": {"status": "running", "started_at": "..."},
    "execute": {"status": "pending"},
    "validate": {"status": "pending"},
    "ship": {"status": "pending"}
  }
}

Each pipeline stage records its own timing metrics. This granularity is essential for identifying bottlenecks. If the gate stage consistently takes longer than expected, it signals that policy evaluation rules may need optimization.

Agent Coordination Endpoints

Agents in OCCP are autonomous workers that claim and execute tasks. The agent coordination API handles registration, heartbeats, and task assignment:

# Register an agent
POST /api/v1/agents
{
  "name": "security-scanner-01",
  "capabilities": ["code_analysis", "vulnerability_scan"],
  "max_concurrent_tasks": 3
}

# Agent heartbeat
POST /api/v1/agents/agent-sec-01/heartbeat
{
  "status": "idle",
  "current_tasks": 0,
  "resource_usage": {"cpu_percent": 12, "memory_mb": 256}
}

# Claim next available task
POST /api/v1/agents/agent-sec-01/claim
{
  "capabilities": ["code_analysis"]
}

The claim endpoint uses capability matching. An agent declares what it can do, and the orchestrator returns the highest-priority pending task that matches those capabilities. This is a pull model -- agents request work rather than having work pushed to them. The pull model simplifies scaling: add more agents and they automatically distribute the load.

FastAPI and Automatic Schema Generation

Pydantic Models as the Single Source of Truth

Every request and response in OCCP is defined as a Pydantic v2 model. FastAPI uses these models to generate the OpenAPI schema at /docs and /openapi.json. This schema becomes the contract that both SDKs implement against:

from pydantic import BaseModel, Field
from datetime import datetime
from enum import Enum

class TaskStatus(str, Enum):
    pending = "pending"
    running = "running"
    completed = "completed"
    failed = "failed"
    cancelled = "cancelled"

class TaskCreate(BaseModel):
    description: str = Field(..., min_length=1, max_length=2000)
    agent_id: str | None = None
    priority: str = Field(default="normal")
    metadata: dict = Field(default_factory=dict)

class TaskResponse(BaseModel):
    id: str
    status: TaskStatus
    created_at: datetime
    pipeline_stage: str | None = None
    audit_hash: str

The Pydantic models enforce validation at the boundary. Invalid requests are rejected with structured error responses before reaching the business logic layer. This is particularly important for an orchestration platform where malformed input could cascade through the pipeline.

From OpenAPI to SDK Methods

The OpenAPI schema generated by FastAPI documents every endpoint, parameter, request body, and response type. Our SDK generation process maps these directly to typed methods:

# Python SDK (generated from OpenAPI)
task = client.tasks.create(
    description="Analyze repository",
    agent_id="agent-sec-01",
    priority="high"
)
# task.id, task.status, task.audit_hash are typed

# TypeScript SDK (generated from OpenAPI)
const task = await client.tasks.create({
  description: "Analyze repository",
  agentId: "agent-sec-01",
  priority: "high",
});
// task.id, task.status, task.auditHash are typed
The OpenAPI schema is not documentation -- it is a machine-readable contract. Any change to the API surface is immediately visible as a schema diff, making breaking changes impossible to miss in code review.

Error Handling and Status Codes

OCCP uses standard HTTP status codes with structured error bodies. Every error response includes a machine-readable error code, a human-readable message, and a request correlation ID for debugging:

# 422 Validation Error
{
  "error": "validation_error",
  "message": "Field 'description' is required",
  "request_id": "req-x9y8z7",
  "details": [
    {"field": "description", "type": "missing"}
  ]
}

# 409 Conflict (invalid state transition)
{
  "error": "invalid_transition",
  "message": "Cannot transition from 'completed' to 'running'",
  "request_id": "req-a1b2c3",
  "current_status": "completed",
  "requested_status": "running"
}

We chose 409 Conflict for invalid state transitions rather than 400 Bad Request. The request itself is well-formed -- the conflict is with the current state of the resource. This distinction matters for SDK clients implementing retry logic: a 409 should not be retried with the same payload, while a 503 should.

Versioning and Forward Compatibility

The API is versioned with a /api/v1/ prefix. New fields added to response objects are always additive -- SDKs must ignore unknown fields. Removing or renaming fields requires a major version bump. This contract allows OCCP to evolve the API without breaking existing integrations.

Endpoint deprecation follows a three-release cycle: the endpoint is marked deprecated in the OpenAPI schema, then returns a Sunset header with the removal date, and finally returns 410 Gone after the sunset date.

Rate Limiting and Authentication

All endpoints require JWT authentication via the Authorization: Bearer header. Rate limiting is enforced per-token with configurable limits. The rate limit headers follow the IETF draft standard:

X-RateLimit-Limit: 100
X-RateLimit-Remaining: 87
X-RateLimit-Reset: 1702396800

Agent heartbeat endpoints have higher rate limits than task creation endpoints. This ensures that agent health monitoring is never throttled during load spikes.

// Key Takeaways

  • Task lifecycle maps to HTTP methods: POST to create, GET to read, PATCH to update state
  • Pipeline stages are sub-resources of tasks, each with independent timing metrics
  • Agent coordination uses a pull model with capability-based task matching
  • Pydantic models serve as the single source of truth for API contracts and SDK generation
  • HTTP status codes are semantically precise: 409 for state conflicts, 422 for validation errors
  • API versioning uses path prefixes with additive-only changes within a major version
