OCCP Roadmap: What Comes After v0.8

Eight releases in five months have established the core: the Verified Autonomy Pipeline, the policy engine, the audit chain, RBAC, sandbox isolation, encryption, and the dashboard. The foundation is solid. The next releases shift focus from building governance primitives to building a production-grade platform that enterprises can deploy, scale, and operate with confidence.

Where We Are: v0.8.x

The v0.8 series represents the completion of the core governance stack. The current state includes: a five-stage Verified Autonomy Pipeline with policy-as-code enforcement, AES-256-GCM envelope encryption for API tokens, Casbin RBAC with a four-role hierarchy, SHA-256 hash-chained audit logging with automated retention enforcement, sandbox isolation with nsjail and bubblewrap, 24-pattern prompt injection detection, PII scanning with configurable sensitivity levels, output sanitization, supply chain verification for MCP packages, agent boundary enforcement at three levels, EU AI Act Article 12/14/19 compliance, a Next.js 15 dashboard with React 19, dual SDKs (Python and TypeScript), and a comprehensive test suite covering 313+ test cases across Python 3.11, 3.12, and 3.13.

The test infrastructure validates the entire stack on every pull request through six CI checks: three Python version matrices, Node.js dashboard tests, TypeScript SDK tests, and secrets scanning. The codebase is production-audited and documented.

v0.9.0: Production Database and Multi-Tenancy Foundations

PostgreSQL Cutover

OCCP currently supports SQLite for development and testing, with PostgreSQL support through SQLAlchemy's dual-backend abstraction. The v0.9.0 release makes PostgreSQL the primary production backend. This is not a simple configuration change — it involves migration path verification (ensuring all Alembic migrations apply cleanly to PostgreSQL), performance optimization (query plans, connection pooling with asyncpg, prepared statements), and operational tooling (backup scripts, replication configuration, health checks that verify PostgreSQL-specific properties).

# v0.9.0 PostgreSQL configuration
database:
  backend: postgresql
  url: "postgresql+asyncpg://occp:***@db:5432/occp"
  pool_size: 20
  max_overflow: 10
  pool_recycle: 3600
  echo: false

  # Production-specific settings
  ssl_mode: "verify-full"
  ssl_ca: "/certs/ca.pem"
  statement_timeout: "30s"
  lock_timeout: "10s"

Multi-Tenancy Data Layer

The v0.9.0 release introduces the data isolation layer for multi-tenancy: schema-per-tenant with automated provisioning. When a new tenant is created, the system creates a tenant-specific database schema, applies all current migrations to that schema, initializes the tenant's audit chain genesis entry, and provisions the tenant's encryption keys. The orchestrator routes all database operations through tenant-scoped sessions, ensuring that queries never cross tenant boundaries.

Connection Pool Management

Multi-tenancy introduces connection pool complexity. A naive implementation creates a separate pool per tenant, which does not scale (1,000 tenants times 20 connections per pool equals 20,000 database connections). OCCP's approach uses a shared connection pool with per-request schema switching: connections are allocated from a single pool, and the tenant's schema is set at the session level before executing queries. This bounds the total connection count to the pool size, regardless of tenant count.

v1.0.0: Production-Stable Release

Stability Guarantees

The v1.0.0 release marks the transition from "working" to "stable." Stable means: the API surface is frozen (no breaking changes without a major version bump), the database schema migration path is guaranteed (upgrades from any v0.x to v1.0 are supported), configuration format is documented and versioned, and the CLI interface is locked. Enterprises that deploy v1.0.0 can plan their integrations with confidence that the API contract will not change unexpectedly.

Observability: Prometheus Metrics

Production operation requires observability. v1.0.0 exposes Prometheus-compatible metrics from every component of the platform. The orchestrator reports task throughput, latency percentiles, error rates, and queue depth. The policy engine reports evaluation counts, denial rates, and per-policy evaluation latency. The audit system reports chain write latency, retention status, and archive throughput. The encryption layer reports decryption latency and key version distribution.

Grafana Dashboards

Alongside Prometheus metrics, v1.0.0 ships with pre-built Grafana dashboards that visualize the platform's operational state. The Pipeline Overview dashboard shows task flow through each stage with real-time error rates and latencies. The Security Dashboard shows injection detection rates, PII detection events, and encryption operations. The Compliance Dashboard shows audit chain health, retention status, and policy evaluation statistics. These dashboards are JSON-exportable and customizable, not a hosted-only feature.

Agent Marketplace

The agent marketplace is a registry of verified agent templates that organizations can deploy within their OCCP instance. Each template includes the agent configuration, declared task types and capabilities, recommended policy sets, and sample integration code. Templates are community-contributed and OCCP-verified: they pass the supply chain verification process (allowlist, static analysis, signature verification) before being listed in the marketplace.

Beyond v1.0: The Long-Term Vision

Federated Governance Across Organizations

Today, each OCCP deployment governs agents within a single organization. The long-term vision is federated governance: organizations that collaborate on multi-org AI workflows can share governance policies, audit evidence, and compliance status across organizational boundaries. A supply chain AI that spans manufacturer, logistics provider, and retailer needs governance that spans all three organizations. Federated governance provides a standardized protocol for cross-org policy sharing and audit verification.

# Federated governance concept
federation:
  org_id: "manufacturer-corp"
  peers:
    - org_id: "logistics-inc"
      trust_level: "verified"
      shared_policies: ["data_classification", "pii_handling"]
      audit_sync: "bidirectional"

    - org_id: "retailer-group"
      trust_level: "verified"
      shared_policies: ["data_classification"]
      audit_sync: "outbound_only"

Agent Mesh for Cross-Org Collaboration

Building on federated governance, the agent mesh enables agents from different organizations to collaborate on shared tasks while respecting each organization's governance policies. An agent from Organization A can request assistance from an agent in Organization B, with both agents subject to their respective governance pipelines. The mesh handles cross-org authentication, policy reconciliation (when two organizations have conflicting policies, the stricter policy wins), and audit trail synchronization.

Compliance Certification Program

As OCCP adoption grows, a formal compliance certification program will validate that specific deployment configurations meet regulatory requirements. A "EU AI Act Compliant" certification would verify that the deployment has Article 12 record-keeping active, Article 14 human oversight configured for high-risk tasks, and Article 19 retention enforcement enabled with a minimum 365-day period. Similar certifications for SOC 2, ISO 27001, and industry-specific regulations (HIPAA, PCI DSS) would follow.

Release Timeline

The roadmap is ambitious but grounded in the engineering velocity demonstrated through v0.1 to v0.8: eight major releases in five months, with each release building on tested, production-audited foundations.

  • v0.9.0 (Q1 2026): PostgreSQL production backend, multi-tenancy data layer, connection pool management
  • v0.9.x (Q1-Q2 2026): Multi-tenancy execution isolation, tenant-scoped audit chains, tenant management API
  • v1.0.0 (Q2 2026): API freeze, Prometheus metrics, Grafana dashboards, agent marketplace, stability guarantees
  • v1.x (Q3-Q4 2026): Federated governance protocol, agent mesh foundations, compliance certification framework
  • v2.0 (2027): Full federation, cross-org agent mesh, certified compliance packages

The Guiding Principle

Every feature on this roadmap follows the same principle that guided the first eight releases: governance is not a feature to be added to an orchestration tool. Governance is the infrastructure layer that makes orchestration trustworthy. PostgreSQL cutover makes governance durable. Multi-tenancy makes governance scalable. Observability makes governance visible. Federation makes governance composable. Each milestone strengthens the same foundation: the governed deployment of AI agents at any scale.
