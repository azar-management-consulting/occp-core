Reverse Proxy Architecture: Routing AI Traffic at Scale

The OCCP platform runs three subdomains -- occp.ai, api.occp.ai, and dash.occp.ai -- each pointing to a different backend service on the same host. Apache2 sits in front, handling SSL termination, security headers, and traffic routing. This article details the configuration and the reasoning behind it.

Why a Reverse Proxy?

Without a reverse proxy, each service would need its own public IP address or run on non-standard ports. Users would access the API at occp.ai:8000 and the dashboard at occp.ai:3000. Beyond the usability problem, this exposes each service directly to the internet, making every service responsible for its own SSL, security headers, and rate limiting.

A reverse proxy centralizes these concerns. Apache2 handles SSL for all services, injects security headers uniformly, and routes traffic based on the requested subdomain. The backend services (FastAPI on port 8000, Next.js on port 3000) bind only to localhost and never see external traffic directly.

The Routing Architecture

The OCCP domain structure routes three distinct traffic patterns:

  • occp.ai -- Landing page and documentation. Static HTML served directly by Apache from the filesystem.
  • api.occp.ai -- FastAPI REST API. Proxied to the Docker container on port 8000. Handles JSON payloads and WebSocket connections.
  • dash.occp.ai -- Next.js dashboard. Proxied to the Docker container on port 3000. Serves the React application and its assets.

Virtual Host Configuration

Each subdomain has its own Apache virtual host. The API virtual host illustrates the core pattern:

<VirtualHost *:443>
    ServerName api.occp.ai

    SSLEngine on
    SSLCertificateFile /etc/letsencrypt/live/api.occp.ai/fullchain.pem
    SSLCertificateKeyFile /etc/letsencrypt/live/api.occp.ai/privkey.pem

    # Security headers
    Header always set Strict-Transport-Security "max-age=63072000; includeSubDomains; preload"
    Header always set X-Content-Type-Options "nosniff"
    Header always set X-Frame-Options "DENY"
    Header always set X-XSS-Protection "1; mode=block"
    Header always set Referrer-Policy "strict-origin-when-cross-origin"

    # Proxy configuration
    ProxyPreserveHost On
    ProxyPass / http://127.0.0.1:8000/
    ProxyPassReverse / http://127.0.0.1:8000/

    # WebSocket proxy
    RewriteEngine On
    RewriteCond %{HTTP:Upgrade} websocket [NC]
    RewriteCond %{HTTP:Connection} upgrade [NC]
    RewriteRule ^/?(.*) ws://127.0.0.1:8000/$1 [P,L]

    # Request size limit (AI payloads can be large)
    LimitRequestBody 10485760

    # Logging
    ErrorLog ${APACHE_LOG_DIR}/api.occp.ai-error.log
    CustomLog ${APACHE_LOG_DIR}/api.occp.ai-access.log combined
</VirtualHost>

The ProxyPreserveHost On directive forwards the original Host header to the backend. Without this, FastAPI would see all requests as coming from 127.0.0.1:8000, breaking CORS validation and any host-based routing logic in the application.

SSL Termination with Let's Encrypt

SSL certificates are provisioned and renewed automatically using Certbot with the Apache plugin. Each subdomain has its own certificate, allowing independent renewal and avoiding the complexity of wildcard certificates (which require DNS challenge validation):

# Initial certificate provisioning
certbot --apache -d occp.ai -d www.occp.ai
certbot --apache -d api.occp.ai
certbot --apache -d dash.occp.ai

# Automatic renewal (cron, twice daily)
0 */12 * * * certbot renew --quiet --deploy-hook "systemctl reload apache2"

The --deploy-hook reloads Apache after a successful renewal, ensuring the new certificate is picked up without a full restart. The twice-daily cron schedule follows Let's Encrypt's recommendation: certificates are renewed when they have 30 days remaining, and the frequent check ensures timely renewal even if the first attempt fails.

Security Headers in Detail

Strict-Transport-Security (HSTS)

The HSTS header with max-age=63072000 (two years) tells browsers to always use HTTPS for this domain. The includeSubDomains directive extends this to all subdomains, preventing mixed-content issues. The preload flag declares intent to be included in browser HSTS preload lists, though actual submission to the preload list is a separate process.

Content-Security-Policy (CSP)

The dashboard virtual host includes a Content-Security-Policy header that restricts resource loading. This is the most complex security header because it must balance security with functionality:

# Dashboard CSP
Header always set Content-Security-Policy "\
  default-src 'self'; \
  script-src 'self' 'unsafe-inline'; \
  style-src 'self' 'unsafe-inline' https://fonts.googleapis.com; \
  font-src 'self' https://fonts.gstatic.com; \
  connect-src 'self' https://api.occp.ai wss://api.occp.ai; \
  img-src 'self' data:; \
  frame-ancestors 'none';"

The connect-src directive explicitly allows connections to api.occp.ai and its WebSocket endpoint (wss://api.occp.ai). Without this, the dashboard's API calls and real-time updates would be blocked by the browser. The frame-ancestors 'none' prevents the dashboard from being embedded in iframes, complementing the X-Frame-Options: DENY header.

X-Content-Type-Options

The nosniff value prevents browsers from MIME-type sniffing. Without this header, a browser might interpret a JSON API response as HTML if the content looks HTML-like, potentially executing embedded scripts. For an API that processes user-generated content through LLMs, this is essential.

WebSocket Proxying

The dashboard's real-time pipeline visualization depends on WebSocket connections to the API server. Standard HTTP proxy directives do not handle WebSocket upgrades. Apache requires explicit rewrite rules to detect the Upgrade: websocket header and route the connection using the ws:// protocol:

RewriteEngine On
RewriteCond %{HTTP:Upgrade} websocket [NC]
RewriteCond %{HTTP:Connection} upgrade [NC]
RewriteRule ^/?(.*) ws://127.0.0.1:8000/$1 [P,L]

The [NC] flag makes the header check case-insensitive, handling browsers that send Websocket, websocket, or WebSocket. The [P] flag in the rewrite rule forces proxy behavior rather than redirect behavior, which is essential for maintaining the WebSocket connection.

WebSocket proxying is the configuration most frequently broken during Apache upgrades. After any Apache update, the first thing we verify is that WebSocket connections from the dashboard still connect. A broken WebSocket does not produce an error in the dashboard -- it silently falls back to no real-time updates, which is worse than an obvious failure.

HTTP to HTTPS Redirect

Every domain has a companion port 80 virtual host that redirects all traffic to HTTPS:

<VirtualHost *:80>
    ServerName api.occp.ai
    RewriteEngine On
    RewriteRule ^(.*)$ https://%{HTTP_HOST}$1 [R=301,L]
</VirtualHost>

The redirect uses a 301 (permanent) status code, telling browsers and search engines to always use the HTTPS version. Combined with HSTS, this ensures that after the first visit, all subsequent connections go directly to HTTPS without the redirect roundtrip.

Request Size Limits

The API virtual host sets LimitRequestBody 10485760 (10MB). AI workloads can involve large payloads -- document uploads for analysis, code repositories for scanning, or batch task submissions. The 10MB limit balances functionality with protection against denial-of-service through oversized requests. The dashboard virtual host uses a smaller limit (1MB) since it only submits form data and API calls.

Performance Considerations

Keep-Alive Settings

Connection keep-alive is enabled for both the client-to-proxy and proxy-to-backend connections. This is critical for the dashboard, which makes multiple API calls on each page load (task list, agent status, pipeline states). Without keep-alive, each API call would require a new TCP connection and TLS handshake, adding 50-100ms of latency per request.

# Backend keep-alive
ProxyPass / http://127.0.0.1:8000/ keepalive=On
SetEnv proxy-nokeepalive 0

Compression

Apache's mod_deflate compresses JSON responses from the API and HTML/CSS/JS from the dashboard. API responses (JSON task lists, audit logs) compress well -- typically 70-80% reduction. This reduces bandwidth usage and improves response times, particularly for operators on slower connections.

AddOutputFilterByType DEFLATE application/json
AddOutputFilterByType DEFLATE text/html text/css application/javascript

Compression is applied at the proxy level rather than in the backend services, keeping the application code focused on business logic rather than transport optimization.

// Key Takeaways

  • Reverse proxy centralizes SSL, security headers, and routing for all OCCP services
  • Three subdomains route to different backends: static site, FastAPI API, Next.js dashboard
  • HSTS with includeSubDomains and preload enforces HTTPS across the entire domain
  • WebSocket proxy requires explicit rewrite rules; standard ProxyPass does not handle upgrades
  • Content-Security-Policy must explicitly allow API and WebSocket connections for the dashboard
  • Backend services bind to localhost only, never exposed directly to the internet
