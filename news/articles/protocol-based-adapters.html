Protocol-Based Adapters: Making AI Orchestration Pluggable

Each stage of the Verified Autonomy Pipeline is defined by a protocol — a Python interface that specifies inputs, outputs, and behavioral contracts. Any component that implements the protocol can be used at that stage. This adapter pattern makes OCCP pluggable by design, enabling new LLM providers, validators, and delivery mechanisms without modifying the core pipeline.

The Problem with Hard-Coded Integrations

Most AI orchestration frameworks start with a specific LLM provider in mind. The orchestrator calls the OpenAI API directly, or imports the Anthropic client library, or uses a specific embedding service. When a new provider is needed, the orchestrator code must be modified: new import statements, new API call patterns, new response parsing logic. Each integration is a code change to the core system.

This approach creates several problems. Adding a new provider requires modifying core code, which risks introducing regressions. Testing requires mocking the specific API clients of each provider. Switching providers for a deployment means changing code, not configuration. And the core orchestrator accumulates provider-specific logic that makes it harder to understand and maintain.

When your orchestrator imports a specific LLM client library, you have not built an orchestrator. You have built a wrapper around that library.

Protocols vs. Abstract Base Classes

Python offers two mechanisms for defining interfaces: Abstract Base Classes (ABC) from the abc module, and Protocols from the typing module (introduced in Python 3.8). OCCP uses both, for different purposes.

ABCs require explicit inheritance: a class must declare that it extends the ABC. This provides strong guarantees (the class is checked at instantiation time) but requires coupling — the implementing class must know about and import the ABC.

Protocols use structural subtyping (duck typing with static checking): a class satisfies a Protocol if it has the required methods and attributes, without declaring any relationship to the Protocol. This enables loose coupling — an adapter can be written in a separate package that does not depend on OCCP's core.

# ABC approach — requires inheritance
from abc import ABC, abstractmethod

class ExecutorABC(ABC):
    @abstractmethod
    async def execute(self, plan: TaskPlan) -> ExecutionResult:
        pass

class ClaudeExecutor(ExecutorABC):  # Must inherit
    async def execute(self, plan: TaskPlan) -> ExecutionResult:
        ...

# Protocol approach — structural subtyping
from typing import Protocol, runtime_checkable

@runtime_checkable
class Executor(Protocol):
    async def execute(self, plan: TaskPlan) -> ExecutionResult:
        ...

class ClaudeExecutor:  # No inheritance needed
    async def execute(self, plan: TaskPlan) -> ExecutionResult:
        ...

OCCP uses ABCs for the core pipeline stages (where strong guarantees are important) and Protocols for extension points (where third-party adapters should not need to depend on OCCP's internals). The @runtime_checkable decorator enables isinstance() checks at runtime, providing a safety net even with structural subtyping.

The Four Adapter Protocols

Planner Adapter

The Planner adapter is responsible for decomposing a raw task into a structured TaskPlan. The default planner uses rule-based decomposition, but the protocol allows LLM-based planning (using the model itself to decompose complex tasks) or hybrid approaches.

class Planner(Protocol):
    async def plan(self, task: Task) -> TaskPlan:
        """Decompose a task into a structured plan.

        Args:
            task: Raw task with description and parameters.

        Returns:
            TaskPlan with subtasks, strategy, risk level, and resource estimates.

        Raises:
            PlanningError: If the task cannot be decomposed.
        """
        ...

The Planner protocol is deliberately minimal: one method, one input type, one output type. This constraint forces planners to have a single responsibility. A planner that also validates or executes is a violation of the protocol's intent, even if it satisfies the type signature.

Executor Adapter

The Executor adapter handles the actual LLM API call or code execution. This is where provider-specific logic lives: API endpoint URLs, authentication headers, request formatting, response parsing, error handling, and retry logic.

class Executor(Protocol):
    async def execute(self, plan: TaskPlan) -> ExecutionResult:
        """Execute a planned task.

        Args:
            plan: A TaskPlan that has passed Gate evaluation.

        Returns:
            ExecutionResult with output, token usage, and timing.

        Raises:
            ExecutionError: If execution fails (timeout, API error, etc.).
        """
        ...

    @property
    def provider_name(self) -> str:
        """Human-readable name of the LLM provider."""
        ...

The Executor protocol includes a provider_name property used for logging and metrics. Each executor implementation maps to a specific provider: ClaudeExecutor, GPT4Executor, GeminiExecutor. The orchestrator selects the executor based on the task plan's provider specification, without knowing anything about the provider's API details.

Validator Adapter

The Validator adapter checks execution results against task constraints. The default validator performs PII scanning and schema validation. Custom validators can implement domain-specific checks: verifying that generated SQL is syntactically valid, checking that generated code compiles, or ensuring that summarization output preserves key facts.

class Validator(Protocol):
    async def validate(self, result: ExecutionResult, plan: TaskPlan) -> ValidationResult:
        """Validate an execution result against task constraints.

        Args:
            result: The output from the Execute stage.
            plan: The original TaskPlan, for constraint reference.

        Returns:
            ValidationResult with passed/failed status and issues list.
        """
        ...

Validators receive both the execution result and the original plan. This is intentional: the plan contains the constraints that the validator must check against. Without the plan, the validator would need to re-derive constraints from the raw task, duplicating the planner's work.

Shipper Adapter

The Shipper adapter handles result delivery after validation and audit logging. Different deployment scenarios require different delivery mechanisms: a REST API returns results as HTTP responses, a batch system writes results to a file system, an event-driven system publishes results to a message queue.

class Shipper(Protocol):
    async def ship(self, result: ValidationResult, audit_id: str) -> ShipResult:
        """Deliver a validated result to its destination.

        Args:
            result: The validated execution result.
            audit_id: The ID of the committed audit entry.

        Returns:
            ShipResult with delivery confirmation.

        Raises:
            ShipError: If delivery fails.
        """
        ...

The Shipper receives the audit_id as a parameter, ensuring that every delivered result can be traced back to its audit entry. This is a deliberate design constraint: results cannot be shipped without an audit record.

Adding a New LLM Provider

To integrate a new LLM provider with OCCP, a developer implements the Executor protocol. Here is a simplified example showing the structure:

class NewProviderExecutor:
    """Executor adapter for a hypothetical LLM provider."""

    provider_name = "new-provider"

    def __init__(self, api_endpoint: str, model: str):
        self.api_endpoint = api_endpoint
        self.model = model

    async def execute(self, plan: TaskPlan) -> ExecutionResult:
        start_time = time.monotonic()

        # Provider-specific API call
        request_body = self._format_request(plan)
        response = await self._call_api(request_body)
        output = self._parse_response(response)

        elapsed = time.monotonic() - start_time
        return ExecutionResult(
            output=output,
            tokens_used=response.usage.total_tokens,
            elapsed_seconds=elapsed,
            provider=self.provider_name,
            model=self.model,
        )

    def _format_request(self, plan: TaskPlan) -> dict:
        # Provider-specific request formatting
        ...

    async def _call_api(self, body: dict) -> ProviderResponse:
        # Provider-specific HTTP call with error handling
        ...

    def _parse_response(self, response: ProviderResponse) -> str:
        # Provider-specific response extraction
        ...

Notice that this class does not import or inherit from anything in OCCP's core. It satisfies the Executor protocol through structural subtyping: it has an execute method with the correct signature and a provider_name property. This means the adapter can be distributed as an independent package, tested independently, and versioned independently from OCCP.

Benefits of Protocol-Based Design

Testability

Each adapter can be tested in isolation. A planner adapter can be tested with mock tasks. An executor adapter can be tested with a mock API server. A validator adapter can be tested with pre-fabricated execution results. The pipeline itself can be tested with mock adapters that return predictable results, enabling comprehensive integration testing without any LLM API calls.

Pluggability

Switching LLM providers is a configuration change, not a code change. A deployment can use Claude for code generation tasks and GPT-4 for summarization tasks by registering different executor adapters for different agent types. Provider selection is a scheduling decision, not an architectural one.

Clear Contracts

Each protocol defines a precise contract: what goes in, what comes out, and what exceptions can be raised. These contracts are enforced by Python's type system (with mypy or pyright) and by runtime checks. A developer implementing a new adapter knows exactly what is expected, without reading the orchestrator's implementation.

This contract-based architecture is what makes OCCP a platform rather than a framework. Frameworks require you to work within their structure. Platforms define interfaces and let you bring your own implementations. The protocol-based adapter pattern is the mechanism that makes this distinction concrete.

// Key Takeaways

  • Each pipeline stage is defined by a protocol with clear input/output contracts
  • Four adapter protocols: Planner, Executor, Validator, Shipper — each with a single responsibility
  • Structural subtyping (Python Protocol) enables adapters without inheritance or OCCP dependencies
  • New LLM providers are added by implementing the Executor protocol — no core code changes
  • Protocol-based design enables independent testing, independent packaging, and independent versioning
  • Provider selection becomes a configuration decision, not an architectural one
