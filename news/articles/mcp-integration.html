MCP Integration: Connecting AI Agents to External Tools

AI agents are only as useful as the tools they can access. The Model Context Protocol (MCP) provides a universal standard for connecting AI systems to external services — filesystems, databases, APIs, cloud providers. OCCP's MCP integration routes every tool call through the policy gate, turning an open protocol into a governed one.

What Is MCP

The Model Context Protocol is an open specification for connecting AI models to external tools and data sources. Developed by Anthropic in late 2024, MCP defines a standard interface for tool discovery, parameter validation, and result formatting. An MCP server exposes a set of tools — each with a name, description, and JSON Schema for its parameters — and an MCP client (the AI application) discovers and invokes those tools through a well-defined protocol.

The protocol uses JSON-RPC 2.0 as its message format, with two transport mechanisms: stdio (standard input/output for local processes) and HTTP with Server-Sent Events (for remote servers). This dual-transport design means MCP servers can run as local processes on the same machine as the AI application or as remote services accessed over the network.

// MCP JSON-RPC request — listing available tools
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/list",
  "params": {}
}

// MCP JSON-RPC response — tool definitions
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "tools": [
      {
        "name": "read_file",
        "description": "Read the contents of a file at the given path",
        "inputSchema": {
          "type": "object",
          "properties": {
            "path": { "type": "string", "description": "Absolute file path" }
          },
          "required": ["path"]
        }
      }
    ]
  }
}

OCCP's MCP Client Architecture

OCCP implements an MCP client that manages connections to multiple MCP servers simultaneously. The client handles three core operations: server lifecycle management (start, connect, health check, reconnect), tool discovery (listing and caching available tools from each server), and tool execution (calling tools with validated parameters and processing results).

Server Lifecycle Management

Each MCP server connection is managed through a MCPConnection class that handles the transport-specific initialization. For stdio servers, the client spawns the server process and communicates through stdin/stdout. For HTTP+SSE servers, the client establishes an HTTP connection and opens an SSE stream for server-initiated messages.

# occp/mcp/connection.py — MCP server connection manager
class MCPConnection:
    def __init__(self, server_config: MCPServerConfig):
        self.config = server_config
        self.transport = None
        self.tools: list[MCPTool] = []
        self.status = "disconnected"

    async def connect(self):
        if self.config.transport == "stdio":
            self.transport = await StdioTransport.create(
                command=self.config.command,
                args=self.config.args,
                env=self.config.env,
            )
        elif self.config.transport == "sse":
            self.transport = await SSETransport.create(
                url=self.config.url,
                headers=self.config.headers,
            )
        self.status = "connected"
        await self.discover_tools()

    async def discover_tools(self):
        response = await self.transport.send({
            "jsonrpc": "2.0",
            "id": self._next_id(),
            "method": "tools/list",
            "params": {}
        })
        self.tools = [MCPTool(**t) for t in response["result"]["tools"]]

Tool Discovery and Registry

When an MCP server connects, OCCP queries its available tools and registers them in a global tool registry. The registry maps tool names to their source server, parameter schema, and description. This registry is used by the skills system to resolve tool dependencies and by the policy engine to evaluate tool access permissions.

Tool names are namespaced by server to prevent collisions. A read_file tool from the filesystem server becomes filesystem.read_file in OCCP's registry, while a read_file tool from a different server would have its own namespace. This namespacing is transparent to the agent — the agent requests a tool by its fully qualified name, and the MCP client routes the call to the correct server.

Tool Execution Through the Policy Gate

This is where OCCP's MCP integration differs fundamentally from a standard MCP client. In a typical setup, an AI agent discovers MCP tools and calls them directly. In OCCP, every MCP tool call is routed through the Verified Autonomy Pipeline's Gate stage before execution.

# occp/mcp/executor.py — policy-gated tool execution
class MCPExecutor:
    def __init__(self, policy_engine: PolicyEngine, registry: ToolRegistry):
        self.policy = policy_engine
        self.registry = registry

    async def execute_tool(self, tool_name: str, params: dict, agent_id: int) -> dict:
        # 1. Resolve tool to its MCP server
        tool = self.registry.get(tool_name)
        if not tool:
            raise ToolNotFoundError(f"Tool '{tool_name}' not registered")

        # 2. Validate parameters against JSON Schema
        validate(params, tool.input_schema)

        # 3. Policy gate evaluation
        policy_result = await self.policy.evaluate_tool_call(
            tool_name=tool_name,
            params=params,
            agent_id=agent_id,
        )
        if not policy_result.allowed:
            raise PolicyViolationError(policy_result.reason)

        # 4. Execute through MCP transport
        connection = self.registry.get_connection(tool.server_name)
        result = await connection.call_tool(tool_name, params)

        # 5. Output sanitization
        sanitized = await self.policy.sanitize_output(result)

        # 6. Audit logging
        await self.audit.log_tool_call(
            tool=tool_name, params=params, result=sanitized,
            agent_id=agent_id, policy_result=policy_result,
        )

        return sanitized

The six-step execution flow ensures that no MCP tool call bypasses governance. Parameter validation catches malformed inputs before they reach the external service. Policy evaluation checks the agent's permissions for the specific tool and the specific parameters. Output sanitization strips any sensitive data from the tool's response before it returns to the agent. Audit logging records the complete interaction.

MCP without governance is a backdoor. Every tool an agent can call is a potential attack vector: filesystem access can leak data, database access can modify records, API access can consume resources. OCCP's contribution is not the MCP client itself — it is the policy gate that wraps it.

Integration with 22 MCP Servers

OCCP ships with pre-configured connection profiles for 22 MCP servers, organized into five categories:

  • Development Tools — Filesystem (file read/write/search), GitHub (repos, PRs, issues, actions), Git (local repository operations).
  • Databases — SQLite (local queries), PostgreSQL (production database), MySQL (legacy systems).
  • Cloud Infrastructure — Hetzner Cloud (server management), Cloudflare (DNS, Workers, R2 storage), Hostinger (hosting management).
  • Monitoring and Automation — Grafana (dashboards, alerts), n8n (workflow automation), Docker (container management).
  • Security and Research — Semgrep (static analysis), Snyk (dependency scanning), GitGuardian (secret detection), web search, web scraping, documentation lookup.

Each pre-configured profile includes the server command or URL, required environment variables, and a default policy configuration. Enabling a new MCP server in the dashboard automatically applies the appropriate policy rules: database servers get query-only access by default, filesystem servers get read-only access, and cloud infrastructure servers require explicit admin approval before write operations.

Why MCP Is the Universal Standard

Before MCP, every AI framework implemented its own tool integration mechanism. LangChain had Tools, CrewAI had Actions, AutoGen had Functions. Each required a different interface, different parameter format, and different error handling approach. A tool written for LangChain could not be used in CrewAI without a wrapper, and neither could be used outside their respective frameworks.

MCP solves this fragmentation by defining a single protocol that any AI system can implement. An MCP server written for Claude Desktop works in OCCP without modification. An MCP server built for another orchestration platform works in OCCP without modification. The protocol is the interface, not the framework. This is the same architectural insight that made HTTP, SMTP, and SQL successful: standardize the protocol, and the ecosystem builds itself.

OCCP's specific contribution is adding the governance layer. MCP defines how to call tools. OCCP defines the rules about whether you should call them, what to do with the results, and how to record that the call happened. Protocol standardization plus governance enforcement is the combination that makes MCP viable for enterprise AI deployments.

Transport Security

Stdio transport inherits the process-level isolation of the operating system — the MCP server runs as a child process with the same security context as OCCP. HTTP+SSE transport introduces network-level concerns: MCP calls over HTTP must use TLS, authentication tokens must be included in headers, and server certificates must be validated. OCCP's MCP client enforces TLS for all HTTP+SSE connections and stores authentication tokens using the same AES-256-GCM encryption used for LLM API keys.

// Key Takeaways

  • MCP provides a universal protocol for AI-to-tool communication via JSON-RPC 2.0 over stdio or HTTP+SSE
  • OCCP routes every MCP tool call through the policy gate: parameter validation, permission check, output sanitization, audit log
  • Tool names are namespaced by server to prevent collisions across 22+ integrated servers
  • Pre-configured server profiles apply default security policies: read-only filesystem, query-only database, admin-gated cloud infrastructure
  • MCP standardizes the protocol; OCCP adds the governance that makes it enterprise-viable
