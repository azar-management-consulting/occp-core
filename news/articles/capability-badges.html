Capability Badges: Visualizing Agent Permissions

A permission model is only useful if operators can understand it at a glance. Capability badges transform the abstract data in the agent registry into immediate visual indicators — colored, categorized, and interactive — that let platform operators assess agent risk profiles in seconds rather than minutes.

The Visibility Problem

The agent registry introduced in OCCP v0.5.0 captures detailed capability declarations for every registered agent. An agent might declare access to three tools, two data sources, network connectivity, and filesystem write permissions. This information is precise and machine-readable, but when an operator is managing twenty agents across multiple teams, reading JSON capability blocks is not practical.

In incident response scenarios, the gap is even more critical. When an agent behaves unexpectedly, the first question is always: "What can this agent do?" If answering that question requires navigating to the agent detail page, expanding the capabilities section, and parsing a structured data block, the response time is measured in minutes. Those minutes matter.

Security information that requires navigation to find is security information that gets ignored. Capability badges put permission context directly in the operator's line of sight, where it belongs.

Badge Taxonomy

OCCP defines five primary badge categories, each representing a fundamental capability dimension. These categories were derived from the STRIDE threat model adapted for AI agent systems, where each capability dimension maps to a potential attack surface.

READ

The read badge indicates that an agent has access to data sources. The badge displays the number of declared data sources. A read-only agent with access to three datasets shows READ:3. Read badges use a blue color scheme (matching the --blue design token) to indicate passive, non-destructive access. Read access is the lowest risk tier, but the number of data sources affects the risk profile — an agent with access to one public dataset is different from one with access to fifteen internal databases.

WRITE

Write badges indicate that an agent can modify persistent state — writing to databases, updating configuration, or generating files. Write badges use an amber color scheme to signal elevated risk. The badge includes the specific write targets: WRITE:db,config tells the operator exactly what the agent can modify.

EXECUTE

Execute badges indicate that an agent can invoke tools or run code. This is the highest-risk individual capability because tool execution can have side effects that are difficult to predict or reverse. Execute badges use a red color scheme. The badge lists declared tools: EXEC:sql_query,chart_gen.

NETWORK

Network badges indicate that an agent can make outbound network requests. This capability is significant because it represents a potential data exfiltration vector. An agent with network access and read access to sensitive data can, in theory, transmit that data externally. Network badges use a purple color scheme and display either NET:ON or are absent if network access is not declared.

FILESYSTEM

Filesystem badges indicate read or write access to the host filesystem. This is separated from the general read/write categories because filesystem access has distinct security implications — path traversal vulnerabilities, symlink attacks, and access to configuration files. Filesystem badges use a distinct orange variant and display FS:R for read-only or FS:RW for read-write.

Color Coding and Risk Tiers

The badge color system is not arbitrary. It implements a three-tier risk classification that maps directly to the capability's potential impact.

// Badge color mapping in the dashboard
const BADGE_COLORS = {
  read:       { bg: 'rgba(96,165,250,0.15)',  text: '#60a5fa', border: 'rgba(96,165,250,0.3)'  },
  write:      { bg: 'rgba(255,176,0,0.15)',   text: '#ffb000', border: 'rgba(255,176,0,0.3)'   },
  execute:    { bg: 'rgba(248,113,113,0.15)', text: '#f87171', border: 'rgba(248,113,113,0.3)' },
  network:    { bg: 'rgba(167,139,250,0.15)', text: '#a78bfa', border: 'rgba(167,139,250,0.3)' },
  filesystem: { bg: 'rgba(251,146,60,0.15)',  text: '#fb923c', border: 'rgba(251,146,60,0.3)'  },
};

// Risk tier classification
const RISK_TIERS = {
  low:    ['read'],                          // Blue badges only
  medium: ['read', 'write'],                 // Blue + amber
  high:   ['read', 'write', 'execute'],      // Blue + amber + red
  critical: ['execute', 'network', 'filesystem']  // Red + purple + orange
};

When an agent's badge combination falls into the "critical" tier — meaning it has execute, network, and filesystem capabilities simultaneously — the agent card in the dashboard receives a subtle pulsing border animation. This draws the operator's eye to the highest-risk agents without cluttering the interface with alert banners.

Dashboard Implementation

Badges are rendered as React components in the Next.js dashboard. Each badge is a self-contained unit that receives the agent's capability data as props and renders the appropriate visual representation.

// CapabilityBadge component
function CapabilityBadge({ type, details }: BadgeProps) {
  const colors = BADGE_COLORS[type];
  const label = formatBadgeLabel(type, details);

  return (
    <span
      className="capability-badge"
      style={{
        background: colors.bg,
        color: colors.text,
        border: `1px solid ${colors.border}`,
        padding: '2px 8px',
        fontSize: '0.7rem',
        fontFamily: 'var(--font)',
        letterSpacing: '0.05em',
        textTransform: 'uppercase',
      }}
      title={generateTooltip(type, details)}
    >
      {label}
    </span>
  );
}

The generateTooltip function produces a detailed text description that appears on hover. For a WRITE:db,config badge, the tooltip might read: "This agent can write to: database (financial_reports), configuration (agent-settings). Last modified: 2026-01-20." This layered information design means the badge conveys quick context at a glance, while the tooltip provides full detail on demand.

Permission Matrix View

Beyond individual agent badges, the dashboard provides a permission matrix view that displays all agents and their capabilities in a grid format. The rows are agents, the columns are capability categories, and each cell is a colored indicator showing whether the agent has that capability.

Agent                | READ | WRITE | EXEC | NET | FS
---------------------|------|-------|------|-----|----
data-analyst-01      |  3   |   -   |  2   |  -  |  -
code-executor-02     |  1   |   2   |  4   | ON  | RW
report-generator-03  |  5   |   1   |  1   |  -  | R
admin-agent-04       |  8   |   4   |  6   | ON  | RW

The matrix makes cross-agent comparison immediate. If you need to identify every agent with network access, you scan the NET column. If you want to find agents with the highest tool count, you sort by the EXEC column. This tabular view is particularly valuable during security reviews, where auditors need to assess the overall permission landscape.

Badge-Driven Alerting

Badges are not just visual indicators — they feed into the alerting system. When a new agent registers with a critical-tier capability combination, or when an existing agent's capabilities are updated to include a new high-risk dimension, the platform generates an alert. These alerts appear in the dashboard notification feed and can be routed to external systems through webhooks.

The alerting logic is intentionally simple. It does not attempt to evaluate whether a specific capability combination is "dangerous" in context — that judgment belongs to the human operator. Instead, it ensures that capability changes are never silent. Every badge change is a visible event.

Operator Workflow Integration

In practice, the typical operator workflow for agent review follows a consistent pattern. The operator opens the agent list, scans the badge rows for unexpected combinations, and clicks into agents that warrant closer inspection. The badge system reduces this review from a per-agent investigation to a visual scan that covers the entire fleet in seconds.

During incident investigations, badges serve a different purpose. When a task fails or produces unexpected output, the operator checks the executing agent's badges to quickly establish what the agent could have done. If the agent has no network badge but the error suggests an attempted outbound connection, that discrepancy immediately narrows the investigation.

Badges answer the most frequent operational question in AI governance: "What is this agent allowed to do?" If the answer requires more than a glance, the visualization has failed.

Design Decisions

Several design decisions shaped the badge system. First, badges show declared capabilities, not observed behavior. An agent with an EXEC badge has declared tool execution capability; whether it has actually executed tools recently is a separate metric. This distinction is important because capability is about potential, not history.

Second, badges are non-interactive in the agent list view. You cannot revoke a capability by clicking a badge. This is deliberate — capability revocation is a consequential action that should require explicit navigation to the agent management interface, not an accidental click in a crowded list view.

Third, the badge vocabulary is intentionally small. Five categories cover the full capability space without requiring operators to learn a complex taxonomy. Each category maps to a single question: Can the agent read data? Write data? Execute tools? Access the network? Access the filesystem? Five questions, five badges, complete coverage.

// Key Takeaways

  • Five badge categories (READ, WRITE, EXECUTE, NETWORK, FILESYSTEM) cover the complete agent capability space
  • Color coding maps to a three-tier risk classification: low (blue), medium (amber), high/critical (red/purple/orange)
  • Tooltips provide detailed capability breakdowns on hover without cluttering the primary view
  • Permission matrix view enables cross-agent comparison for security reviews and audits
  • Badge changes trigger alerts to ensure capability modifications are never silent
  • Badges show declared capabilities, not observed behavior — potential, not history
