Deployment Challenges: IP Migration and Production Resilience

In February 2026, our Hetzner primary IP was blocked due to an abuse report. The server was reachable over SSH through the console but unreachable from the internet. This is the story of the recovery: allocating a new IP, migrating DNS, verifying SSL, updating CI/CD secrets, and the infrastructure practices that turned a potential disaster into a 45-minute incident.

The Incident

The first sign was a failing health check. The monitoring system reported that api.occp.ai was unreachable. SSH through the Hetzner console confirmed the server was running — all Docker containers were healthy, Apache2 was serving responses, and the application logs showed no errors. The problem was at the network level.

A traceroute from an external host revealed the issue: at hop 10, traffic was being routed to blocked.hetzner.com instead of reaching our server. Hetzner had blocked our primary IPv4 address due to an abuse report. The IPv6 address was also affected. The server was operational but invisible to the internet.

$ traceroute api.occp.ai
 1  gateway (192.168.1.1)       1.234 ms
 2  isp-router (10.0.0.1)      5.678 ms
 ...
 9  hetzner-edge.de (213.x.x.x)  18.432 ms
10  blocked.hetzner.com (...)     19.876 ms  # <-- traffic blackholed here
11  * * *

The Recovery Plan

With the IP blocked at the datacenter level, there was no way to unblock it immediately — abuse reviews take days. The recovery required allocating a new IP address and migrating everything to it. The plan had six steps, executed in sequence because each step depended on the previous one:

  1. Allocate a new primary IPv4 address from Hetzner
  2. Power off the server, reassign the new IP, power on
  3. Update DNS A records for all three subdomains
  4. Update the SPF TXT record for email deliverability
  5. Re-issue SSL certificates for the new IP
  6. Update the GitHub Actions DEPLOY_HOST secret

Step 1: Allocate New Primary IP

Hetzner's cloud API allows allocating primary IP addresses independently of servers. We allocated a new IPv4 address in the same datacenter (fsn1-dc14) and assigned it to our server. The critical detail: Hetzner requires the server to be powered off before reassigning a primary IP. This meant a brief period of total unavailability — even console SSH access would be interrupted.

# Hetzner API — allocate new primary IP
POST /v1/primary_ips
{
  "type": "ipv4",
  "datacenter": "fsn1-dc14",
  "auto_delete": false,
  "assignee_type": "server",
  "name": "occp-primary-v2"
}

# Response: new IP 195.201.238.144 (Primary IP ID: 119875925)

Step 2: Server Power Cycle

The server was powered off through the Hetzner Cloud Console. The old primary IP was unassigned and the new IP was assigned. Power on. The entire power cycle took approximately 90 seconds. Once the server was back, SSH access was verified through the new IP and all Docker containers came back automatically (Docker Compose is configured with restart: unless-stopped).

Step 3: DNS Migration

DNS records were managed through Hostinger's nameservers. Three A records needed updating: @ (occp.ai), api (api.occp.ai), and dash (dash.occp.ai). All three were pointed to the new IP address with a TTL of 300 seconds (5 minutes). The low TTL was a deliberate choice made months earlier — we anticipated the possibility of IP changes and set the TTL low enough to propagate quickly.

# DNS A record updates (Hostinger)
@ (occp.ai)     -> 195.201.238.144  TTL 300
api              -> 195.201.238.144  TTL 300
dash             -> 195.201.238.144  TTL 300
DNS TTL is a setting you configure when nothing is wrong and benefit from when everything is wrong. A 300-second TTL means the entire internet will have your new IP within 5 minutes. A 3600-second TTL means you wait an hour. A 86400-second TTL means you spend a day explaining to users why the site is down even though you fixed it.

Step 4: SPF Record Update

The SPF TXT record for email deliverability referenced the old IP address. Emails sent from the OCCP domain would fail SPF validation if the record was not updated. The new SPF record was set to v=spf1 mx a ip4:195.201.238.144 ~all. TXT record propagation follows the same TTL as A records, so the SPF update was effective within 5 minutes.

Step 5: SSL Certificate Verification

Let's Encrypt certificates are domain-validated, not IP-validated. The existing certificates for occp.ai, api.occp.ai, and dash.occp.ai remained valid after the IP change because they are tied to the domain names, not the IP address. However, we triggered a manual renewal cycle to confirm that the ACME challenge process worked with the new IP. The HTTP-01 challenge succeeded on all three domains, confirming that traffic was correctly routing through the new IP to the Apache2 reverse proxy and then to the ACME validation endpoint.

Step 6: CI/CD Secret Update

The GitHub Actions deployment workflow uses an SSH connection to deploy new releases. The DEPLOY_HOST secret stored the old IP address. This was updated to the new IP through the GitHub repository settings API. The next CI run after the update confirmed successful deployment to the new IP.

Timeline

From incident detection to full recovery:

  • T+0 min — Health check failure detected
  • T+5 min — Root cause identified (IP blocked at datacenter)
  • T+10 min — New primary IP allocated
  • T+15 min — Server power cycled, new IP assigned
  • T+20 min — DNS A records updated, SPF record updated
  • T+25 min — DNS propagation verified from multiple locations
  • T+30 min — SSL certificates verified, ACME renewal confirmed
  • T+35 min — GitHub DEPLOY_HOST secret updated
  • T+40 min — Full production verification: API, dashboard, CI/CD
  • T+45 min — Incident closed, all 12 verification checks passed

Lessons in Production Resilience

DNS TTL Management

The 300-second TTL was set during initial deployment, not during the incident. Setting a low TTL during an incident is useless because the old high TTL is already cached by resolvers worldwide. The lesson: set your TTL low before you need it. The cost of a low TTL is slightly more DNS queries. The benefit is significantly faster recovery from any IP-level incident.

Infrastructure-as-Code

Every DNS record, every Apache2 configuration, every Docker Compose service, and every SSL certificate is documented in the project's infrastructure documentation. During the incident, there was no guessing about which records existed or what values they should have. The recovery plan was mechanical: update each documented record to the new IP. Without documentation, recovery would have required discovery — figuring out which records exist before figuring out what to change them to.

Health Check Monitoring

The incident was detected by an automated health check within minutes. Without monitoring, the outage would have been detected by user reports — potentially hours later. The health check monitors three endpoints (api.occp.ai/health, dash.occp.ai, occp.ai) every 60 seconds and alerts on two consecutive failures. This two-failure threshold prevents false alerts from transient network issues while still detecting real outages quickly.

Docker Restart Policies

When the server powered back on after the IP reassignment, all Docker containers restarted automatically. The restart: unless-stopped policy in Docker Compose ensures that containers survive server reboots without manual intervention. Without this policy, the recovery would have required an additional step: manually starting each container and verifying its health.

Disaster Recovery Playbook

The six-step recovery plan was not invented during the incident. It was a documented playbook created during initial deployment planning. The playbook covered three scenarios: IP change (which is what happened), server migration (moving to a different Hetzner server), and datacenter migration (moving to a different Hetzner location). Having a pre-written playbook transformed the incident from a crisis into a checklist.

The difference between a 45-minute incident and a 4-hour outage is preparation. Every minute spent writing disaster recovery documentation before an incident saves ten minutes of decision-making during one.

Post-Incident Changes

After the recovery, we made three infrastructure changes to improve future resilience. First, we added the new IP to a monitoring allowlist that alerts if the IP appears in any public blocklist databases. Second, we documented the exact Hetzner API calls for IP allocation and reassignment, turning the playbook from a conceptual guide into an executable script. Third, we added a secondary health check that runs from a different geographic location to detect region-specific routing issues that a single-location monitor might miss.

// Key Takeaways

  • Hetzner IP blocks route traffic to blocked.hetzner.com — detectable via traceroute at hop 10
  • Recovery: allocate new IP, power cycle server, update DNS (A + SPF), verify SSL, update CI/CD secrets
  • DNS TTL of 300 seconds enabled full propagation within 5 minutes of the DNS update
  • Let's Encrypt certificates are domain-validated, not IP-validated — no reissuance required on IP change
  • Docker restart: unless-stopped ensures containers survive server power cycles automatically
  • Pre-written disaster recovery playbooks transform incidents from crises into checklists
