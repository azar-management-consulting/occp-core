OCCP v0.4.0: Dashboard Authentication and the Next.js 15 Migration

Version 0.4.0 brought two significant changes: JWT-based authentication in the browser dashboard, and a migration from Next.js 14 to Next.js 15 with React 19. Together, these changes transformed the dashboard from a read-only monitoring tool into a secured operations interface.

What Changed in v0.4.0

The v0.3.0 dashboard could display pipeline status and agent monitoring, but it had no concept of user identity. Anyone who could reach dash.occp.ai could see everything. For an internal tool during early development, this was acceptable. As OCCP moved toward production deployment with real LLM API calls (and real costs), authentication became non-negotiable.

Version 0.4.0 introduced:

  • JWT login flow -- username/password authentication against the FastAPI backend, with token storage in the browser
  • AuthGuard component -- a React wrapper that redirects unauthenticated users to the login page
  • Automatic token refresh -- seamless re-authentication before tokens expire
  • Next.js 15 + React 19 migration -- server components for initial load performance, client components for interactive elements

JWT Authentication in the Browser

The Login Flow

Authentication follows a standard JWT flow with one OCCP-specific addition: the audit hash. Every authentication event is recorded in the SHA-256 hash chain, creating a cryptographic record of who accessed the platform and when.

// Login request
const response = await fetch("https://api.occp.ai/api/v1/auth/login", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ username, password }),
});

// Response includes JWT and audit reference
const { access_token, refresh_token, expires_in, audit_hash } = await response.json();

// Store tokens
localStorage.setItem("occp_access_token", access_token);
localStorage.setItem("occp_refresh_token", refresh_token);
localStorage.setItem("occp_token_expiry", String(Date.now() + expires_in * 1000));

Token Storage Decision

We chose localStorage over cookies for token storage. This decision is often debated in the web security community. Our reasoning:

  • The dashboard is a single-page application. There are no server-rendered pages that need automatic cookie attachment. Every API call is made explicitly with fetch, where we control the Authorization header.
  • CSRF is eliminated. Cookies are automatically sent with every request to the cookie's domain, making CSRF attacks possible. Tokens in localStorage are never sent automatically -- they must be explicitly attached to each request.
  • XSS is the real risk. localStorage is vulnerable to XSS attacks. We mitigate this through strict Content-Security-Policy headers (blocking inline scripts from untrusted sources) and output sanitization in both the API and the dashboard.
There is no perfectly secure token storage mechanism in browsers. localStorage trades CSRF immunity for XSS vulnerability. HttpOnly cookies trade XSS immunity for CSRF vulnerability. We chose the trade-off that aligns with our architecture: a CSP-hardened SPA where every API call is explicit.

Automatic Token Refresh

JWT access tokens have a 15-minute lifetime. The refresh token, stored alongside the access token, has a 7-day lifetime. An interceptor runs before every API call to check the token expiry and refresh proactively:

class AuthenticatedClient {
  private async ensureValidToken(): Promise<string> {
    const expiry = Number(localStorage.getItem("occp_token_expiry"));
    const buffer = 60_000; // Refresh 60s before expiry

    if (Date.now() + buffer < expiry) {
      return localStorage.getItem("occp_access_token")!;
    }

    // Token is expired or about to expire -- refresh
    const refreshToken = localStorage.getItem("occp_refresh_token");
    if (!refreshToken) {
      throw new AuthenticationError("No refresh token available");
    }

    const response = await fetch("https://api.occp.ai/api/v1/auth/refresh", {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({ refresh_token: refreshToken }),
    });

    if (!response.ok) {
      // Refresh failed -- force re-login
      this.logout();
      throw new AuthenticationError("Session expired");
    }

    const { access_token, expires_in } = await response.json();
    localStorage.setItem("occp_access_token", access_token);
    localStorage.setItem("occp_token_expiry", String(Date.now() + expires_in * 1000));
    return access_token;
  }

  async fetch(url: string, options: RequestInit = {}): Promise<Response> {
    const token = await this.ensureValidToken();
    return fetch(url, {
      ...options,
      headers: {
        ...options.headers,
        Authorization: `Bearer ${token}`,
      },
    });
  }
}

The 60-second buffer ensures the token is refreshed before it actually expires, preventing a brief window where API calls fail with 401 errors. The refresh is transparent to the calling code -- it simply uses client.fetch() and the authentication is handled automatically.

The AuthGuard Component

The AuthGuard is a React component that wraps protected routes. If the user is not authenticated, they are redirected to the login page. If they are authenticated, the wrapped content renders normally:

"use client";

import { useEffect, useState } from "react";
import { useRouter } from "next/navigation";

export function AuthGuard({ children }: { children: React.ReactNode }) {
  const [isAuthenticated, setIsAuthenticated] = useState<boolean | null>(null);
  const router = useRouter();

  useEffect(() => {
    const token = localStorage.getItem("occp_access_token");
    const expiry = Number(localStorage.getItem("occp_token_expiry"));

    if (!token || Date.now() > expiry) {
      router.replace("/login");
      return;
    }

    // Verify token with server
    fetch("https://api.occp.ai/api/v1/auth/verify", {
      headers: { Authorization: `Bearer ${token}` },
    })
      .then((res) => {
        if (res.ok) setIsAuthenticated(true);
        else router.replace("/login");
      })
      .catch(() => router.replace("/login"));
  }, [router]);

  if (isAuthenticated === null) {
    return <div className="loading-screen">Verifying...</div>;
  }

  return <>{children}</>;
}

The three-state pattern (null, true, false) prevents the protected content from flashing briefly before the redirect. While authentication is being verified, a loading indicator is shown. Only after server-side token verification completes does the content render.

Next.js 15 + React 19 Migration

Why Migrate During a Security Release?

The timing was deliberate. Adding authentication required significant changes to the component architecture (new client components for login forms, context providers for auth state, route guards). Making these changes in a Next.js 14 codebase and then migrating to 15 would mean touching the same code twice. We chose to migrate first, then build auth into the new architecture.

Server Components for Initial Load

React 19 server components enabled us to render the initial dashboard state on the server. The task list, agent panel, and pipeline overview are server-rendered with fresh data, then hydrated on the client for interactivity. This reduced the time-to-first-meaningful-paint from 2.8 seconds (v0.3.0, full client-side render) to 1.1 seconds (v0.4.0, server-rendered with streaming).

// Server component: loads data at request time
// app/dashboard/page.tsx
export default async function DashboardPage() {
  const tasks = await fetchTasks(); // Runs on server
  const agents = await fetchAgents(); // Runs on server

  return (
    <AuthGuard>
      <DashboardLayout>
        <TaskList initialTasks={tasks} />
        <AgentPanel initialAgents={agents} />
        <PipelineOverview />
      </DashboardLayout>
    </AuthGuard>
  );
}

The Client/Server Boundary

The migration required careful decisions about which components are server and which are client. The rule we established: if a component needs browser APIs (localStorage, WebSocket, event handlers), it is a client component marked with "use client". Everything else defaults to server.

This boundary is particularly interesting for the dashboard because authentication state lives in the browser (localStorage), but initial data loading benefits from the server. The AuthGuard is a client component that wraps server-rendered content, creating a clean separation between the auth layer and the data layer.

Metrics from the Release

The v0.4.0 release added 23 new tests (authentication flows, token refresh, AuthGuard behavior), bringing the total to 107. The Next.js migration changed 34 files but required no changes to the API backend -- the dashboard is a pure consumer of the REST API, so backend changes were isolated to adding the /auth/login, /auth/refresh, and /auth/verify endpoints.

Bundle size decreased by 18% after the migration to React 19, primarily due to server components eliminating client-side JavaScript for the initial data fetching layer. The dashboard's Lighthouse performance score improved from 62 to 84.

// Key Takeaways

  • JWT authentication with localStorage storage eliminates CSRF; CSP headers mitigate XSS
  • Automatic token refresh with 60-second buffer prevents expired-token API failures
  • AuthGuard uses three-state rendering (null/true/false) to prevent content flash
  • Next.js 15 migration was timed to coincide with auth work to avoid double-touching code
  • Server components reduced time-to-first-meaningful-paint from 2.8s to 1.1s
  • Client/server boundary rule: browser APIs = client component, everything else = server
