CLI-First Design: Why We Started Without a Dashboard

OCCP's first interface was a command-line tool, not a web dashboard. This was a deliberate architectural decision: building the CLI first forced clean API design, enabled automation from day one, and produced faster iteration cycles. The dashboard came later, and it was better for the wait.

The Temptation of the Dashboard

When building a new platform, there is enormous pressure to build a dashboard first. Dashboards are demonstrable: they produce screenshots for investor decks, demos for sales teams, and visual proof of progress. A terminal window running occp run --task "hello" does not have the same visual impact as a polished web interface with pipeline visualizations and real-time status indicators.

But dashboards are expensive. A minimal dashboard requires: a frontend framework (React, Vue, or similar), a design system (colors, typography, spacing, components), API integration (HTTP client, state management, error handling), authentication UI (login form, token storage, session management), and deployment infrastructure (build pipeline, static hosting or SSR). Each of these is a multi-day effort. For an early-stage platform, this effort diverts engineering time from the core product: the governance pipeline.

A dashboard is a rendering of your API. If the API is not right, the dashboard renders the wrong thing beautifully.

What CLI-First Forces

Clean API Separation

A CLI is a thin layer over an internal API. The occp run command parses arguments, calls the orchestrator's run() method, and formats the result for terminal display. If the CLI can do everything the platform supports, it proves that the internal API is complete and well-structured. If the CLI requires workarounds or special access to internal state, the API has gaps.

# CLI command structure — thin layer over internal API
import click

@click.group()
def cli():
    """OCCP — OpenCloud Control Plane CLI"""
    pass

@cli.command()
@click.option("--task", required=True, help="Task description")
@click.option("--agent", default="default", help="Agent to execute the task")
@click.option("--token-budget", default=4096, type=int, help="Maximum token budget")
@click.option("--timeout", default=60, type=int, help="Execution timeout in seconds")
def run(task: str, agent: str, token_budget: int, timeout: int):
    """Submit a task to the Verified Autonomy Pipeline."""
    orchestrator = get_orchestrator()
    result = orchestrator.run(Task(
        description=task,
        agent=agent,
        token_budget=token_budget,
        timeout=timeout,
    ))
    render_result(result)

This command adds exactly one layer of functionality: argument parsing and output formatting. All business logic lives in the orchestrator. When the REST API was built in v0.2.0, the endpoint handler looked almost identical — different input source (HTTP request instead of CLI arguments), same orchestrator call, different output format (JSON instead of terminal formatting).

Automation Compatibility

A CLI is automatable by default. Shell scripts, cron jobs, CI/CD pipelines, and other automation tools can invoke CLI commands without any integration effort. A dashboard is interactive by default: it requires a human to click buttons and read displays. By building the CLI first, OCCP was automatable from the first release.

# Automation example: scheduled task execution
#!/bin/bash
# Run nightly report generation through OCCP
RESULT=$(occp run --task "Generate daily metrics summary" --agent summarizer --timeout 120)
STATUS=$(echo "$RESULT" | jq -r '.status')

if [ "$STATUS" != "completed" ]; then
    echo "OCCP task failed: $RESULT" | mail -s "Daily Report Failure" ops@company.com
    exit 1
fi

echo "$RESULT" | jq -r '.output' > /var/reports/daily-$(date +%F).md

This kind of integration is trivial with a CLI and difficult with a dashboard. The CLI's structured output (JSON by default, with human-readable formatting as an option) makes it a first-class component in shell pipelines.

Faster Iteration

Modifying a CLI command takes minutes. Modifying a dashboard page takes hours. Adding a new --verbose flag to occp run is a one-line change to the argument parser and a few lines in the output formatter. Adding the equivalent "verbose mode" toggle to a dashboard requires: a UI component, state management, API parameter passing, and visual design for the additional output. The CLI's lower surface area means faster iteration on the core platform's capabilities.

The OCCP CLI Commands

occp run

The run command submits a task to the full Verified Autonomy Pipeline. It accepts a task description, agent selection, resource limits, and output format preferences. The command blocks until the pipeline completes (or until the timeout is reached) and returns the complete result including the audit trail entry ID.

occp status

The status command queries the current state of a task. For completed tasks, it returns the final status and audit reference. For running tasks (in future async versions), it returns the current pipeline stage and elapsed time. For denied tasks, it returns the denial reason and the denying policy.

$ occp status --task-id tsk_7f3a2e1d

Completed: 2025-12-02T14:30:24Z

occp audit

The audit command retrieves audit trail entries. It can query by task ID (all entries for a specific task), by time range (all entries within a date window), by action type (all gate denials, all execution failures), or by agent (all entries for a specific agent). The output includes chain verification status.

From CLI to REST API

When v0.2.0 introduced the FastAPI REST API, the transition was straightforward. Each CLI command mapped directly to an API endpoint. The internal orchestrator API, which the CLI had already validated through months of use, became the foundation for the REST layer.

# CLI → REST mapping
# occp run   → POST /api/v1/tasks
# occp status → GET  /api/v1/tasks/{task_id}
# occp audit  → GET  /api/v1/audit?task_id={task_id}
#              → GET  /api/v1/audit?since={date}&action_type={type}

The REST API endpoints accepted the same parameters as the CLI commands (as JSON instead of CLI flags), called the same orchestrator methods, and returned the same data (as JSON instead of formatted terminal output). The API design was not invented for the REST layer; it was discovered through CLI usage.

The Dashboard Transition

The web dashboard, introduced in v0.3.0, consumed the REST API without any backend changes. The API already supported every operation the dashboard needed because every operation had already been validated through the CLI. The dashboard added visual representations (pipeline progress indicators, audit trail timelines, policy evaluation trees) but no new backend capabilities.

This is the payoff of CLI-first design: the dashboard is purely a visualization layer. It does not contain business logic, does not call internal APIs, and does not bypass the REST layer. If the dashboard disappears, every capability is still available through the CLI. This architectural property — the dashboard as a non-critical visualization layer — is a direct consequence of building the CLI first.

When CLI-First Is Wrong

CLI-first is not universally correct. Products where the primary value is visual (design tools, data visualization, video editing) should start with the GUI. Products where the target user is non-technical (business analysts, marketing teams) may need a GUI from day one. But for developer infrastructure — CI/CD, deployment, orchestration, governance — CLI-first produces better architecture. OCCP is developer infrastructure, and the CLI-first approach served it well.

// Key Takeaways

  • CLI-first forces clean API separation: if the CLI works, the internal API is complete
  • CLIs are automatable by default — shell scripts, cron, and CI/CD can use them immediately
  • CLI iteration is faster than dashboard iteration: minutes vs. hours per change
  • Three core commands: occp run, occp status, occp audit cover the full platform surface
  • The REST API (v0.2.0) mapped directly from CLI commands — design discovered through usage, not invented
  • The dashboard (v0.3.0) is a pure visualization layer with no unique backend capabilities
