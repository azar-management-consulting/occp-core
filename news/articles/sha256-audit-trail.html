SHA-256 Hash Chains: Building a Tamper-Evident Audit Trail

Every action in OCCP produces an audit entry. Each entry contains the SHA-256 hash of the previous entry, forming an append-only chain where any modification to historical records is cryptographically detectable. This is how OCCP builds trust in autonomous AI systems.

Why Simple Logging Is Not Enough

Most software systems log events to files or databases. These logs record what happened, when, and sometimes why. For debugging and monitoring, simple logging is sufficient. For governance, it is not.

The problem with simple logs is that they are mutable. An administrator with database access can modify a log entry, delete entries, or insert fabricated entries. A compromised system can alter its own logs to conceal malicious activity. In a legal or regulatory context, a simple log cannot prove that it has not been tampered with. It can show you what it currently contains, but it cannot prove that its contents are the same as what was originally recorded.

A log that can be modified without detection is not evidence. It is a narrative that anyone with write access can revise.

For AI governance, this limitation is critical. When an AI agent takes an action — sends an email, modifies a database, generates a document — the audit trail must be able to prove, retroactively, that the recorded action is what actually happened. This is not a theoretical requirement; it is a legal one under frameworks like the EU AI Act.

Hash Chain Architecture

A hash chain is a sequence of records where each record includes a cryptographic hash of the previous record. This creates a dependency chain: to verify that record N has not been modified, you recompute its hash and compare it to the hash stored in record N+1. To verify the entire chain, you walk it from the beginning, recomputing and comparing hashes at each step.

# Hash chain entry structure
@dataclass
class AuditEntry:
    entry_id: str              # Unique identifier
    timestamp: str             # ISO 8601 timestamp
    task_id: str               # Associated task
    action_type: str           # plan, gate, execute, validate, ship
    actor: str                 # Agent or user identifier
    resource: str              # What was acted upon
    result: str                # Action outcome
    details: dict              # Stage-specific data
    previous_hash: str         # SHA-256 hash of previous entry
    entry_hash: str            # SHA-256 hash of this entry

Hash Computation

The entry hash is computed over a canonical representation of all fields except the hash itself. This includes the previous hash, which chains the entry to its predecessor. The canonical representation is a deterministic JSON serialization with sorted keys and no whitespace, ensuring that the same data always produces the same hash.

import hashlib
import json

def compute_entry_hash(entry: AuditEntry) -> str:
    """Compute SHA-256 hash of an audit entry."""
    # Canonical representation: all fields except entry_hash
    canonical = {
        "entry_id": entry.entry_id,
        "timestamp": entry.timestamp,
        "task_id": entry.task_id,
        "action_type": entry.action_type,
        "actor": entry.actor,
        "resource": entry.resource,
        "result": entry.result,
        "details": entry.details,
        "previous_hash": entry.previous_hash,
    }
    # Deterministic serialization
    serialized = json.dumps(canonical, sort_keys=True, separators=(",", ":"))
    return hashlib.sha256(serialized.encode("utf-8")).hexdigest()

SHA-256 was chosen for several reasons: it is a widely-accepted cryptographic hash function with no known practical collision attacks, it is included in Python's standard library (no external dependencies), and it produces a 64-character hexadecimal string that is compact enough for storage and display.

Tamper Detection

The hash chain enables tamper detection through chain verification. The verification algorithm walks the chain from the genesis entry (the first entry, whose previous hash is a known constant) to the most recent entry, recomputing each hash and comparing it to the stored value.

def verify_chain(entries: list[AuditEntry]) -> ChainVerification:
    """Verify the integrity of the entire audit chain."""
    if not entries:
        return ChainVerification(valid=True, entries_checked=0)

    # Genesis entry: previous_hash must be the known constant
    GENESIS_HASH = "0" * 64
    if entries[0].previous_hash != GENESIS_HASH:
        return ChainVerification(valid=False, broken_at=0, reason="Invalid genesis")

    for i, entry in enumerate(entries):
        # Recompute the hash
        expected_hash = compute_entry_hash(entry)
        if expected_hash != entry.entry_hash:
            return ChainVerification(
                valid=False, broken_at=i,
                reason=f"Entry {entry.entry_id}: hash mismatch"
            )

        # Verify chain linkage (except for genesis)
        if i > 0 and entry.previous_hash != entries[i - 1].entry_hash:
            return ChainVerification(
                valid=False, broken_at=i,
                reason=f"Entry {entry.entry_id}: chain link broken"
            )

    return ChainVerification(valid=True, entries_checked=len(entries))

What Tampering Looks Like

If an attacker modifies any field of entry N, its recomputed hash will differ from the stored hash. The verification algorithm detects this immediately. If the attacker also updates entry N's stored hash to match the modified content, the chain breaks at entry N+1, because N+1's previous_hash no longer matches N's hash. To fix this, the attacker would need to update N+1's previous_hash, which changes N+1's content, which changes N+1's hash, which breaks the chain at N+2. The cascade continues to the end of the chain.

In other words, to tamper with a single entry without detection, the attacker must rewrite every subsequent entry in the chain. This is computationally trivial for a local chain, which is why OCCP's design includes additional tamper-resistance measures: periodic chain snapshots with external hash publication, and support for external witness services in enterprise deployments.

Data Stored Per Entry

Each audit entry records the complete context of an action in the Verified Autonomy Pipeline:

  • entry_id — UUID v4 identifier, unique across the entire audit trail
  • timestamp — ISO 8601 UTC timestamp with millisecond precision
  • task_id — The task this entry belongs to, linking related entries
  • action_type — The pipeline stage: plan, gate_pass, gate_deny, execute, validate_pass, validate_fail, ship
  • actor — The agent or user that performed the action
  • resource — What was acted upon (model endpoint, file path, API target)
  • result — The action outcome (success, failure, denial, with reason)
  • details — Stage-specific structured data (policy evaluation results, token usage, validation findings)
  • previous_hash — SHA-256 hash of the previous entry in the chain
  • entry_hash — SHA-256 hash of this entry's canonical representation

A single task typically produces 5-8 audit entries: one for planning, one or more for gate evaluation (one per policy check), one for execution, one for validation, and one for shipping. A denied task produces fewer entries — the chain records exactly where the pipeline stopped.

EU AI Act Compliance

Article 12: Record-Keeping

Article 12 of the EU AI Act requires high-risk AI systems to have "automatic recording of events" that enables tracing the system's operation throughout its lifecycle. The OCCP audit chain directly implements this requirement. Every action, from task submission through result delivery, is automatically recorded with a complete data set that enables reconstruction of the system's decision process.

What Article 12 Requires

The regulation specifies that records must include: the identification of the AI system and its version, dates and times of operation, input data that triggered the system's operation, the system's output, and the persons involved in the operation. The OCCP audit entry structure maps directly to these requirements: entry_id and actor for identification, timestamp for dates and times, details for input data, result for output, and the task chain for operational reconstruction.

Tamper Evidence as a Compliance Feature

While Article 12 does not explicitly require tamper-evident logging, the integrity of audit records is implicitly required for their evidentiary value. A record that cannot prove it has not been modified has limited value in a regulatory investigation. The SHA-256 hash chain provides the mathematical guarantee that records are unmodified since creation, strengthening the evidentiary value of the audit trail for regulatory compliance and legal proceedings.

Performance Characteristics

SHA-256 computation is fast: hashing a typical audit entry (1-5 KB of canonical JSON) takes microseconds on modern hardware. The hash chain adds negligible overhead to pipeline execution. Chain verification, which walks the entire chain, scales linearly with the number of entries. For the expected audit volume (thousands to tens of thousands of entries per day), full-chain verification completes in milliseconds. For very large chains, incremental verification (checking only new entries since the last verified point) reduces verification time to constant regardless of total chain length.
