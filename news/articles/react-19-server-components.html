React 19 Server Components in AI Dashboards

AI operations dashboards must serve two masters: fast initial loads for operators checking system status, and real-time interactivity for live monitoring. React 19 server components let us achieve both by rendering data-heavy views on the server while keeping interactive elements client-side.

The Dashboard Performance Problem

Before the React 19 migration, the OCCP dashboard was a fully client-rendered React application. On every page load, the browser downloaded the JavaScript bundle (247KB gzipped), parsed and executed it, then made API calls to fetch task lists, agent status, and pipeline states. Only after all three API responses returned could the dashboard render meaningful content. The total time from navigation to first meaningful paint was 2.8 seconds on a broadband connection.

For an AI operations dashboard, 2.8 seconds is not just slow -- it is a trust problem. When an operator opens the dashboard to check if an agent is behaving correctly, every second of loading time increases anxiety. The operator does not know if the system is down, if the dashboard is broken, or if it is simply loading. Server components eliminate most of this latency by moving data fetching to the server, where the API is on the same network (or the same host).

Server Components for Initial Data Loading

React 19 server components run on the server at request time. They can directly call APIs, query databases, or read files, then send the rendered HTML to the client. No JavaScript bundle is shipped for server components -- they are pure HTML by the time they reach the browser.

The Dashboard Page Structure

// app/dashboard/page.tsx (Server Component)
import { TaskList } from "./components/TaskList";
import { AgentPanel } from "./components/AgentPanel";
import { PipelineOverview } from "./components/PipelineOverview";
import { fetchTasks, fetchAgents, fetchPipelines } from "@/lib/api";

export default async function DashboardPage() {
  // These fetch calls run on the server
  // Latency is ~2ms (localhost) instead of ~150ms (browser to API)
  const [tasks, agents, pipelines] = await Promise.all([
    fetchTasks({ limit: 20, status: "running" }),
    fetchAgents({ status: "active" }),
    fetchPipelines({ active: true }),
  ]);

  return (
    <div className="dashboard-grid">
      <TaskList initialTasks={tasks} />
      <AgentPanel initialAgents={agents} />
      <PipelineOverview initialPipelines={pipelines} />
    </div>
  );
}

The three API calls run in parallel on the server using Promise.all. Because the Next.js server and the FastAPI backend are on the same host, each call completes in approximately 2ms. The total server-side data fetching time is ~5ms, compared to 400-600ms when the same calls run from a browser. The rendered HTML is streamed to the client as it becomes available.

Client Components for Interactivity

Not everything can be a server component. Interactive elements -- forms, WebSocket connections, event handlers, anything using browser APIs -- must be client components. In the OCCP dashboard, the boundary is clear:

// Server Components (no "use client" directive)
// - DashboardPage: initial data loading
// - TaskRow: static display of task data
// - AgentCard: static display of agent info
// - PipelineStageBar: initial rendering of stages

// Client Components ("use client" required)
// - TaskActions: buttons for cancel, retry, reassign
// - LivePipelineStatus: WebSocket updates for active pipelines
// - AgentHeartbeatIndicator: animated status dot
// - TaskFilter: search and filter controls
// - AuthGuard: authentication state management

The TaskList Component: Hybrid Architecture

The TaskList demonstrates the hybrid approach. The list container and initial task data are server-rendered. Each task row contains a client component for interactive actions:

// components/TaskList.tsx (Server Component)
import { TaskActions } from "./TaskActions";
import { LiveStatusBadge } from "./LiveStatusBadge";
import type { Task } from "@/types";

interface TaskListProps {
  initialTasks: Task[];
}

export function TaskList({ initialTasks }: TaskListProps) {
  return (
    <div className="task-list">
      <h2>Active Tasks</h2>
      <table>
        <thead>
          <tr>
            <th>ID</th><th>Description</th>
            <th>Status</th><th>Actions</th>
          </tr>
        </thead>
        <tbody>
          {initialTasks.map((task) => (
            <tr key={task.id}>
              <td><code>{task.id}</code></td>
              <td>{task.description}</td>
              <td><LiveStatusBadge
                    taskId={task.id}
                    initialStatus={task.status} /></td>
              <td><TaskActions taskId={task.id} /></td>
            </tr>
          ))}
        </tbody>
      </table>
    </div>
  );
}
// components/LiveStatusBadge.tsx (Client Component)
"use client";

import { useState, useEffect } from "react";

interface LiveStatusBadgeProps {
  taskId: string;
  initialStatus: string;
}

export function LiveStatusBadge({ taskId, initialStatus }: LiveStatusBadgeProps) {
  const [status, setStatus] = useState(initialStatus);

  useEffect(() => {
    const ws = new WebSocket(`wss://api.occp.ai/ws/task/${taskId}`);
    ws.onmessage = (event) => {
      const data = JSON.parse(event.data);
      if (data.type === "status_change") {
        setStatus(data.status);
      }
    };
    return () => ws.close();
  }, [taskId]);

  return <span className={`status-badge status-${status}`}>{status}</span>;
}

The server renders the task table with the initial status. The browser hydrates only the LiveStatusBadge and TaskActions components, establishing WebSocket connections for real-time updates. The table structure, task descriptions, and IDs are pure HTML -- no JavaScript needed for their display.

Streaming SSR for Faster Time-to-Interactive

Next.js 15 streams the server-rendered HTML to the browser as it becomes ready. Combined with React 19's Suspense boundaries, this means the browser can begin rendering content before all data has loaded:

// app/dashboard/page.tsx with streaming
import { Suspense } from "react";

export default async function DashboardPage() {
  return (
    <div className="dashboard-grid">
      <Suspense fallback={<TaskListSkeleton />}>
        <TaskListAsync />
      </Suspense>
      <Suspense fallback={<AgentPanelSkeleton />}>
        <AgentPanelAsync />
      </Suspense>
      <Suspense fallback={<PipelineOverviewSkeleton />}>
        <PipelineOverviewAsync />
      </Suspense>
    </div>
  );
}

// Each async component fetches its own data independently
async function TaskListAsync() {
  const tasks = await fetchTasks({ limit: 20, status: "running" });
  return <TaskList initialTasks={tasks} />;
}

Each Suspense boundary is independent. If the task list API responds in 5ms but the agent list takes 200ms, the browser renders the task list immediately and shows a skeleton for the agent panel. The operator sees the task data first, the agent data second, and the pipeline overview third -- each appearing as its data becomes available.

Bundle Size Reduction

Server components do not ship JavaScript to the browser. This has a direct impact on bundle size. In the OCCP dashboard, the server component migration reduced the initial JavaScript bundle from 247KB to 198KB (gzipped) -- a 20% reduction. The savings come from three areas:

  • Data fetching code -- API client configuration, response parsing, and error handling for initial data loads moved to the server.
  • Data transformation -- Sorting, filtering, and formatting of task lists and agent data for initial display is now server-side.
  • Component code -- The structural components (table layouts, card containers, heading elements) are server-rendered HTML, not client-side React components.

The remaining 198KB consists entirely of interactive components: the authentication layer, WebSocket managers, form controls, and real-time status indicators. These are genuinely client-side concerns that cannot be server-rendered.

The Challenge: Server State + WebSocket Real-Time

The hardest architectural problem in the OCCP dashboard is combining server-rendered initial state with WebSocket real-time updates. The server renders a task with status "running." Milliseconds later, the WebSocket receives a status change to "completed." The component must transition smoothly from server-rendered state to client-managed state.

// The handoff pattern
"use client";

import { useState, useEffect, useRef } from "react";

export function LivePipelineStatus({ taskId, initialStages }) {
  const [stages, setStages] = useState(initialStages);
  const wsRef = useRef<WebSocket | null>(null);

  useEffect(() => {
    // Connect WebSocket after hydration
    wsRef.current = new WebSocket(
      `wss://api.occp.ai/ws/pipeline/${taskId}`
    );

    wsRef.current.onmessage = (event) => {
      const data = JSON.parse(event.data);
      if (data.type === "stage_transition") {
        setStages((prev) =>
          prev.map((stage) =>
            stage.name === data.to_stage
              ? { ...stage, status: "active" }
              : stage.name === data.from_stage
              ? { ...stage, status: "completed", duration_ms: data.duration_ms }
              : stage
          )
        );
      }
    };

    return () => wsRef.current?.close();
  }, [taskId]);

  return (
    <div className="pipeline-stages">
      {stages.map((stage) => (
        <div key={stage.name} className={`stage ${stage.status}`}>
          {stage.name}
          {stage.duration_ms &&
            <span className="duration">{stage.duration_ms}ms</span>}
        </div>
      ))}
    </div>
  );
}

The initialStages prop comes from the server component. It is the state at request time. The useState(initialStages) initializes client state from server data. The WebSocket onmessage handler updates client state from that point forward. The transition is seamless because React's reconciliation handles the DOM update regardless of whether the state came from the server or the WebSocket.

Performance Results

After the React 19 migration, the OCCP dashboard performance improved measurably:

  • Time to first meaningful paint: 2.8s to 1.1s (61% improvement)
  • Time to interactive: 3.4s to 1.8s (47% improvement)
  • JavaScript bundle size: 247KB to 198KB gzipped (20% reduction)
  • Lighthouse performance score: 62 to 84

These improvements come entirely from architectural changes -- no code optimization, no lazy loading tricks, no compression changes. Server components simply eliminate unnecessary work from the browser's critical path.

// Key Takeaways

  • Server components move data fetching to the server where API latency is ~2ms instead of ~150ms
  • The client/server boundary follows browser API dependency: WebSocket, localStorage = client
  • Streaming SSR with Suspense boundaries renders each panel independently as data arrives
  • Bundle size reduced 20% by eliminating server-only code from the browser bundle
  • Server-to-client state handoff uses useState(initialProps) with WebSocket takeover
  • Time to first meaningful paint improved from 2.8s to 1.1s with no code-level optimization
