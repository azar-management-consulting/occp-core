Designing the Verified Autonomy Pipeline: First Principles

The Verified Autonomy Pipeline (VAP) is OCCP's core architectural concept: a five-stage pipeline where every AI action is planned, policy-checked, sandboxed, validated, and audited before reaching production. This article details the design decisions behind each stage.

The Pipeline as an Invariant

Most AI frameworks treat processing pipelines as flexible sequences: you can add stages, remove stages, reorder stages. This flexibility is a feature for orchestration, but a liability for governance. If a developer can remove the policy-checking stage, governance is advisory, not enforced.

The VAP is designed as an invariant, not a configurable pipeline. The five stages always execute in order. Stages cannot be removed, skipped, or reordered. Individual stages can be extended with additional checks, but the base pipeline is immutable. This design choice means that governance is a structural property of the system, not a runtime configuration.

A governance pipeline that can be bypassed is not a governance pipeline. It is a suggestion.

Stage 1: Plan

Inputs and Outputs

The Plan stage receives a raw task description from the user or calling system. Its output is a structured TaskPlan object containing: the decomposed subtasks, the selected execution strategy, a risk classification (low, medium, high, critical), estimated resource requirements (token budget, timeout), and the set of capabilities required to execute the task.

# TaskPlan structure
{
  "task_id": "tsk_a1b2c3",
  "subtasks": [
    {"id": "sub_01", "action": "analyze_input", "risk": "low"},
    {"id": "sub_02", "action": "generate_response", "risk": "medium"}
  ],
  "strategy": "sequential",
  "risk_level": "medium",
  "token_budget": 4096,
  "timeout_seconds": 30,
  "required_capabilities": ["llm_call", "text_analysis"]
}

Task Decomposition

Complex tasks are decomposed into atomic subtasks, each with its own risk classification. This decomposition serves two purposes. First, it enables granular policy evaluation: a task with one high-risk subtask and five low-risk subtasks does not need blanket high-risk treatment. Second, it enables partial execution: if the high-risk subtask is denied at the Gate, the low-risk subtasks can still proceed.

Strategy Selection

The Plan stage selects an execution strategy based on the task structure: sequential for ordered subtasks with dependencies, parallel for independent subtasks that can execute concurrently, or conditional for subtasks where later stages depend on the output of earlier stages. The strategy determines how the Execute stage processes the subtask list.

Stage 2: Gate

Policy Evaluation

The Gate stage evaluates every subtask against the active policy set. Policies are defined as code, not configuration, and follow a deny-by-default model: a subtask must pass all policies to proceed. If any policy denies the subtask, execution halts and the denial reason is recorded in the audit trail.

# Gate evaluation pseudocode
def gate_evaluate(subtask, policies):
    results = []
    for policy in policies:
        result = policy.evaluate(subtask)
        results.append(result)
        if result.denied:
            audit_log("GATE_DENIED", subtask, policy, result.reason)
            return GateResult(allowed=False, reason=result.reason)
    audit_log("GATE_PASSED", subtask, results)
    return GateResult(allowed=True)

PII Scanning

The Gate runs PII detection on all input text before it reaches the LLM. The initial implementation uses regex-based pattern matching for email addresses, phone numbers, social security numbers, and credit card numbers. Each detected PII element triggers a policy evaluation: the PII policy can deny the request, redact the PII, or allow it with a warning depending on the configured sensitivity level.

Injection Detection

Prompt injection detection runs as a separate Gate sub-stage. The detector scans for known injection patterns: role override attempts ("ignore all previous instructions"), delimiter-based attacks ("---\nSYSTEM:"), encoding-based evasion (base64-encoded instructions), and context manipulation. When an injection pattern is detected, the subtask is denied unconditionally — there is no sensitivity level for injection, only deny.

Stage 3: Execute

Sandboxed Execution

The Execute stage runs the approved subtask in an isolated environment. For LLM calls, this means a controlled API invocation with enforced token limits and timeout constraints. For code execution tasks, this means running the generated code in a sandboxed environment with restricted filesystem access, no network access, and memory limits.

The sandbox boundary is a critical security primitive. Code generated by an LLM is untrusted by definition: it has not been reviewed, tested, or verified. The Execute stage treats all LLM-generated artifacts as untrusted inputs, regardless of the model's reputation or the prompt's specificity.

Resource Enforcement

During execution, the system enforces the resource limits specified in the TaskPlan. If an LLM call exceeds its token budget, the call is terminated. If a code execution exceeds its timeout, the process is killed. If a subtask attempts to make more API calls than its plan allows, the additional calls are blocked. Resource enforcement is not advisory; it is a hard limit enforced by the Execute stage.

Stage 4: Validate

Output Verification

The Validate stage examines the output of the Execute stage before it proceeds to delivery. Validation includes: output PII scanning (to catch PII generated by the LLM, not just PII in the input), output format verification (does the output match the expected schema), constraint checking (does the output satisfy the task's stated requirements), and coherence checking (is the output internally consistent).

# Validation chain
validators = [
    OutputPIIScanner(),
    SchemaValidator(expected_schema),
    ConstraintChecker(task_constraints),
    CoherenceValidator()
]

for validator in validators:
    result = validator.validate(output)
    if not result.valid:
        audit_log("VALIDATE_FAILED", validator.name, result.issues)
        return ValidationResult(passed=False, issues=result.issues)

Constraint Checking

Task constraints are defined at the Plan stage and verified at the Validate stage. Examples include: maximum output length, required output format (JSON, markdown, plain text), prohibited content categories, and required content elements. Constraint checking ensures that the LLM's output meets the task's specifications, not just the model's preferences.

Stage 5: Ship

Audit Logging

The Ship stage writes a complete audit entry for the task, including: the original request, the task plan, all Gate evaluation results, the execution output, all validation results, the final result, and a SHA-256 hash linking this entry to the previous entry in the audit chain. The audit entry is immutable once written — it cannot be modified, deleted, or overwritten.

Result Delivery

After the audit entry is committed, the Ship stage delivers the validated result to the calling system. Delivery can take multiple forms: a direct API response, a webhook callback, a message queue event, or a file write. The delivery method is specified in the original task request and is itself subject to policy evaluation — a policy can restrict which delivery methods are available for different risk levels.

Failure Modes and Recovery

Each stage has defined failure modes. Plan failure (invalid task description) returns a 400-level error. Gate denial returns a 403-level response with the denial reason. Execute failure (timeout, API error) returns a 500-level error with retry eligibility. Validate failure (output does not meet constraints) returns the validation issues for correction. Ship failure (audit write error) blocks delivery until the audit entry is committed — results are never delivered without an audit record.

This failure mode design ensures that the pipeline degrades predictably. A failure at any stage produces a clear, actionable error message, and the audit trail records where and why the pipeline stopped.

// Key Takeaways

  • The VAP is an immutable five-stage pipeline: Plan, Gate, Execute, Validate, Ship — no stages can be skipped or reordered
  • Plan stage decomposes tasks into atomic subtasks with individual risk classifications and resource budgets
  • Gate stage enforces deny-by-default policies including PII detection and prompt injection blocking
  • Execute stage runs all LLM calls in sandboxed environments with hard resource limits
  • Validate stage verifies outputs against schema, constraints, and PII scanning before delivery
  • Ship stage commits an immutable SHA-256 audit entry before delivering any result
