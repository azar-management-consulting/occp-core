SDK Design Philosophy: Python and TypeScript Clients for Agent Systems

OCCP ships dual SDKs: a Python client built on stdlib urllib and a TypeScript client using native fetch. Both have zero external dependencies. This article explains why that constraint exists and how it influences every design decision.

Why Dual SDKs?

An AI orchestration platform lives at the intersection of two ecosystems. Data scientists, ML engineers, and backend developers work in Python. Frontend developers, full-stack engineers, and DevOps teams increasingly work in TypeScript. If OCCP only shipped a Python SDK, every TypeScript integration would require building a custom HTTP client. If we only shipped TypeScript, the Python ecosystem -- where most AI development happens -- would be underserved.

The decision to support both languages from v0.2.0 was deliberate. SDKs are not afterthoughts; they are the primary way developers interact with the platform. The quality and ergonomics of the SDK determine adoption velocity more than the quality of the underlying API.

The Zero-Dependency Constraint

Both SDKs have exactly zero external dependencies. The Python SDK uses only urllib.request from the standard library. The TypeScript SDK uses the native fetch API available in Node.js 18+ and all modern browsers. This is not a limitation -- it is a deliberate architectural choice with specific benefits:

  • No dependency conflicts. Enterprise environments often have strict dependency policies. A SDK that pulls in requests, httpx, or axios becomes a negotiation point with security teams. Zero dependencies means zero conflicts.
  • No supply chain risk. Every dependency is an attack surface. The npm and PyPI ecosystems have both experienced malicious package incidents. Our SDKs cannot be compromised through a transitive dependency because there are none.
  • Predictable behavior. Standard library HTTP clients have stable, well-documented behavior across runtime versions. Third-party libraries change APIs between major versions, requiring SDK updates unrelated to OCCP functionality.
  • Minimal install footprint. pip install occp-sdk downloads exactly one package. No dependency resolution, no version conflicts, no build steps.
The best dependency is the one you do not have. Every line of code you do not own is a liability. For an SDK that enterprise security teams must approve, zero dependencies is a feature, not a constraint.

Python SDK: stdlib urllib

The Python SDK wraps urllib.request with a typed, ergonomic interface. The raw urllib API is verbose and awkward -- our SDK abstracts it into method calls that mirror the REST API structure:

from occp_sdk import OCCPClient

client = OCCPClient(
    base_url="https://api.occp.ai",
    api_key="your-api-key"
)

# Create a task
task = client.tasks.create(
    description="Run security audit",
    priority="high"
)

# Check task status
status = client.tasks.get(task.id)
print(f"Stage: {status.pipeline_stage}")

# List agents
agents = client.agents.list(status="active")

Async Support Without asyncio Libraries

The Python SDK provides async methods using asyncio and urllib.request wrapped in asyncio.to_thread. This approach avoids adding aiohttp or httpx as dependencies while still supporting async/await patterns:

import asyncio
from occp_sdk import AsyncOCCPClient

async def main():
    client = AsyncOCCPClient(
        base_url="https://api.occp.ai",
        api_key="your-api-key"
    )
    task = await client.tasks.create(
        description="Run security audit"
    )
    # Polling with async
    while task.status == "running":
        await asyncio.sleep(2)
        task = await client.tasks.get(task.id)

asyncio.run(main())

The to_thread approach has a performance trade-off: each HTTP call runs in a thread pool rather than using non-blocking I/O. For the typical OCCP use case -- submitting tasks and polling status -- this overhead is negligible. If a user needs true non-blocking HTTP at high concurrency, they can use the REST API directly with their preferred async HTTP library.

TypeScript SDK: Native Fetch

The TypeScript SDK targets the fetch API, which became stable in Node.js 18 and has been available in browsers for years. The SDK provides full type definitions, making every response field autocomplete-friendly in VS Code and other TypeScript-aware editors:

import { OCCPClient } from "@occp/sdk";

const client = new OCCPClient({
  baseUrl: "https://api.occp.ai",
  apiKey: "your-api-key",
});

// Create a task - fully typed
const task = await client.tasks.create({
  description: "Analyze code quality",
  priority: "normal",
});

// task.id, task.status, task.pipelineStage are all typed
console.log(`Task ${task.id}: ${task.status}`);

Type Safety from API Schema

The TypeScript types are generated from the FastAPI OpenAPI schema. This means the SDK types always match the server's actual response format. If a field is added to the API response, the TypeScript types are regenerated and consumers get compile-time awareness of the new field:

// Generated from OpenAPI schema
interface TaskResponse {
  id: string;
  status: "pending" | "running" | "completed" | "failed" | "cancelled";
  createdAt: string;
  pipelineStage: string | null;
  auditHash: string;
}

// Discriminated union for type-safe status handling
type TaskResult =
  | { status: "completed"; result: unknown }
  | { status: "failed"; error: string }
  | { status: "running"; pipelineStage: string };

The field naming follows language conventions: pipeline_stage in Python becomes pipelineStage in TypeScript. The SDK handles this transformation transparently during serialization and deserialization.

Shared Design Principles

Error Handling

Both SDKs throw typed exceptions that map to HTTP status codes. The exception hierarchy is identical across languages:

# Python
from occp_sdk.exceptions import (
    OCCPError,           # Base class
    AuthenticationError, # 401
    ForbiddenError,      # 403
    NotFoundError,       # 404
    ValidationError,     # 422
    RateLimitError,      # 429
    ServerError,         # 500+
)

try:
    task = client.tasks.get("nonexistent-id")
except NotFoundError as e:
    print(f"Task not found: {e.request_id}")
// TypeScript
import {
  OCCPError,
  AuthenticationError,
  NotFoundError,
  RateLimitError,
} from "@occp/sdk";

try {
  const task = await client.tasks.get("nonexistent-id");
} catch (e) {
  if (e instanceof NotFoundError) {
    console.error(`Task not found: ${e.requestId}`);
  }
}

Every exception includes the request_id from the server response, enabling correlation between client errors and server logs. This is essential for debugging in distributed systems where the client and server are operated by different teams.

Automatic Retry

Both SDKs implement exponential backoff retry for transient errors (429 Too Many Requests, 502 Bad Gateway, 503 Service Unavailable). The retry behavior is configurable:

client = OCCPClient(
    base_url="https://api.occp.ai",
    api_key="your-api-key",
    max_retries=3,
    retry_delay=1.0,  # seconds, doubles each retry
)

The retry logic respects the Retry-After header when present. For 429 responses, the server tells the client exactly when to retry. The SDK uses that value instead of its default backoff, preventing unnecessary wait times.

SDK Method Mapping

Every SDK method maps 1:1 to a REST API endpoint. The mapping is documented in both the SDK README and the OpenAPI schema:

| SDK Method                    | HTTP Endpoint                      |
|-------------------------------|-------------------------------------|
| client.tasks.create()         | POST   /api/v1/tasks               |
| client.tasks.get(id)          | GET    /api/v1/tasks/{id}           |
| client.tasks.list()           | GET    /api/v1/tasks                |
| client.tasks.update(id, ...)  | PATCH  /api/v1/tasks/{id}           |
| client.agents.register()      | POST   /api/v1/agents              |
| client.agents.heartbeat(id)   | POST   /api/v1/agents/{id}/heartbeat|
| client.pipeline.execute(id)   | POST   /api/v1/tasks/{id}/pipeline  |
| client.pipeline.status(id)    | GET    /api/v1/tasks/{id}/pipeline  |

This 1:1 mapping means developers familiar with the REST API can predict SDK method names, and SDK users can read the API documentation interchangeably. There are no "convenience" methods that combine multiple API calls -- composition is left to the user.

How SDK Design Influences Adoption

The SDK is the first code a developer writes when evaluating OCCP. If the "hello world" example requires installing 15 transitive dependencies, configuring a build step, or reading 200 lines of documentation, evaluation stops. Our zero-dependency, type-safe, copy-paste-friendly SDK design means the time from pip install to first API call is under 60 seconds. That speed compounds: every developer who completes the first API call is more likely to complete the second one.

// Key Takeaways

  • Dual SDKs serve both Python (ML/backend) and TypeScript (frontend/fullstack) ecosystems
  • Zero external dependencies eliminates supply chain risk and enterprise approval friction
  • Python SDK uses stdlib urllib with asyncio.to_thread for async support
  • TypeScript SDK uses native fetch with full type definitions generated from OpenAPI
  • Typed exception hierarchies with request_id enable cross-boundary debugging
  • 1:1 method-to-endpoint mapping keeps the SDK predictable and documentation interchangeable
