Agent Scheduling and Concurrency Control in Distributed AI Systems

When multiple AI agents operate concurrently, they compete for shared resources: LLM API rate limits, compute capacity, and data access. OCCP's scheduler coordinates agent execution to prevent resource contention, enforce priority ordering, and maintain pipeline governance across concurrent workflows.

The Concurrency Problem in AI Orchestration

Single-agent systems are straightforward: one task enters the pipeline, passes through all five stages, and exits. Concurrency introduces complexity at every level. Two agents making simultaneous LLM API calls can exceed rate limits. Two agents writing to the same audit chain can create ordering conflicts. Two agents processing related tasks can produce inconsistent results if their execution interleaves incorrectly.

Traditional concurrency control (mutexes, semaphores, database locks) solves part of the problem. But AI workloads have unique characteristics that require specialized scheduling: LLM API calls have variable latency (200ms to 30s depending on output length), token consumption is unpredictable until execution completes, and rate limits are typically measured in requests-per-minute and tokens-per-minute simultaneously.

The hardest concurrency problem in AI orchestration is not thread safety. It is resource contention against external APIs with opaque rate limiting and variable response times.

Resource-Aware Scheduling

OCCP's scheduler maintains a resource model that tracks available capacity across multiple dimensions. Each LLM provider has a capacity profile: maximum concurrent requests, maximum tokens per minute, and maximum requests per minute. The scheduler admits tasks to the Execute stage only when sufficient capacity is available across all dimensions.

# Resource capacity model
@dataclass
class ProviderCapacity:
    provider: str
    max_concurrent: int        # Max simultaneous requests
    max_rpm: int               # Max requests per minute
    max_tpm: int               # Max tokens per minute
    current_concurrent: int = 0
    requests_this_minute: int = 0
    tokens_this_minute: int = 0

    def can_admit(self, estimated_tokens: int) -> bool:
        """Check if a new request can be admitted."""
        if self.current_concurrent >= self.max_concurrent:
            return False
        if self.requests_this_minute >= self.max_rpm:
            return False
        if self.tokens_this_minute + estimated_tokens > self.max_tpm:
            return False
        return True

Token Estimation

Accurate token estimation is critical for resource-aware scheduling. Over-estimating wastes capacity (the scheduler reserves more tokens than needed, reducing throughput). Under-estimating risks rate limit violations. OCCP uses a conservative estimation strategy: the task plan's token budget is used as the estimate, which is always an upper bound on actual consumption. After execution completes, actual token usage is recorded and the capacity model is updated.

Sliding Window Rate Tracking

Rate limits are tracked using a sliding window algorithm rather than fixed minute boundaries. A fixed boundary creates burst behavior: if all requests arrive in the last 5 seconds of a minute, the first 5 seconds of the next minute are also saturated, appearing as a double-burst to the provider. The sliding window smooths request distribution by tracking the exact timestamp of each request and counting only those within the trailing 60-second window.

class SlidingWindowTracker:
    def __init__(self, window_seconds: int = 60):
        self.window = window_seconds
        self.timestamps: list[float] = []

    def count(self) -> int:
        """Count requests within the current window."""
        now = time.monotonic()
        cutoff = now - self.window
        # Remove expired entries
        self.timestamps = [t for t in self.timestamps if t > cutoff]
        return len(self.timestamps)

    def record(self):
        """Record a new request."""
        self.timestamps.append(time.monotonic())

Task Queue Design

Tasks waiting for execution are held in a priority queue. The queue is not a simple FIFO; it considers multiple factors when selecting the next task to execute.

Priority Levels

OCCP defines four priority levels: critical (system operations, security responses), high (user-initiated interactive tasks), normal (standard batch processing), and low (background tasks, maintenance). Higher-priority tasks are always dequeued before lower-priority tasks, regardless of arrival order. Within the same priority level, tasks are processed in FIFO order.

# Priority queue with multi-level ordering
import heapq
from enum import IntEnum

class Priority(IntEnum):
    CRITICAL = 0  # Lowest number = highest priority
    HIGH = 1
    NORMAL = 2
    LOW = 3

class TaskQueue:
    def __init__(self):
        self._queue: list[tuple[int, int, Task]] = []
        self._counter = 0  # Tie-breaker for FIFO within priority

    def enqueue(self, task: Task, priority: Priority = Priority.NORMAL):
        heapq.heappush(self._queue, (priority.value, self._counter, task))
        self._counter += 1

    def dequeue(self) -> Task | None:
        if self._queue:
            _, _, task = heapq.heappop(self._queue)
            return task
        return None

Queue Backpressure

When the queue exceeds a configurable depth threshold, the scheduler applies backpressure by rejecting new task submissions with a 429 (Too Many Requests) response. This prevents unbounded queue growth during sustained overload. The backpressure threshold is configurable per deployment: a high-throughput deployment might allow 10,000 queued tasks, while a resource-constrained deployment might limit to 100.

Concurrency Limits Per Agent Type

Different agent types have different resource requirements and operational characteristics. A code generation agent that produces long outputs consumes significantly more tokens per request than a classification agent that produces single-word responses. OCCP allows per-agent-type concurrency limits that restrict how many instances of a given agent type can execute simultaneously.

# Per-agent concurrency configuration
agent_limits = {
    "code_generator": {"max_concurrent": 3, "token_multiplier": 2.0},
    "classifier": {"max_concurrent": 20, "token_multiplier": 0.5},
    "summarizer": {"max_concurrent": 10, "token_multiplier": 1.0},
    "researcher": {"max_concurrent": 5, "token_multiplier": 1.5},
}

The token_multiplier adjusts token budget estimates for scheduling purposes. A code generation agent with a 2.0 multiplier is treated as consuming twice the planned token budget for capacity calculations, providing a safety margin against underestimation for agents with variable output length.

Coordinating Related Tasks

When multiple agents work on related tasks, their execution order can affect result quality. For example, if Agent A summarizes a document and Agent B answers questions about the same document, running B before A wastes tokens (B cannot reference A's summary). If A and B are independent, they can run in parallel. If B depends on A's output, they must run sequentially.

Task Dependencies

OCCP supports explicit task dependencies through a directed acyclic graph (DAG) model. When submitting related tasks, the caller specifies which tasks depend on which. The scheduler uses the DAG to determine execution order, running independent tasks in parallel and dependent tasks sequentially.

# Task dependency specification
tasks = [
    Task(id="summarize", agent="summarizer", depends_on=[]),
    Task(id="qa_1", agent="researcher", depends_on=["summarize"]),
    Task(id="qa_2", agent="researcher", depends_on=["summarize"]),
    Task(id="compile", agent="writer", depends_on=["qa_1", "qa_2"]),
]

# Execution order:
# 1. "summarize" executes first (no dependencies)
# 2. "qa_1" and "qa_2" execute in parallel (both depend only on "summarize")
# 3. "compile" executes last (depends on qa_1 and qa_2)

Conflict Detection

For tasks that access shared resources (the same database table, the same file, the same external API), the scheduler can detect potential conflicts and serialize access. Conflict detection uses a resource locking model: each task declares the resources it will access, and the scheduler ensures that conflicting tasks do not execute concurrently. This prevents race conditions where two agents modify the same resource simultaneously, producing inconsistent results.

Scheduler Metrics

The scheduler exposes metrics for monitoring and capacity planning:

  • Queue depth — Number of tasks waiting for execution, broken down by priority level
  • Queue wait time — P50, P95, and P99 latency from task submission to execution start
  • Provider utilization — Current capacity usage as a percentage of maximum, per provider and per dimension
  • Agent concurrency — Current and maximum concurrent executions per agent type
  • Rejection rate — Percentage of task submissions rejected due to backpressure
  • Throughput — Tasks completed per minute, broken down by agent type and priority

These metrics are essential for right-sizing deployments: if queue depth consistently grows during peak hours, more LLM API capacity is needed. If a specific agent type has high rejection rates, its concurrency limit may need adjustment. The scheduler's resource model provides the data needed for informed capacity decisions, preventing both over-provisioning (wasted cost) and under-provisioning (degraded performance).

Design for Evolution

The v0.1.0 scheduler is single-node: all scheduling decisions are made by one process. This is sufficient for development and small-scale deployments, but will not scale to multi-node production environments where tasks can be submitted to any node. The scheduler interface is designed for future evolution toward distributed scheduling, with the current single-node implementation serving as the reference behavior that distributed implementations must match.

// Key Takeaways

  • Resource-aware scheduling tracks concurrent requests, RPM, and TPM per LLM provider simultaneously
  • Sliding window rate tracking prevents burst behavior at minute boundaries
  • Four-level priority queue (critical, high, normal, low) with FIFO ordering within each level
  • Per-agent-type concurrency limits with configurable token multipliers for conservative estimation
  • DAG-based task dependencies enable parallel execution of independent tasks and sequential ordering of dependent ones
  • Scheduler metrics provide data for capacity planning and deployment right-sizing
