OCCP v0.1.0: The First Verified Orchestration Cycle

The initial release of OCCP delivers the core platform: an orchestrator engine implementing the Verified Autonomy Pipeline, a policy engine with five initial guards, a command-line interface, and a Python SDK. For the first time, an AI task can be planned, policy-checked, executed, validated, and audited in a single end-to-end cycle.

What Shipped in v0.1.0

The v0.1.0 release represents the minimum viable control plane: the smallest set of components that implement the complete Verified Autonomy Pipeline from task submission to audited result delivery. The release contains four major components, each solving a distinct problem in the governance stack.

Orchestrator Engine

The orchestrator is the central coordination component. It receives a task, routes it through the five VAP stages, and returns the audited result. The v0.1.0 orchestrator supports sequential execution of single-agent tasks — one task at a time, one agent at a time. Multi-agent and parallel execution are deferred to future releases.

# Orchestrator core loop
class Orchestrator:
    def __init__(self, policy_engine, executor, validator, auditor):
        self.policy_engine = policy_engine
        self.executor = executor
        self.validator = validator
        self.auditor = auditor

    async def run(self, task: Task) -> TaskResult:
        # Stage 1: Plan
        plan = self.plan(task)

        # Stage 2: Gate
        gate_result = self.policy_engine.evaluate(plan)
        if not gate_result.allowed:
            return TaskResult(status="denied", reason=gate_result.reason)

        # Stage 3: Execute
        execution = await self.executor.execute(plan)

        # Stage 4: Validate
        validation = self.validator.validate(execution)
        if not validation.passed:
            return TaskResult(status="validation_failed", issues=validation.issues)

        # Stage 5: Ship
        audit_entry = self.auditor.log(task, plan, gate_result, execution, validation)
        return TaskResult(status="completed", output=execution.output, audit_id=audit_entry.id)

The orchestrator's architecture enforces the pipeline invariant: there is no code path that reaches executor.execute() without first passing through policy_engine.evaluate(). The pipeline ordering is a structural property of the code, not a runtime configuration.

Policy Engine

The policy engine evaluates task plans against a set of active policies. The v0.1.0 release ships with five initial policy guards:

  1. PII Detection — Regex-based scanning for email addresses, phone numbers, social security numbers, and credit card patterns in input text
  2. Prompt Injection Detection — Pattern matching for five initial injection vectors: role override, delimiter injection, instruction leak requests, encoding-based evasion, and context boundary manipulation
  3. Token Budget Enforcement — Verifies that the planned token usage does not exceed the configured maximum per-task limit
  4. Timeout Enforcement — Ensures that planned execution time falls within the configured timeout boundary
  5. Content Category Filter — Checks task descriptions against a configurable deny-list of content categories

Policies are evaluated in order, and the first denial stops evaluation. All results — both passes and denials — are recorded in the audit trail for compliance purposes.

CLI Tool

The occp command-line interface provides three primary commands for interacting with the platform:

# Submit a task and run through the full VAP pipeline
$ occp run --task "Summarize the quarterly report" --agent default

# Check the status of a running or completed task
$ occp status --task-id tsk_a1b2c3

# View the audit trail for a specific task or time range
$ occp audit --task-id tsk_a1b2c3
$ occp audit --since 2025-11-01 --until 2025-11-05

The CLI was deliberately built as the first interface, before any REST API or dashboard. This CLI-first approach forced clean separation between the interface layer and the core engine. The CLI communicates with the orchestrator through the same internal API that future REST endpoints and SDK clients will use, ensuring consistency across all access paths.

Python SDK

The Python SDK provides programmatic access to OCCP from Python applications. The v0.1.0 SDK uses only Python standard library modules — no external dependencies — to minimize installation friction and dependency conflicts.

from occp import OCCPClient

client = OCCPClient(base_url="http://localhost:8000")

# Submit a task
result = client.run_task(
    description="Analyze customer feedback for sentiment",
    agent="default",
    token_budget=2048,
    timeout=30
)

# Check task result
print(f"Status: {result.status}")
print(f"Output: {result.output}")
print(f"Audit ID: {result.audit_id}")

# Retrieve audit trail
audit = client.get_audit(task_id=result.task_id)
for entry in audit.entries:
    print(f"  [{entry.stage}] {entry.action}: {entry.result}")

The SDK mirrors the CLI's capabilities: task submission, status checking, and audit trail retrieval. The client handles connection management, error handling, and response parsing. All HTTP communication uses the standard library urllib module, avoiding any dependency on requests or httpx.

The First End-to-End Cycle

The most significant milestone in v0.1.0 is not any individual component but the fact that they work together as a complete pipeline. A task enters through the CLI or SDK, passes through all five VAP stages, and produces an audited result with a SHA-256 hash chain entry. This end-to-end cycle is the proof of concept for the entire OCCP architecture.

Architecture Decisions and Trade-offs

Synchronous Pipeline

The v0.1.0 pipeline is synchronous within the async Python runtime: each stage completes before the next begins. This simplifies reasoning about the pipeline's behavior but limits throughput to one stage at a time. Future versions will introduce stage-level parallelism where appropriate (for example, running multiple independent policy checks concurrently within the Gate stage).

File-Based Audit Storage

The v0.1.0 audit trail is stored in append-only JSON files on the local filesystem. This is sufficient for single-node development but will not scale to multi-node production deployments. The storage backend is abstracted behind an interface, allowing future migration to SQLite, PostgreSQL, or a dedicated audit database without changing the pipeline code.

Single-Agent Focus

The v0.1.0 release supports only single-agent execution. There is no agent registry, no multi-agent coordination, and no concurrent task execution. This constraint allowed us to focus the initial release on getting the pipeline stages correct before adding the complexity of multi-agent orchestration. The orchestrator's internal design, however, was built with multi-agent support in mind: the agent interface is abstract, and the executor accepts agent identifiers that will map to registered agents in future versions.

What v0.1.0 Does Not Include

To set expectations clearly: v0.1.0 does not include a REST API (tasks are submitted via CLI or SDK only), does not include authentication or authorization (single-user, single-machine), does not include a web dashboard, does not include Docker deployment support, and does not include real LLM integration (the executor uses a mock provider for initial testing). Each of these capabilities is planned for subsequent releases.

// Key Takeaways

  • v0.1.0 delivers the minimum viable control plane: orchestrator, policy engine, CLI, and Python SDK
  • The first complete Plan-Gate-Execute-Validate-Ship cycle works end-to-end with SHA-256 audit trail
  • Five initial policy guards: PII detection, injection detection, token budget, timeout, and content filtering
  • Python SDK uses zero external dependencies — stdlib-only HTTP client
  • CLI-first design ensures clean API separation before REST or dashboard layers are added
  • Trade-offs: synchronous pipeline, file-based audit, single-agent only — each with a clear upgrade path
