OCCP v0.2.0: From CLI to Production-Ready Platform

The v0.2.0 release transforms OCCP from a single-user CLI tool into a multi-user production platform. Major additions include a FastAPI REST API with full OpenAPI documentation, JWT authentication with bcrypt password hashing, Docker Compose deployment with healthchecks, and a TypeScript SDK for frontend integration.

The Production Gap

OCCP v0.1.0 proved that the Verified Autonomy Pipeline worked end-to-end. But it was a single-user system: one developer, one machine, one CLI. Production environments require multi-user access, network-based APIs, authentication, and reproducible deployment. The v0.2.0 release closes this gap with four major additions that collectively move OCCP from prototype to production-ready.

FastAPI REST API

The REST API layer was built with FastAPI, chosen for three reasons: native async support (matching the orchestrator's async architecture), automatic OpenAPI schema generation (providing self-documenting endpoints), and Pydantic integration (enabling request/response validation with Python type hints).

Endpoint Design

The API endpoints map directly from the CLI commands established in v0.1.0. This is the payoff of CLI-first design: the API surface was already validated through months of CLI usage.

# Task lifecycle endpoints
POST   /api/v1/tasks              # Submit a new task (replaces: occp run)
GET    /api/v1/tasks/{task_id}     # Get task status (replaces: occp status)
GET    /api/v1/tasks               # List tasks with filtering

# Audit endpoints
GET    /api/v1/audit/{task_id}     # Audit trail for a task (replaces: occp audit)
GET    /api/v1/audit               # Query audit entries with filters

# Health and metadata
GET    /api/v1/health              # Healthcheck endpoint
GET    /api/v1/info                # Platform version and configuration

Request Validation

Every request is validated through Pydantic models before reaching the orchestrator. This is a separate validation layer from the policy engine — Pydantic validates the request structure (are required fields present, are types correct), while the policy engine validates the request content (does it contain PII, does it request excessive resources).

from pydantic import BaseModel, Field

class TaskCreate(BaseModel):
    description: str = Field(
        ...,
        min_length=1,
        max_length=10000,
        description="Task description to be processed by the pipeline"
    )
    agent: str = Field(
        default="default",
        description="Agent identifier for task execution"
    )
    token_budget: int = Field(
        default=4096,
        ge=1,
        le=32768,
        description="Maximum token budget for LLM calls"
    )
    timeout: int = Field(
        default=60,
        ge=1,
        le=300,
        description="Execution timeout in seconds"
    )
    priority: str = Field(
        default="normal",
        pattern="^(critical|high|normal|low)$",
        description="Task priority level"
    )

OpenAPI Documentation

FastAPI generates OpenAPI 3.0 specifications automatically from the endpoint definitions and Pydantic models. The generated documentation is available at /docs (Swagger UI) and /redoc (ReDoc). This means the API documentation is always synchronized with the implementation — a discrepancy between documentation and behavior is structurally impossible because both are generated from the same source code.

JWT Authentication

Production deployment requires authentication: knowing who is making each API call. OCCP v0.2.0 implements JSON Web Token (JWT) authentication, chosen over session-based authentication for three reasons specific to AI platforms.

Why JWT for AI Platforms

  • Stateless verification — The API server can verify a JWT without querying a session store. This is important for AI workloads where the server may be handling long-running pipeline executions and should not block on database queries for every request.
  • SDK compatibility — Python and TypeScript SDKs can include the JWT as a bearer token in HTTP headers. Session cookies are browser-specific and do not work cleanly with programmatic SDK clients.
  • Role embedding — JWT payloads can include user role claims, enabling the API to make authorization decisions from the token alone without additional database lookups. This is the foundation for the RBAC system added in later versions.

Password Hashing

User passwords are hashed using bcrypt with 12 rounds. bcrypt was chosen over faster alternatives (SHA-256, PBKDF2) because its deliberate slowness makes brute-force attacks computationally expensive. The 12-round configuration provides a balance between security (approximately 250ms per hash on modern hardware) and usability (login is not perceptibly slow).

# Authentication flow
from datetime import datetime, timedelta, timezone
import bcrypt
import jwt

class AuthService:
    def __init__(self, secret_key: str, algorithm: str = "HS256"):
        self.secret_key = secret_key
        self.algorithm = algorithm
        self.token_expiry = timedelta(hours=24)

    def hash_password(self, password: str) -> str:
        return bcrypt.hashpw(
            password.encode("utf-8"),
            bcrypt.gensalt(rounds=12)
        ).decode("utf-8")

    def verify_password(self, password: str, hashed: str) -> bool:
        return bcrypt.checkpw(
            password.encode("utf-8"),
            hashed.encode("utf-8")
        )

    def create_token(self, user_id: str, role: str) -> str:
        now = datetime.now(timezone.utc)
        payload = {
            "sub": user_id,
            "role": role,
            "iat": now,
            "exp": now + self.token_expiry,
        }
        return jwt.encode(payload, self.secret_key, algorithm=self.algorithm)

Auth Middleware

Every API endpoint (except /api/v1/health and the authentication endpoint itself) requires a valid JWT in the Authorization: Bearer header. The middleware validates the token signature, checks expiration, and extracts the user identity and role before the request reaches the endpoint handler. Invalid or expired tokens produce a 401 response with a clear error message.

Docker Compose Deployment

The v0.2.0 release includes a Docker Compose configuration for reproducible deployment. The composition includes the OCCP API server, a reverse proxy for TLS termination, and volume mounts for persistent data (audit trail, configuration).

# docker-compose.yml (simplified)
services:
  occp-api:
    build: .
    ports:
      - "8000:8000"
    environment:
      - OCCP_ENV=production
      - OCCP_LOG_LEVEL=info
    volumes:
      - occp-data:/app/data
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/api/v1/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 10s
    restart: unless-stopped

volumes:
  occp-data:

Healthcheck Design

The healthcheck endpoint verifies three things: the API server is responding to HTTP requests, the policy engine is loaded and functional, and the audit trail storage is writable. If any check fails, the healthcheck returns a 503 status, triggering Docker's restart policy. This three-point check ensures that a "healthy" container is genuinely operational, not just running a process that has lost connectivity to its dependencies.

TypeScript SDK

The TypeScript SDK enables frontend applications to interact with OCCP's API. Like the Python SDK, it uses zero external dependencies — the native fetch API handles all HTTP communication.

// TypeScript SDK usage
import { OCCPClient } from "@occp/sdk";

const client = new OCCPClient({
  baseUrl: "https://api.occp.ai",
  token: "eyJhbGciOiJIUzI1NiIs...",
});

// Submit a task
const result = await client.createTask({
  description: "Analyze customer feedback for sentiment",
  agent: "classifier",
  tokenBudget: 2048,
  timeout: 30,
});

console.log(`Status: ${result.status}`);
console.log(`Output: ${result.output}`);
console.log(`Audit ID: ${result.auditId}`);

The TypeScript SDK mirrors the Python SDK's surface area: task creation, status queries, and audit trail retrieval. Both SDKs target the same API endpoints with the same parameters, ensuring behavioral consistency across languages. The TypeScript SDK is essential for the dashboard integration planned for v0.3.0 — the dashboard will be a Next.js application that uses the TypeScript SDK to communicate with the API.

Testing and CI/CD

The v0.2.0 release includes 37 tests covering the core pipeline, policy engine, API endpoints, and authentication flow. Tests are organized by component: test_orchestrator.py, test_policy_engine.py, test_api.py, test_auth.py. The CI pipeline runs tests against Python 3.11, 3.12, and 3.13 to ensure compatibility across supported versions.

GitHub Actions runs the full test suite on every push and pull request. The CI configuration also includes a secret scanning step using GitGuardian to prevent accidental credential commits. Protected branch rules require all CI checks to pass before merging to main.

What Changed: v0.1.0 vs. v0.2.0

The transition from v0.1.0 to v0.2.0 is a shift in deployment model, not in core architecture. The Verified Autonomy Pipeline is unchanged. The policy engine is unchanged. The audit chain is unchanged. What changed is the access layer: CLI-only became CLI + REST API + SDK. What was added is operational infrastructure: authentication, containerized deployment, and automated testing. The core governance pipeline that was validated in v0.1.0 is now accessible to multiple users, deployable to production servers, and secured with authentication. This is the transition from "it works on my machine" to "it works in production."
