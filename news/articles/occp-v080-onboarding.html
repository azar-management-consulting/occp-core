OCCP v0.8.0: The Onboarding Revolution

For seven versions, we built capabilities: pipeline stages, policy rules, authentication, RBAC, sandboxing, ORM. v0.8.0 changes direction. Instead of adding new capabilities, it makes existing capabilities accessible. The MCP connector framework, skills system, settings pages, and 10-step onboarding wizard transform OCCP from a platform that developers build on to a platform that developers can start using in minutes.

The Accessibility Problem

By v0.7.0, OCCP had a comprehensive feature set: JWT authentication, a four-role RBAC hierarchy, a policy engine with 24 detection patterns, sandboxed code execution, an async ORM with migration infrastructure, a React 19 dashboard, and dual SDKs. The problem was not missing features — it was that using those features required reading source code, understanding internal APIs, and manually configuring each component.

A developer evaluating OCCP faced a steep onboarding cliff. They could clone the repository and run the tests, but going from "tests pass" to "my first agent is running a governed task" required intimate knowledge of the system's internals. This is the classic infrastructure software trap: the platform is powerful but inaccessible, and inaccessible power is indistinguishable from missing features.

The gap between "this platform can do X" and "I can make this platform do X" is the only gap that matters to the person evaluating your software.

MCP Connector Framework

The Model Context Protocol (MCP) connector framework is the first major addition in v0.8.0. MCP is an open standard for connecting AI models to external tools — filesystems, databases, APIs, cloud services. OCCP's MCP integration enables agents to discover and invoke tools from any MCP-compatible server, with every tool call routed through the policy gate for security evaluation.

The connector framework supports 22+ MCP server integrations out of the box:

  • Infrastructure — filesystem, GitHub, Docker, Kubernetes
  • Databases — SQLite, PostgreSQL, MySQL
  • Cloud providers — Hetzner, Cloudflare, Hostinger
  • Monitoring — Grafana, n8n workflow automation
  • Security — Semgrep SAST, Snyk SCA, GitGuardian secret detection
  • Research — web search, web scraping, documentation lookup

Each connector is configured through the new settings pages in the dashboard, not through environment variables or configuration files. A developer can enable the GitHub MCP server, provide an API token, and immediately give their agents the ability to create pull requests, read issues, and manage repositories — all through the governed pipeline.

Skills System

Skills are the second major addition. A skill is a named, reusable capability that bundles together the tools, permissions, prompts, and validation rules needed for a specific domain task. Instead of configuring an agent with raw tool access and hoping it figures out the right workflow, a skill provides a structured template that guides the agent through a well-defined process.

# Example skill definition
{
  "name": "code-review",
  "description": "Review a pull request for code quality and security issues",
  "required_tools": ["github.get_pr", "github.get_diff", "github.create_review"],
  "permissions": ["read:repository", "write:review"],
  "validation_rules": [
    "output_must_contain_severity_rating",
    "output_must_not_contain_credentials"
  ],
  "prompt_template": "Review the following pull request diff for: ..."
}

Skills are acquired by agents through a request-and-validation flow. An agent requests a skill, the platform validates that the agent has the required tool access and permissions, and if validation passes, the skill is attached to the agent's capability profile. This creates a composable model: agents are assembled from skills, skills are assembled from tools, and the platform enforces boundaries at every level.

Skills vs. Raw Tool Access

The distinction between skills and raw tool access is governance. Raw tool access means the agent can call any function on any connected MCP server, subject only to broad policy rules. Skills narrow the scope: a code-review skill can only call GitHub read and review endpoints, not repository deletion or settings modification. The skill definition becomes a contract between the platform operator and the agent, making it clear what the agent can and cannot do for a given task type.

Settings Pages

The settings system replaces the previous configuration approach (environment variables and YAML files) with a dashboard-based interface. The settings pages are organized into four tabs:

  1. LLM Configuration — Select provider (Claude, GPT-4, Gemini, local models), enter API key (encrypted at rest), configure model parameters (temperature, max tokens, timeout).
  2. Tool Configuration — Enable/disable MCP servers, configure connection parameters, test connectivity.
  3. Security Settings — Configure policy engine rules, set rate limits, manage injection detection sensitivity.
  4. General Settings — Organization name, default agent settings, audit retention period.

Settings follow a hierarchical override model: system defaults serve as the base, organization-level overrides customize for the deployment, and user-level preferences personalize the experience. This three-tier hierarchy allows a platform administrator to set security baselines that individual users cannot weaken while still permitting customization of non-security settings.

10-Step Onboarding Wizard

The onboarding wizard is the centerpiece of v0.8.0. It guides a new user from initial landing to their first governed autonomous task in exactly ten steps. Each step has prerequisite validation — you cannot proceed to step 5 until steps 1 through 4 are complete. But the wizard does not hand-hold: each step presents the configuration interface directly, with contextual documentation for users who need it and a clean form for users who do not.

The ten steps are:

  1. Landing — Platform overview, capability summary, "Get Started" action.
  2. Account Creation — Username, email, password with strength validation.
  3. LLM Provider — Select from supported providers (Anthropic, OpenAI, Google, local).
  4. API Key — Enter provider API key, validate connectivity, encrypt and store.
  5. First Agent — Register an agent with a name and description.
  6. Agent Capabilities — Assign skills and tool access to the agent.
  7. First Task — Create a task with title, description, and assigned agent.
  8. Pipeline Execution — Execute the task through Plan, Gate, Execute, Validate, Ship.
  9. Audit Review — View the complete audit trail for the executed task.
  10. Production Path — Configuration for production deployment: Docker Compose, environment variables, SSL.

The wizard stores progress in local storage and the user's server-side profile, so a user who leaves mid-wizard can resume from where they stopped. Each step renders in the same dashboard container, maintaining visual consistency with the rest of the application.

The wizard is not a tutorial. Tutorials teach concepts. The wizard produces a working system. When a user completes step 10, they have a registered agent, a configured LLM provider, and a verified audit trail. They have not learned about OCCP — they have used OCCP.

The Shift from Building to Enabling

v0.8.0 represents a philosophical shift in the project. For seven versions, the question was "what should the platform do?" Starting with v0.8.0, the question becomes "how do people use what the platform does?" This is a harder problem. Building a policy engine is an engineering challenge with clear success criteria: does it detect injections? Does it scan for PII? Does it enforce rate limits? Making a policy engine usable is a design challenge with subjective success criteria: can a new user configure it? Can they understand why a request was blocked? Can they adjust the sensitivity without reading the source code?

The MCP connector framework, skills system, settings pages, and onboarding wizard are not new capabilities. They are access layers for existing capabilities. The policy engine is the same engine from v0.6.0. The RBAC system is the same Casbin implementation from v0.7.0. The pipeline is the same five-stage loop from v0.1.0. What changed is that a developer can now configure and use all of these through a guided interface instead of through source code archaeology.

Upgrade Path from v0.7.0

Upgrading from v0.7.0 to v0.8.0 requires running alembic upgrade head to apply the new migrations for skills, MCP connections, and settings tables. Existing agents, users, and audit data are preserved. The new settings pages will initialize with system defaults, and existing environment variable configurations continue to work as the base layer that dashboard settings override. No breaking changes to the REST API — v0.8.0 adds new endpoints but does not modify existing ones.
