Settings Architecture: LLM Provider Configuration at Scale

Before v0.8.0, configuring OCCP meant editing environment variables and restarting the server. The settings architecture replaces that with a hierarchical configuration system: system defaults, organization overrides, and user preferences — all manageable through the dashboard, all enforcing security boundaries that individual users cannot weaken.

The Configuration Problem

OCCP needs to configure multiple concerns across multiple scopes. LLM provider selection, API key storage, model parameters, MCP server connections, policy engine sensitivity, rate limiting thresholds, audit retention periods — each of these has a default value, may need organization-level customization, and might have user-level preferences. The pre-v0.8.0 approach of environment variables handled the system default level but offered no mechanism for per-organization or per-user overrides.

The additional challenge is security boundaries. An organization administrator should be able to set a minimum password length that individual users cannot lower. A platform operator should be able to restrict which LLM providers are available, preventing users from routing data through unapproved providers. The settings system must support both customization (users can change things) and enforcement (administrators can prevent users from changing things).

Hierarchical Settings Model

The settings system uses a three-tier hierarchy with cascading overrides:

# Settings resolution order (highest priority last)
SYSTEM_DEFAULTS      # Compiled into the application
    |
    v
ORGANIZATION_SETTINGS  # Set by org admin, stored in DB
    |
    v
USER_PREFERENCES       # Set by individual user, stored in DB

Each setting has a resolution strategy. Most settings use "last wins" — the most specific scope (user) overrides the less specific scope (organization), which overrides the default. Security-critical settings use "most restrictive wins" — the user cannot set a value less restrictive than the organization's setting, and the organization cannot set a value less restrictive than the system default.

# occp/settings/resolver.py — hierarchical setting resolution
class SettingsResolver:
    def resolve(self, key: str, user_id: int, org_id: int) -> SettingValue:
        definition = self.definitions[key]
        system_val = definition.default
        org_val = self.org_store.get(org_id, key)
        user_val = self.user_store.get(user_id, key)

        if definition.enforcement == "most_restrictive":
            return self._most_restrictive(system_val, org_val, user_val, definition)
        else:
            # Last-wins cascade
            return user_val or org_val or system_val

    def _most_restrictive(self, system, org, user, definition):
        """For security settings, always use the most restrictive value."""
        values = [v for v in [system, org, user] if v is not None]
        if definition.type == "integer":
            return max(values) if definition.restrict_direction == "higher" else min(values)
        if definition.type == "boolean":
            # If any level enforces True, result is True
            return any(values) if definition.restrict_direction == "enabled" else all(values)

Setting Definitions

Every setting is defined in a central registry with its key, type, default value, validation rules, enforcement strategy, and which scopes can modify it. This registry serves as both the source of truth for the API and the schema for the settings UI.

LLM Provider Configuration

The LLM provider settings tab handles provider selection, model configuration, and API key management. OCCP supports four provider categories, each with its own configuration requirements:

  • Anthropic — Models: Claude 3 Opus, Claude 3 Sonnet, Claude 3 Haiku. API key format: sk-ant-api03-*. Endpoint: https://api.anthropic.com. Custom headers: anthropic-version.
  • OpenAI — Models: GPT-4 Turbo, GPT-4o, GPT-4o-mini. API key format: sk-*. Endpoint: https://api.openai.com/v1. Organization ID optional.
  • Google — Models: Gemini 1.5 Pro, Gemini 1.5 Flash. API key format: Google API key. Endpoint: https://generativelanguage.googleapis.com.
  • Local Models — Ollama, vLLM, or custom OpenAI-compatible endpoints. Base URL configurable. No API key required for local deployments.

Provider-specific validation runs when an API key is entered. The validation makes a lightweight, zero-cost API call to the provider — listing available models for Anthropic and OpenAI, a simple ping for Google — to confirm the key is valid and has the necessary permissions. The validation result is displayed immediately, and the key is encrypted with AES-256-GCM before being stored in the database.

API keys are write-once secrets. After storage, the settings UI displays only the last four characters. The full key is never transmitted back to the client, never logged, and never included in audit entries. The only way to verify a stored key is to use it for a provider API call.

Tool Configuration Tab

The tool configuration tab presents connected MCP servers as a list of toggleable cards. Each card shows the server name, transport type (stdio or HTTP+SSE), connection status (connected, disconnected, error), the number of available tools, and a health indicator. Enabling a server connects it and discovers its tools. Disabling a server disconnects it and suspends all skills that depend on its tools.

Configuration parameters vary by server type. Stdio servers require a command path and optional arguments. HTTP+SSE servers require a URL, optional authentication token, and TLS settings. All connection parameters are validated before saving — the system attempts a test connection and reports success or failure immediately.

Security Settings Tab

The security tab provides controls for policy engine configuration without exposing implementation details. Instead of asking users to write regex patterns, the tab presents security concerns as toggleable options with sensitivity sliders:

  • PII Detection — Enable/disable, sensitivity level (low/medium/high), which PII types to detect (email, phone, SSN, credit card).
  • Injection Detection — Enable/disable, sensitivity level, false positive reporting mechanism.
  • Output Sanitization — Enable/disable, which patterns to strip (API keys, passwords, tokens, file paths).
  • Rate Limiting — Requests per minute, burst allowance, per-user versus per-IP enforcement.
  • Audit Retention — Retention period in days, automatic cleanup schedule.

Security settings follow the "most restrictive wins" enforcement model. If the organization sets injection detection to high sensitivity, individual users cannot lower it. If the system default enforces output sanitization, no scope can disable it. This ensures that security baselines are maintained regardless of user preferences.

General Settings Tab

The general tab handles non-security, non-LLM configuration: organization name, default agent settings (max concurrent tasks, default capability set), notification preferences, and UI customization (theme, language, timezone). These settings use the "last wins" enforcement model — user preferences override organization defaults for all general settings.

API Design

The settings API exposes three endpoints that mirror the hierarchical model:

GET  /api/v1/settings                    # Resolved settings for current user
GET  /api/v1/settings/organization       # Organization-level settings (admin only)
PUT  /api/v1/settings                    # Update user preferences
PUT  /api/v1/settings/organization       # Update org settings (admin only)

# Response includes resolution metadata
{
  "llm.provider": {
    "value": "anthropic",
    "source": "organization",   // where this value came from
    "locked": false,            // can the current user change it?
    "default": "anthropic"      // system default for reference
  },
  "security.rate_limit_per_minute": {
    "value": 60,
    "source": "organization",
    "locked": true,             // most_restrictive enforcement
    "default": 100
  }
}

The source field tells the UI where the current value originated. The locked field indicates whether the user can modify it. The UI disables input controls for locked settings and displays a tooltip explaining which scope enforced the restriction. This transparency prevents user frustration — a user who cannot change the rate limit understands that it is an organizational policy, not a bug.

Migration from Environment Variables

Existing deployments that use environment variables for configuration are not broken by the settings system. Environment variables serve as the system default tier. Dashboard settings override environment variables at the organization and user tiers. If an administrator has not configured any dashboard settings, the platform behaves identically to pre-v0.8.0: environment variables drive all configuration. This backward compatibility ensures that upgrades from v0.7.0 require no configuration changes.

// Key Takeaways

  • Three-tier hierarchy: system defaults, organization overrides, user preferences
  • Security settings use "most restrictive wins" enforcement — users cannot weaken organizational baselines
  • API keys are encrypted with AES-256-GCM and never returned to the client after storage
  • Settings API includes resolution metadata (source, locked) for transparent UI behavior
  • Environment variables serve as the system default tier — backward compatible with pre-v0.8.0 deployments
