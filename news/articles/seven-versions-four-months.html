2026-02-16 Vision v0.7.0

What We Learned Building 7 Versions in 4 Months

Between October 2025 and February 2026, OCCP went from an idea to a production-deployed AI orchestration platform with JWT authentication, a policy engine with 24 detection patterns, Casbin RBAC, sandboxed execution, a React 19 dashboard, dual SDKs, and 313 tests. Here is what that journey taught us about building infrastructure software at speed.

The Timeline

Looking back at the git history, each month had a clear architectural focus that built on the previous month's foundation.

October 2025: The Idea and First Commit

OCCP began with a straightforward observation: the AI ecosystem had powerful models and capable orchestration frameworks, but no governance layer between them and production. The first commit was a Python module with three files: an orchestrator implementing the five-stage Verified Autonomy Pipeline, a policy engine with basic PII detection, and a CLI that could submit tasks and view results. The entire system ran in a single process, used an in-memory dictionary for storage, and called no external APIs. But the pipeline worked: Plan, Gate, Execute, Validate, Ship. Every subsequent version is an elaboration on that initial five-stage loop.

November 2025: v0.1.0 — The Working Pipeline

The first release formalized what the prototype demonstrated. The orchestrator gained configurable stages with retry logic. The policy engine added prompt injection detection alongside PII scanning. The CLI expanded to support task creation, status checking, and audit trail export. The SHA-256 hash-chained audit trail was implemented, creating a tamper-evident record of every pipeline execution. And the Python SDK shipped — a stdlib-only HTTP client with zero external dependencies. The test suite had 37 tests.

December 2025: v0.2.0 + v0.3.0 — API and Intelligence

December was the month of external interfaces. v0.2.0 introduced the FastAPI REST API with JWT authentication, replacing the CLI-only interaction model. Docker Compose deployment landed, with health checks and security contexts. The TypeScript SDK was added for Node.js consumers. v0.3.0 followed two weeks later with real Claude integration through the Anthropic API, prompt injection detection upgraded to regex-based pattern matching, and the first dashboard screens implemented in Next.js. The test count jumped to 127.

January 2026: v0.4.0 + v0.5.0 — Dashboard and Agents

January focused on making the platform usable by people who are not the developers who built it. v0.4.0 delivered the full Next.js 15 dashboard with React 19, JWT-based AuthGuard route protection, and the retro-terminal visual identity that became OCCP's signature. v0.5.0 introduced the agent registry — the subsystem that manages agent lifecycle, capabilities, and runtime boundaries. Agents could now be registered with declared capabilities, and the platform would enforce those boundaries at execution time. Tests reached 203.

February 2026: v0.6.0 + v0.7.0 — Security and Enterprise

February was security month. v0.6.0 hardened the policy engine from 5 to 24 injection detection patterns, added output sanitization that strips credentials from LLM responses, and implemented GDPR-compliant legal pages. v0.7.0 was the maturity release: SQLAlchemy 2.0 async ORM replaced raw SQL queries, Casbin RBAC introduced a four-role hierarchy (viewer, operator, admin, super_admin), nsjail/bubblewrap sandboxing isolated code execution, and Alembic migrations provided schema evolution infrastructure. By the end of February, the test suite contained 313 tests across Python 3.11, 3.12, and 3.13.

Lesson 1: Start With the Pipeline

The single best architectural decision was implementing the five-stage pipeline first, before the API, before the dashboard, before authentication. The pipeline is the core abstraction: every feature is either a stage in the pipeline, a modifier of a stage, or a consumer of pipeline output. Authentication gates who can submit tasks to the pipeline. RBAC determines which stages a role can configure. The dashboard visualizes pipeline execution. The audit trail records pipeline results.

If we had started with the API or the dashboard, we would have built a web application and then tried to attach a pipeline to it. By starting with the pipeline, we built an orchestration engine and then attached interfaces to it. The difference is architectural gravity: the pipeline pulls everything toward it.

Lesson 2: Security Is Architecture, Not a Feature

The v0.6.0 security hardening sprint took three weeks. It would have taken three days if security had been integrated from v0.1.0. When you add injection detection as an afterthought, you have to retrofit it into every input path. When you add output sanitization late, you discover output paths you did not know existed. When you add RBAC to an existing API, you have to audit every endpoint to determine which roles should access it.

The correct approach — which we adopted starting in v0.6.0 — is to treat security as an architectural layer, not a feature toggle. The policy engine is not optional. The audit trail is not configurable. Output validation is not bypassable. These are properties of the infrastructure, enforced by the middleware stack, and no amount of misconfigured application code can circumvent them.

Lesson 3: Testing Enables Speed

The counterintuitive lesson is that writing tests did not slow us down. It accelerated us. When we refactored from raw SQL to SQLAlchemy 2.0 ORM in v0.7.0, the refactoring touched 40+ files and changed every database interaction in the system. Without the 256 tests from v0.6.0, this refactoring would have been terrifying and slow. With the tests, it was mechanical: change the code, run the tests, fix the failures, repeat. The ORM migration took four days. Without tests, it would have taken four weeks of careful manual verification.

The key insight is that tests are not about preventing bugs in the current version. They are about enabling changes in the next version. Every test is a safety net for a refactoring that has not happened yet. The 313 tests in v0.7.0 exist so that v0.8.0 can change anything without fear.

Lesson 4: Open Source Builds Trust

OCCP is open source not because of ideology but because of pragmatism. An AI governance platform asks its users to trust it with their LLM API keys, their policy configurations, and their audit data. That trust must be verifiable. When the source code is public, any organization can audit the policy engine to verify it does what it claims. They can inspect the encryption implementation to confirm their API keys are protected. They can review the audit chain to confirm it is genuinely tamper-evident.

Closed-source governance is a contradiction. You cannot govern AI systems with a black box. The governance layer itself must be transparent, auditable, and verifiable. Open source is not a business model decision — it is an architectural requirement for the category we are building.

Lesson 5: Ship Versions, Not Features

We did not maintain a feature backlog that we gradually depleted. We shipped versions with coherent themes. v0.2.0 was the API version. v0.4.0 was the dashboard version. v0.6.0 was the security version. v0.7.0 was the ORM version. Each version had a clear identity, a clear set of changes, and a clear upgrade path from the previous version.

This version-oriented approach forces prioritization. You cannot ship an incoherent version — it has to tell a story. The story of v0.7.0 is "enterprise readiness through proper data access, access control, and isolation." Every commit in that version supports that narrative. Features that did not fit the narrative were deferred to the next version, not shoehorned in.

What Comes Next

v0.8.0 shifts focus from "build features" to "make features accessible." The MCP connector framework, skills system, settings pages, and 10-step onboarding wizard are not new capabilities — they are new ways to access existing capabilities. The platform has authentication, policy enforcement, agent management, and pipeline execution. What it needs now is a path from "I just installed this" to "my first agent is running in production." That is the challenge of the next month, and it is a harder problem than any we have solved so far.

// Key Takeaways

  • Start with the core abstraction (the pipeline), not the interface layer (the API or dashboard)
  • Security is an architectural layer, not a feature — retrofitting it costs 10x more than building it in
  • Tests enable speed by making large-scale refactoring mechanical instead of terrifying
  • Open source is an architectural requirement for AI governance, not a business model choice
  • Ship versions with coherent themes, not arbitrary feature batches
