Supply Chain Security: Allowlists, Static Analysis, and Fail-Closed Verification

The most dangerous attack vector in modern software is not the code you write — it is the code you import. In AI orchestration, the attack surface expands further: MCP packages, agent extensions, and tool connectors all execute with the privileges of the orchestrator. OCCP v0.8.2 implements a multi-layered verification system that ensures no unverified code enters the execution environment.

The Supply Chain Threat in AI Orchestration

Traditional software supply chain attacks target package registries: a malicious actor publishes a typosquatted package or compromises a maintainer's account to inject backdoors into a popular library. In AI orchestration, the attack surface is broader. MCP (Model Context Protocol) servers expose tools that agents can invoke. Agent extensions add capabilities to the execution environment. Custom connectors bridge the orchestrator to external services. Each of these is a code injection point.

The consequences are also more severe. A compromised MCP package does not just run in a build environment — it runs in the context of an AI agent that may have access to databases, APIs, and production systems. A malicious tool connector could exfiltrate every prompt and response passing through the orchestrator. A backdoored agent extension could override policy decisions, turning the governance layer into a rubber stamp.

In agent orchestration, supply chain compromise does not just affect the build. It affects every decision the agent makes and every system the agent touches.

Layer 1: Allowlist-Based Verification

The Allowlist Model

OCCP uses an explicit allowlist model for all loadable packages. Only packages that appear on the allowlist can be loaded into the execution environment. The allowlist is a configuration file that specifies the package name, the exact version (not a range), and the expected SHA-256 hash of the package contents.

# occp-allowlist.yaml
packages:
  - name: "mcp-filesystem"
    version: "1.2.3"
    sha256: "a1b2c3d4e5f6..."
    publisher: "occp-verified"
    approved_by: "security-team"
    approved_at: "2026-02-20T14:00:00Z"

  - name: "mcp-database"
    version: "2.0.1"
    sha256: "f6e5d4c3b2a1..."
    publisher: "occp-verified"
    approved_by: "security-team"
    approved_at: "2026-02-18T09:30:00Z"

Version Pinning

The allowlist requires exact version pinning. Version ranges (>=1.2.0, ^2.0.0) are not permitted. This prevents a class of attacks where a malicious version is published within an allowed range. If mcp-filesystem version 1.2.3 is approved, version 1.2.4 — even if published by the same author — must go through the approval process independently. Each version is a distinct trust decision.

Hash Verification

The SHA-256 hash in the allowlist is computed over the complete package contents at approval time. At load time, OCCP recomputes the hash of the package being loaded and compares it to the allowlist entry. If the hashes do not match, the package is rejected. This protects against post-approval tampering: even if an attacker compromises the package registry and replaces the approved version with a modified one, the hash check detects the modification.

Layer 2: Static Analysis

Pre-Execution Scanning

Before a package is loaded into the execution environment, OCCP runs static analysis on its contents. The analysis checks for patterns that indicate potentially dangerous behavior: network calls to unexpected destinations, filesystem operations outside the declared scope, dynamic code execution (eval, exec, importlib usage), environment variable access, and subprocess invocation.

# Static analysis rules (simplified)
DANGEROUS_PATTERNS = [
    r"eval\s*\(",              # Dynamic code execution
    r"exec\s*\(",              # Dynamic code execution
    r"__import__\s*\(",        # Dynamic imports
    r"subprocess\.",           # Subprocess invocation
    r"os\.system\s*\(",        # System command execution
    r"socket\.",               # Raw socket access
    r"requests\.(get|post)",   # HTTP calls (must be declared)
    r"open\s*\([^)]*['\"]w",  # File write operations
]

def scan_package(package_path: str) -> ScanResult:
    violations = []
    for file in walk_python_files(package_path):
        content = read_file(file)
        for pattern in DANGEROUS_PATTERNS:
            matches = re.findall(pattern, content)
            if matches:
                violations.append(Violation(
                    file=file, pattern=pattern,
                    count=len(matches)
                ))
    return ScanResult(
        passed=len(violations) == 0,
        violations=violations
    )

Capability Declaration

Packages must declare their required capabilities in a manifest file. An MCP filesystem package must declare that it needs filesystem access. A database connector must declare that it needs network access to specific hosts. The static analysis verifies that the package only uses capabilities it has declared. If the filesystem package contains code that opens network sockets, the analysis flags a capability violation — the package claims to need only filesystem access but actually performs network operations.

Limitations of Static Analysis

Static analysis is not a complete defense. Obfuscated code can evade pattern matching. Dynamically constructed strings can hide dangerous operations. Runtime behavior can differ from static patterns. OCCP treats static analysis as one layer in a multi-layer defense, not as a standalone solution. Its primary value is catching accidental misuse and low-sophistication attacks. Sophisticated supply chain attacks require the other layers to provide defense.

Layer 3: Fail-Closed Verification

The Fail-Closed Principle

OCCP's package verification is fail-closed: if any verification step fails, the package is rejected. If the allowlist lookup fails (package not found, version mismatch, hash mismatch), the package is rejected. If static analysis cannot complete (corrupted package, timeout during analysis), the package is rejected. If signature verification fails (invalid signature, expired certificate), the package is rejected. There is no fallback to an unverified load path.

A verification system that fails open is not a verification system. It is a logging system that sometimes blocks things.

Verification at Load Time, Not Install Time

Verification runs every time a package is loaded, not just when it is first installed. This protects against time-of-check-to-time-of-use (TOCTOU) attacks where a package is modified on disk after passing initial verification. It also ensures that packages loaded from shared caches or network mounts are verified against the current allowlist, not the allowlist that was active when they were first installed.

Signature Verification for Trusted Publishers

Publisher Identity

OCCP supports cryptographic signature verification for packages from trusted publishers. Publishers sign their packages with Ed25519 keys, and the public keys are registered in the OCCP configuration. At verification time, the package signature is checked against the registered public key for the declared publisher.

# Signature verification flow
def verify_publisher_signature(
    package: bytes,
    signature: bytes,
    publisher: str
) -> bool:
    public_key = get_publisher_key(publisher)
    if public_key is None:
        return False  # Unknown publisher = fail closed
    try:
        public_key.verify(signature, package)
        return True
    except InvalidSignature:
        audit_log("SIGNATURE_INVALID", publisher, package_hash)
        return False

Trust Hierarchy

The trust hierarchy has three levels. First-party packages (published by the OCCP project) are signed with the project's key and are implicitly allowlisted in the default configuration. Verified-publisher packages are signed by third-party publishers whose keys have been manually registered by the platform administrator. Unverified packages have no signature and must rely solely on allowlist hash verification for trust.

Preventing Supply Chain Attacks in Practice

Attack Scenario: Typosquatting

An attacker publishes mcp-filesystm (missing an 'e') with malicious code. The allowlist check stops this immediately: the typosquatted name is not on the allowlist. Even if an administrator accidentally references the wrong name, the hash will not match any approved entry.

Attack Scenario: Compromised Registry

An attacker gains write access to the package registry and replaces mcp-filesystem@1.2.3 with a trojaned version. The SHA-256 hash of the trojaned version does not match the hash recorded in the allowlist at approval time. The package is rejected at load time.

Attack Scenario: Maintainer Compromise

A legitimate maintainer's account is compromised, and the attacker publishes a new version mcp-filesystem@1.2.4 with a backdoor. Because the allowlist requires exact version pinning and each version requires independent approval, the new version is not on the allowlist. The package is rejected.

Each of these scenarios is stopped by a different layer of the verification system. The layers are complementary: allowlist verification catches unauthorized packages, static analysis catches dangerous code patterns, signature verification confirms publisher identity, and fail-closed design ensures that any verification failure results in rejection.

// Key Takeaways

  • Allowlist verification with exact version pinning and SHA-256 hashes prevents unauthorized packages from loading
  • Static analysis scans for dangerous code patterns and verifies that packages only use declared capabilities
  • Fail-closed design ensures any verification failure results in package rejection, with no fallback to unverified loading
  • Verification runs at every load, not just at install time, preventing TOCTOU attacks
  • Publisher signature verification with Ed25519 keys establishes a three-tier trust hierarchy
  • Multiple defense layers work together: each stops a different class of supply chain attack
