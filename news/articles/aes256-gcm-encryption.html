AES-256-GCM Envelope Encryption: Securing LLM API Tokens

LLM API tokens are the most sensitive credentials in an AI orchestration platform. A leaked token does not just compromise one request — it grants unrestricted access to the model provider, with costs measured in dollars per second. OCCP v0.8.2 introduces envelope encryption with AES-256-GCM and HKDF-derived per-token keys to ensure credentials are protected at rest and in transit.

The Problem: API Tokens as High-Value Targets

An AI orchestration platform manages credentials for multiple LLM providers: OpenAI, Anthropic, Google, Cohere, and others. Each provider issues API tokens that grant access to their models. These tokens are stored in the platform's configuration database and used on every API call. If an attacker gains read access to the database — through SQL injection, a backup leak, or a compromised admin account — they obtain every token simultaneously.

Plaintext token storage is the default in most AI frameworks. Some use base64 encoding, which provides no security whatsoever. Others use symmetric encryption with a single key, which means all tokens are compromised if that key is exposed. The threat model requires a system where compromising one token does not compromise all tokens, and where key compromise can be mitigated without re-encrypting every stored credential.

If your encryption strategy means that losing one key compromises every credential in the system, you have centralized your risk rather than distributing it.

Why AES-256-GCM

Authenticated Encryption

AES-256-GCM is an Authenticated Encryption with Associated Data (AEAD) cipher. Unlike AES-CBC or AES-CTR, GCM mode provides both confidentiality and integrity in a single operation. The ciphertext includes a 128-bit authentication tag that the decryption process verifies before returning plaintext. If even a single bit of the ciphertext has been modified — by an attacker, by disk corruption, or by a faulty migration script — decryption fails with an authentication error rather than returning corrupted data.

This property is critical for credential storage. With unauthenticated encryption modes, an attacker can flip bits in the ciphertext without detection. The decrypted result is garbage, but the system does not know it is garbage until the API call fails with an invalid token error. With GCM, tampering is detected immediately at decryption time, before the corrupted token reaches the provider.

Performance Characteristics

AES-256-GCM is hardware-accelerated on modern CPUs through the AES-NI instruction set. On a standard server CPU, encryption throughput exceeds 4 GB/s. For OCCP's use case — encrypting tokens that are typically 40-200 bytes — the per-operation cost is measured in microseconds. There is no performance argument against using the strongest available cipher.

# Python implementation using cryptography library
from cryptography.hazmat.primitives.ciphers.aead import AESGCM
import os

def encrypt_token(plaintext: bytes, key: bytes) -> bytes:
    """Encrypt a token with AES-256-GCM.
    Returns: nonce (12 bytes) || ciphertext || tag (16 bytes)
    """
    nonce = os.urandom(12)  # 96-bit nonce, never reused
    aesgcm = AESGCM(key)
    ciphertext = aesgcm.encrypt(nonce, plaintext, None)
    return nonce + ciphertext

HKDF Key Derivation: Per-Token Unique Keys

Encrypting all tokens with the same key creates a single point of failure. OCCP uses HMAC-based Key Derivation Function (HKDF) to derive a unique encryption key for each token from a single master key. The derivation uses a per-token salt (generated randomly at encryption time) and an info parameter that binds the derived key to its specific context.

from cryptography.hazmat.primitives.kdf.hkdf import HKDF
from cryptography.hazmat.primitives import hashes

def derive_token_key(master_key: bytes, salt: bytes,
                     provider: str, token_id: str) -> bytes:
    """Derive a unique 256-bit key for a specific token."""
    info = f"occp-token-{provider}-{token_id}".encode()
    hkdf = HKDF(
        algorithm=hashes.SHA256(),
        length=32,  # 256 bits
        salt=salt,
        info=info,
    )
    return hkdf.derive(master_key)

Why Per-Token Keys Matter

With per-token derived keys, an attacker who somehow recovers one derived key (through a side-channel attack or a memory dump) can only decrypt the specific token that key was derived for. They cannot use it to decrypt other tokens, because each token's key is derived with a different salt and info parameter. The blast radius of a key compromise is reduced from "all credentials" to "one credential."

The derivation is deterministic: given the same master key, salt, and info, HKDF always produces the same derived key. This means OCCP does not need to store derived keys. It stores the salt alongside the encrypted token, and re-derives the key at decryption time. The only secret that must be protected is the master key.

The Envelope Encryption Pattern

Two-Layer Architecture

Envelope encryption separates the encryption into two layers. Layer 1: each token is encrypted with its own data encryption key (DEK), derived via HKDF as described above. Layer 2: the master key that seeds HKDF is itself encrypted with a key encryption key (KEK) that can be stored in a hardware security module, a cloud KMS, or an environment variable depending on the deployment context.

# Envelope encryption storage format
{
    "token_id": "tok_abc123",
    "provider": "anthropic",
    "salt": "base64(random_32_bytes)",
    "encrypted_token": "base64(nonce || ciphertext || tag)",
    "kek_version": "kek-2026-02",
    "created_at": "2026-02-23T10:00:00Z"
}

Why Two Layers

The two-layer design solves the key rotation problem. When the master key needs to be rotated (scheduled rotation, suspected compromise, compliance requirement), only the master key's encryption needs to change. The individual token encryptions remain valid because the derived keys are computed from the master key at runtime. The KEK wrapping the master key is the only thing that changes during rotation.

In a single-layer system, rotating the encryption key requires decrypting every token with the old key and re-encrypting with the new key. For an organization with hundreds of API tokens across dozens of providers, this is an expensive, error-prone operation that requires the system to hold all plaintext tokens in memory simultaneously. Envelope encryption avoids this entirely.

Key Rotation Without Re-Encryption

Dual-Key Window

During key rotation, OCCP maintains a dual-key window where both the old and new KEK versions are active. When decrypting a token, the system reads the kek_version field from the stored record and uses the corresponding KEK to unwrap the master key. New encryptions use the new KEK version. Old encryptions continue to work until they are naturally re-encrypted (on token update) or until the old KEK is explicitly retired.

Rotation Triggers

Key rotation can be triggered by three events. Scheduled rotation occurs on a configurable interval (default: 90 days). Manual rotation is available through the admin API for immediate response to security incidents. Breach-triggered rotation occurs automatically when the system detects anomalous decryption patterns, such as a burst of decryption failures that might indicate a key compromise attempt.

Every rotation event is recorded in the SHA-256 audit chain with the rotation reason, the old and new KEK version identifiers (never the keys themselves), and the timestamp. This audit trail is essential for compliance reporting: auditors can verify that key rotation policies are being followed without ever seeing the actual keys.

Protection at Rest and in Transit

At-Rest Protection

Tokens are never stored in plaintext. The database contains only the encrypted blob, the salt, and the KEK version identifier. Even a full database dump yields nothing without the master key. The master key is never written to the database — it is loaded from the environment at startup and held only in process memory.

In-Transit Protection

When a token is needed for an LLM API call, it is decrypted in memory, used for the HTTP request (which is itself protected by TLS), and then the plaintext is zeroed from memory. The decrypted token is never written to disk, never included in log output, and never returned in API responses. The output sanitization middleware provides an additional safety net: even if a code path accidentally includes a token in a response, the sanitizer detects and strips credential-like strings before they reach the network.

Memory Handling

Python's garbage collector does not guarantee immediate memory cleanup, which means sensitive data can persist in memory after the variable is reassigned. OCCP mitigates this by overwriting the plaintext buffer with zeros before releasing it. While Python's memory model makes true secure erasure difficult (the runtime may have copied the string internally), the overwrite reduces the window of exposure and eliminates the most common memory dump attack vectors.

# Secure token usage pattern
def use_token_for_api_call(encrypted_record: dict) -> dict:
    plaintext = decrypt_token(encrypted_record)
    try:
        response = make_llm_call(token=plaintext)
        return response
    finally:
        # Overwrite plaintext before GC
        if isinstance(plaintext, bytearray):
            for i in range(len(plaintext)):
                plaintext[i] = 0

Integration with the OCCP Security Stack

Token encryption does not exist in isolation. It integrates with OCCP's layered security model: Casbin RBAC controls who can create, read, update, or delete tokens. The policy engine evaluates whether a given agent is authorized to use a specific provider's token. The audit chain records every token access event. And the output sanitization layer prevents accidental token leakage in API responses.

The encryption layer is the last line of defense. If RBAC is bypassed, if the policy engine has a bug, if an attacker gains database access through any path — the tokens are still encrypted. This is defense in depth: each layer protects against the failure of every other layer.

// Key Takeaways

  • AES-256-GCM provides authenticated encryption that detects both decryption failures and tampering attempts
  • HKDF derives unique per-token encryption keys from a single master key, limiting blast radius of any key compromise
  • Envelope encryption separates data encryption keys from key encryption keys, enabling rotation without re-encrypting all tokens
  • Dual-key windows allow zero-downtime key rotation with automatic fallback to the previous KEK version
  • Tokens are decrypted only in memory, never logged, and overwritten before garbage collection
  • The encryption layer integrates with RBAC, policy engine, and output sanitization for defense in depth
