GDPR by Design: Privacy Engineering for AI Platforms

The General Data Protection Regulation requires that data protection is built into systems from the outset — not retrofitted after deployment. For AI orchestration platforms that process user prompts, task data, and agent interactions, this principle demands specific architectural decisions about what data is stored, why it is stored, and how long it persists.

Why AI Platforms Have a Special Privacy Problem

Traditional web applications process structured personal data: names, email addresses, phone numbers. The GDPR obligations are well-understood — collect only what you need, store it securely, delete it when the purpose is fulfilled. AI orchestration platforms face a different challenge. The data they process is unstructured, unpredictable, and frequently contains personal information that was not explicitly requested.

When a user submits a prompt to an AI agent, that prompt might contain a customer name, a medical condition, a financial figure, or a home address. The user is not filling out a form with labeled fields. They are writing natural language that may or may not contain personal data. The platform cannot know in advance what personal data a prompt will contain, but it must handle all prompts as if they might contain personal data.

In AI systems, personal data does not arrive through defined input channels. It arrives embedded in natural language, hidden in context, and reproduced in output. Privacy engineering for AI must account for data that is unstructured, unsolicited, and unpredictable.

Data Minimization: Store Only What Audit Requires

GDPR Article 5(1)(c) establishes the principle of data minimization: personal data shall be "adequate, relevant and limited to what is necessary in relation to the purposes for which they are processed." For an AI orchestration platform, the primary purpose of data storage is audit compliance — maintaining a verifiable record of what the platform did and why.

OCCP implements data minimization through a tiered storage model:

# Tiered data storage model
class AuditEntry:
    # Tier 1: Always stored (required for audit)
    task_id: str
    agent_id: str
    timestamp: datetime
    action_type: str
    policy_result: str  # pass/fail + policy ID
    sha256_hash: str    # Hash chain integrity

    # Tier 2: Stored with retention limit (useful for debugging)
    prompt_summary: str  # Truncated, PII-stripped summary
    output_summary: str  # Truncated, sanitized summary

    # Tier 3: Never stored
    # - Full raw prompts (may contain PII)
    # - Full raw LLM responses (may contain PII)
    # - User IP addresses (not needed for audit)
    # - Session cookies or auth tokens

The tier model makes data minimization decisions explicit and enforceable. Tier 1 data is the minimum required for the audit chain to function — task identifiers, timestamps, policy results, and hash chain values. Tier 2 data provides operational value (summarized prompts and responses for debugging) but is stored with a configurable retention period and undergoes PII stripping before storage. Tier 3 data is never persisted — full prompts and responses exist only in memory during pipeline execution.

PII Stripping for Summaries

The Tier 2 summaries are not raw truncations. Before a prompt or response summary is stored, it passes through the PII detection engine that the policy engine already uses for injection prevention. Named entities, email patterns, phone number formats, and government ID patterns are replaced with category markers: [PERSON_NAME], [EMAIL], [PHONE]. This means the summary retains its structural information ("User asked about [PERSON_NAME]'s account balance") without storing the actual personal data.

Purpose Limitation: Clear Boundaries on Data Usage

GDPR Article 5(1)(b) requires that personal data is "collected for specified, explicit and legitimate purposes and not further processed in a manner that is incompatible with those purposes." For OCCP, this means defining exactly why each data element is collected and ensuring it is never used for another purpose.

The platform defines three processing purposes:

  • Task execution — Processing the user's prompt through the Verified Autonomy Pipeline to produce a result. Data used for this purpose exists only during pipeline execution and is not retained after the task completes.
  • Audit compliance — Maintaining the tamper-evident audit chain that records platform actions. Data used for this purpose is stored in the audit log with the tiered model described above.
  • Platform improvement — Aggregated, anonymized metrics about platform performance (task completion rates, average response times, error frequencies). No personal data is used for this purpose.

These purpose definitions are enforced architecturally. The task execution pipeline cannot write to the analytics database. The audit log cannot be queried by the metrics aggregation service. The purpose boundaries are not policies that can be reconfigured — they are hard-coded data flow restrictions.

Consent Management in AI Contexts

Consent management for AI platforms is more nuanced than for traditional web applications. A user consenting to "use this AI assistant" does not necessarily consent to their prompt being stored for audit, their data being processed by a specific LLM provider, or their interaction patterns being used for platform metrics.

OCCP implements granular consent through scoped consent records:

class ConsentRecord:
    user_id: str
    scope: ConsentScope  # execution, audit_storage, metrics, third_party_llm
    granted: bool
    granted_at: datetime
    expires_at: datetime | None
    revoked_at: datetime | None
    legal_basis: str  # consent, legitimate_interest, contract

Each consent scope maps to a specific processing purpose. A user can consent to task execution (necessary for the service to function) while declining metrics collection. The platform respects these boundaries at every data decision point. If a user has not consented to audit storage, their task summaries are not persisted in Tier 2 — only the Tier 1 hash chain entries (which contain no personal data) are written.

Legal Basis Beyond Consent

GDPR recognizes six legal bases for data processing, and consent is only one of them. OCCP supports multiple legal bases because enterprise deployments often rely on "legitimate interest" (Article 6(1)(f)) or "contract performance" (Article 6(1)(b)) rather than individual consent. The consent record's legal_basis field documents which basis applies to each processing scope, ensuring that the platform can demonstrate compliance regardless of which basis the deployment uses.

Data Subject Rights Implementation

GDPR grants data subjects specific rights that the platform must support. OCCP implements four of these rights in v0.6.0:

Right of Access (Article 15)

Users can request all data the platform holds about them. The /api/v1/privacy/export endpoint generates a complete data package including the user's profile data, consent records, and any Tier 2 audit summaries associated with their tasks. The export is delivered as a structured JSON file, machine-readable per the portability requirement.

Right to Erasure (Article 17)

Users can request deletion of their personal data. The platform handles this in two phases. Phase 1: immediate deletion of Tier 2 audit summaries and user profile data. Phase 2: the Tier 1 audit chain entries are anonymized (user_id replaced with a hash) rather than deleted, because deleting entries would break the SHA-256 hash chain. The anonymized entries satisfy both the erasure right and the audit integrity requirement.

Right to Rectification (Article 16)

Users can correct inaccurate personal data. For profile data (name, email, organization), this is straightforward CRUD. For data embedded in past task prompts, rectification is handled through the same PII stripping mechanism — the specific inaccurate data point is identified and the relevant audit summaries are re-processed with the corrected information.

Right to Data Portability (Article 20)

The data export endpoint produces machine-readable JSON that can be imported into another system. The export format follows the schema defined by the Data Transfer Project where applicable, ensuring interoperability with other platforms that support the same standard.

Privacy Impact Assessment

GDPR Article 35 requires a Data Protection Impact Assessment (DPIA) for processing that is "likely to result in a high risk to the rights and freedoms of natural persons." AI orchestration platforms that process unstructured user prompts fall squarely within this requirement.

OCCP's architecture documentation includes a DPIA template pre-populated with the platform's data flows, processing purposes, and safeguards. Deploying organizations can use this template as the starting point for their own DPIA, supplementing it with organization-specific context. This reduces the compliance burden from "build a DPIA from scratch" to "customize the existing DPIA for your deployment."

Privacy engineering is not about preventing data processing — it is about making data processing accountable. Every byte of personal data in OCCP has a documented purpose, a defined retention period, and a deletion mechanism. That is what GDPR by design means in practice.

Personal Data in LLM Interactions

The most challenging privacy question for AI platforms is: what happens to personal data that is sent to a third-party LLM provider? The user's prompt, which may contain personal data, is transmitted to the LLM API for processing. The platform controls what it sends and what it stores, but it does not control what the LLM provider does with the data after receipt.

OCCP addresses this through transparency and data minimization at the transmission boundary. The platform logs which data fields are sent to the LLM provider, enabling operators and data subjects to know exactly what was shared. The PII detection engine can optionally strip personal data from prompts before they are sent to the LLM, replacing detected PII with generic placeholders. This reduces the privacy impact of LLM transmission at the cost of some response quality — a tradeoff that operators configure based on their deployment's privacy requirements.

// Key Takeaways

  • Three-tier storage model ensures only audit-critical data is persisted, with PII stripping for operational summaries
  • Purpose limitation is enforced architecturally through hard-coded data flow restrictions, not configurable policies
  • Granular consent records support multiple GDPR legal bases with per-scope consent management
  • Data subject rights (access, erasure, rectification, portability) are implemented as API endpoints
  • Hash chain audit entries are anonymized rather than deleted to preserve audit integrity during erasure requests
  • Pre-populated DPIA template reduces compliance burden for deploying organizations
