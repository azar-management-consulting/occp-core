OCCP v0.5.0: Agent Management Comes to the Dashboard

Before v0.5.0, managing agents required API calls or CLI commands. Version 0.5.0 brought the full agent lifecycle into the browser: registration, configuration, real-time monitoring, and retirement. The release also introduced the agent registry -- a capability-based access control system that enforces what each agent is allowed to do.

What Changed in v0.5.0

Prior releases treated agents as opaque identifiers. The orchestrator knew that "agent-sec-01" existed and could process security-related tasks, but there was no structured way to inspect its capabilities, modify its configuration, or track its operational history through the dashboard. Agent management was an API-only operation.

Version 0.5.0 introduced four major capabilities:

  • Agent registration UI -- a form-based interface for registering new agents with capability declarations
  • Configuration management -- parameter editing for registered agents (max concurrent tasks, timeout settings, capability list)
  • Real-time monitoring -- live status, heartbeat tracking, and task history per agent
  • Agent retirement workflow -- graceful decommissioning with task reassignment

Agent Registration with Capability Declarations

When a new agent is registered in OCCP, it must declare its capabilities. Capabilities are not just labels -- they are the basis for task routing and access control. An agent registered with ["code_analysis", "vulnerability_scan"] will only receive tasks that match those capabilities. It cannot claim tasks requiring "data_processing" or "content_generation".

// Agent registration request
POST /api/v1/agents
{
  "name": "security-scanner-01",
  "description": "Static analysis and vulnerability detection agent",
  "capabilities": [
    "code_analysis",
    "vulnerability_scan",
    "dependency_audit"
  ],
  "config": {
    "max_concurrent_tasks": 3,
    "task_timeout_seconds": 300,
    "heartbeat_interval_seconds": 30
  }
}

// Response
{
  "id": "agent-a1b2c3d4",
  "name": "security-scanner-01",
  "status": "registered",
  "capabilities": ["code_analysis", "vulnerability_scan", "dependency_audit"],
  "created_at": "2026-01-18T10:00:00Z",
  "api_key": "agent-key-..." // Only returned once, at registration
}

The agent API key is returned only during registration. It is never stored in plain text on the server and cannot be retrieved later. If the key is lost, the agent must be re-registered. This follows the principle of least exposure: credentials exist in recoverable form for the minimum possible duration.

The Registration Form

The dashboard registration form maps directly to this API. Each field includes validation feedback and capability suggestions based on existing agents in the registry:

"use client";

import { useState } from "react";
import { CapabilitySelector } from "./CapabilitySelector";

export function AgentRegistrationForm() {
  const [name, setName] = useState("");
  const [capabilities, setCapabilities] = useState<string[]>([]);
  const [maxConcurrent, setMaxConcurrent] = useState(3);
  const [apiKey, setApiKey] = useState<string | null>(null);

  const handleSubmit = async (e: React.FormEvent) => {
    e.preventDefault();
    const response = await authenticatedFetch("/api/v1/agents", {
      method: "POST",
      body: JSON.stringify({
        name,
        capabilities,
        config: { max_concurrent_tasks: maxConcurrent },
      }),
    });
    const data = await response.json();
    setApiKey(data.api_key);
    // Show the API key exactly once
  };

  if (apiKey) {
    return (
      <div className="api-key-display">
        <h3>Agent Registered Successfully</h3>
        <p>Save this API key now. It will not be shown again.</p>
        <code className="api-key">{apiKey}</code>
      </div>
    );
  }

  return (
    <form onSubmit={handleSubmit}>
      <input value={name} onChange={(e) => setName(e.target.value)}
             placeholder="Agent name" required />
      <CapabilitySelector
        selected={capabilities}
        onChange={setCapabilities} />
      <input type="number" value={maxConcurrent} min={1} max={10}
             onChange={(e) => setMaxConcurrent(Number(e.target.value))} />
      <button type="submit">Register Agent</button>
    </form>
  );
}
The CapabilitySelector component shows the union of all capabilities across registered agents, suggesting standardized names while allowing custom entries. This encourages consistency without enforcing a rigid taxonomy. If five agents use "code_analysis" and one agent uses "code-analysis", the selector highlights the inconsistency.

Configuration UI for Agent Parameters

After registration, agent parameters can be modified through the dashboard. The configuration panel shows current values alongside the defaults, making it clear which settings have been customized:

// Agent configuration update
PATCH /api/v1/agents/agent-a1b2c3d4
{
  "config": {
    "max_concurrent_tasks": 5,
    "task_timeout_seconds": 600
  }
}

Configuration changes take effect immediately. The orchestrator reads agent configuration at task assignment time, so increasing max_concurrent_tasks from 3 to 5 allows the agent to claim additional tasks on its next heartbeat. There is no restart required -- the configuration is dynamic.

Capability Modification

Capabilities can be added or removed after registration, but with constraints. Removing a capability while the agent has active tasks using that capability is blocked. The agent must complete or reassign those tasks first. This prevents a situation where an agent loses a capability mid-task, causing the pipeline to fail during execution.

// Attempt to remove a capability with active tasks
PATCH /api/v1/agents/agent-a1b2c3d4
{ "capabilities": ["code_analysis"] }  // removing vulnerability_scan

// Response: 409 Conflict
{
  "error": "capability_in_use",
  "message": "Cannot remove 'vulnerability_scan': 2 active tasks require it",
  "active_tasks": ["task-x1y2z3", "task-a4b5c6"]
}

Real-Time Agent Monitoring

The agent monitoring panel displays each agent as a card with live-updating status. The card shows the agent's name, capabilities (as colored badges), current task count, and last heartbeat timestamp. The status dot uses the same color system as the rest of the dashboard:

  • Green (online) -- heartbeat received within the expected interval
  • Amber (busy) -- agent is at maximum concurrent tasks
  • Gray (offline) -- no heartbeat received for 3x the expected interval
  • Red (error) -- agent reported an error state in its last heartbeat

The heartbeat mechanism is the foundation of agent monitoring. Each agent sends periodic heartbeat requests to the API, reporting its status, current task count, and resource usage:

// Agent heartbeat (sent by the agent process)
POST /api/v1/agents/agent-a1b2c3d4/heartbeat
{
  "status": "active",
  "current_tasks": 2,
  "resource_usage": {
    "cpu_percent": 45,
    "memory_mb": 512,
    "disk_mb": 1024
  }
}

The dashboard receives heartbeat updates through the WebSocket connection. When a heartbeat arrives, the corresponding agent card updates its status dot, task count, and resource indicators without a page refresh. If a heartbeat is missed (3x the interval passes without a heartbeat), the status transitions from green to gray, alerting the operator to a potential agent failure.

Agent Retirement Workflow

Agents cannot simply be deleted. An agent may have active tasks, pending task claims, and historical records in the audit trail. The retirement workflow handles these concerns in order:

  1. Stop task assignment. The agent is marked as "retiring" and the orchestrator stops assigning new tasks to it.
  2. Complete or reassign active tasks. Active tasks either run to completion (if the agent is still responsive) or are reassigned to other agents with matching capabilities.
  3. Archive agent record. The agent's registration, configuration history, and task history are archived. The agent ID is preserved in the audit trail for compliance.
  4. Revoke credentials. The agent's API key is invalidated. Any subsequent requests using that key receive 401 Unauthorized.
// Initiate retirement
POST /api/v1/agents/agent-a1b2c3d4/retire
{
  "reason": "Replaced by security-scanner-02",
  "reassign_tasks": true
}

// Response
{
  "status": "retiring",
  "active_tasks_reassigned": 2,
  "estimated_completion": "2026-01-18T10:05:00Z"
}

The retirement is asynchronous. The dashboard shows a "Retiring..." status with a progress indicator as active tasks complete or reassign. Once all tasks are handled and the agent's credentials are revoked, the status changes to "Retired" and the agent moves to the archived section of the dashboard.

The Agent Registry as Access Control

The agent registry is more than a database of agent records. It functions as a capability-based access control system. The orchestrator consults the registry before every task assignment:

  1. A task is submitted with a required capability (e.g., vulnerability_scan)
  2. The orchestrator queries the registry for active agents with that capability
  3. Matching agents are filtered by availability (not at max concurrent tasks, heartbeat recent)
  4. The highest-priority available agent claims the task

This capability-based routing means an agent can never execute a task type it was not registered for. Even if an attacker compromises an agent's API key, the compromised agent can only claim tasks matching its declared capabilities. An agent registered for code_analysis cannot claim data_export tasks, limiting the blast radius of a credential compromise.

The agent registry transforms capability declarations from documentation into enforcement. What an agent says it can do becomes what it is allowed to do. The declaration is the policy.

Metrics from the Release

Version 0.5.0 added 31 new tests covering agent registration, capability validation, retirement workflow, and the dashboard components. The total test count reached 138. The API added 6 new endpoints (register, update, heartbeat, retire, list, get-by-id) and the dashboard added 4 new pages (agent list, agent detail, registration form, retirement confirmation).

The agent management dashboard introduced the first use of optimistic updates in the OCCP frontend. When an operator changes an agent's configuration, the UI updates immediately and rolls back if the API call fails. This pattern reduced perceived latency for configuration changes from 200ms (round-trip to API) to effectively zero.

// Key Takeaways

  • v0.5.0 moved full agent lifecycle management from API-only to the browser dashboard
  • Capability declarations are not labels -- they enforce task routing and access control
  • Agent API keys are shown once at registration, never stored in recoverable form
  • Capability removal is blocked while active tasks depend on that capability
  • Retirement workflow: stop assignment, reassign tasks, archive record, revoke credentials
  • The agent registry functions as a capability-based access control enforcement layer
