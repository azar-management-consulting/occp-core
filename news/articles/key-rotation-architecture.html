Key Rotation Architecture: Zero-Downtime Credential Management

Cryptographic key rotation is a compliance requirement for every enterprise security standard: PCI DSS, SOC 2, ISO 27001, and the EU AI Act's record-keeping provisions all mandate periodic key rotation. The engineering challenge is performing rotation without service interruption — every encrypted credential must remain accessible throughout the rotation process. OCCP's dual-key rotation architecture solves this problem.

The Key Rotation Challenge

In a system using envelope encryption (as described in the AES-256-GCM article), the key encryption key (KEK) wraps the master key, which seeds the HKDF derivation for per-token data encryption keys. When the KEK must be rotated, the straightforward approach is: decrypt the master key with the old KEK, re-encrypt it with the new KEK, and update the stored record. This sounds simple until you consider the operational constraints.

The system is serving requests while rotation occurs. Multiple application instances may be running behind a load balancer. Each instance has the current KEK cached in memory. The rotation must propagate to all instances without requiring a synchronized restart. And throughout this process, every stored credential must remain decryptable — a customer's LLM API call must not fail because the platform is rotating its internal keys.

Key rotation is not a cryptographic challenge. It is a distributed systems challenge that happens to involve cryptography.

Dual-Key Support During Rotation Windows

The Key Version Registry

OCCP maintains a key version registry that maps version identifiers to their corresponding KEKs. At any point in time, the registry contains at most two active versions: the current version (used for new encryptions) and the previous version (accepted for decryption of existing records). The registry is stored in the application's configuration and loaded into memory at startup.

{
    "current_version": "kek-2026-02",
    "previous_version": "kek-2025-11",
    "versions": {

            "status": "current",
            "activated_at": "2026-02-15T00:00:00Z",
            "source": "env:OCCP_KEK_CURRENT"
        },
        "kek-2025-11": {
            "status": "previous",
            "activated_at": "2025-11-01T00:00:00Z",
            "retire_after": "2026-05-15T00:00:00Z",
            "source": "env:OCCP_KEK_PREVIOUS"
        }
    }
}

Encryption: Always Use Current

When encrypting a new credential or re-encrypting an existing one, the system always uses the current KEK version. The version identifier is stored alongside the encrypted record, creating a binding between the ciphertext and the key that produced it. This binding is critical for the decryption path.

Decryption: Try Current, Fall Back to Previous

When decrypting a credential, the system reads the kek_version field from the stored record. If the version matches the current KEK, decryption proceeds normally. If the version matches the previous KEK, decryption uses the previous key. If the version matches neither (an unknown or retired version), decryption fails with an explicit error rather than trying every key in the system.

def decrypt_credential(record: EncryptedRecord) -> bytes:
    """Decrypt with dual-key fallback."""
    kek_version = record.kek_version
    registry = get_key_registry()

    if kek_version == registry.current_version:
        kek = registry.get_key(registry.current_version)
    elif kek_version == registry.previous_version:
        kek = registry.get_key(registry.previous_version)
        # Log that this record uses the old key
        audit_log("KEY_FALLBACK", record_id=record.id,
                  version=kek_version)
    else:
        audit_log("KEY_VERSION_UNKNOWN", record_id=record.id,
                  version=kek_version)
        raise KeyVersionError(
            f"Unknown KEK version: {kek_version}. "
            f"Active versions: {registry.current_version}, "
            f"{registry.previous_version}"
        )

    master_key = unwrap_master_key(kek, record.wrapped_master_key)
    return decrypt_with_master_key(master_key, record)

The Rotation Process

Step 1: Generate New KEK

A new KEK is generated using a cryptographically secure random number generator. The new key is provisioned to all application instances through the configuration system (environment variables, secrets manager, or KMS). At this stage, no instances are using the new key yet.

Step 2: Promote New KEK to Current

The key version registry is updated: the current version becomes the previous version, and the new version becomes the current version. Application instances pick up this change at their next configuration refresh (typically within seconds for environment-based config, or at next startup for file-based config).

Step 3: Gradual Re-Encryption

After the new KEK is active, existing credentials are gradually re-encrypted with the new version. This happens opportunistically: when a credential is accessed for any reason (an LLM API call, a configuration view, a health check), the system checks its kek_version. If it uses the previous version, the system decrypts with the old key and re-encrypts with the new key in the same transaction. Over time, all credentials migrate to the new key version without requiring a batch re-encryption job.

async def access_credential(record_id: str) -> bytes:
    """Access a credential, opportunistically re-encrypting."""
    record = await get_encrypted_record(record_id)
    plaintext = decrypt_credential(record)

    # Opportunistic re-encryption
    registry = get_key_registry()
    if record.kek_version != registry.current_version:
        new_record = encrypt_credential(
            plaintext, registry.current_version
        )
        await update_encrypted_record(record_id, new_record)
        audit_log("KEY_REENCRYPT", record_id=record_id,
                  old_version=record.kek_version,
                  new_version=registry.current_version)

    return plaintext

Step 4: Retire Previous KEK

After sufficient time has passed for all credentials to be opportunistically re-encrypted (or after a manual sweep ensures no credentials remain on the old version), the previous KEK is retired. The key is removed from the registry, and any subsequent attempt to decrypt a record with the retired version fails with an explicit error. The retirement event is recorded in the audit chain.

Rotation Triggers

Scheduled Rotation

OCCP supports configurable scheduled rotation intervals. The default is 90 days, aligned with common compliance requirements. The rotation schedule is tracked in the key version registry, and the health check endpoint reports when the next rotation is due. Administrators receive warnings (through the dashboard and monitoring alerts) when a rotation is approaching.

Manual Rotation

Administrators can trigger immediate rotation through the admin API. Manual rotation follows the same dual-key process as scheduled rotation but is initiated on demand. The most common reason for manual rotation is personnel changes: when an engineer with access to the KEK leaves the organization, the KEK should be rotated.

Breach-Triggered Rotation

OCCP monitors for anomalous decryption patterns that may indicate key compromise. A burst of decryption failures (indicating someone is trying incorrect keys), decryption requests from unexpected network locations, or decryption of credentials that are not needed for any active task — these patterns trigger an automatic rotation alert. Automatic breach-triggered rotation can be enabled, but it defaults to alert-only mode because automatic rotation during an active breach can create operational confusion.

Audit Logging of Rotation Events

Every key rotation event produces a set of audit entries in the SHA-256 chain. The rotation initiation entry records the trigger (scheduled, manual, breach), the initiator (system, admin user ID), and the new version identifier. The rotation completion entry records the count of credentials that were immediately re-encrypted and the count still pending. Subsequent opportunistic re-encryptions are logged individually. The retirement entry records the final count of credentials migrated and confirms that no credentials remain on the retired version.

Zero-Downtime Guarantees

The dual-key architecture ensures zero downtime during rotation. At no point during the rotation process are credentials inaccessible. Old credentials work because the previous KEK is still active. New credentials work because they use the current KEK. The transition is seamless from the perspective of any system that accesses encrypted credentials.

The only risk window is between Step 1 (generating the new KEK) and Step 2 (promoting it). If the system crashes during this window, the new KEK may not be persisted. This is mitigated by generating and persisting the new KEK before updating the registry — the registry update is the atomic commit point. If the system crashes before the registry update, the new key exists but is unused, and rotation can be retried. If it crashes after the registry update, the rotation is complete and the dual-key window is active.

// Key Takeaways

  • Dual-key windows maintain two active KEK versions simultaneously, ensuring all credentials remain decryptable during rotation
  • Encryption always uses the current KEK version; decryption falls back to the previous version if needed
  • Opportunistic re-encryption migrates credentials to the new key version without batch processing
  • Three rotation triggers: scheduled (default 90 days), manual (admin-initiated), and breach-detected (alert or auto)
  • Every rotation event is recorded in the SHA-256 audit chain with full traceability
  • Zero-downtime guarantee: at no point during rotation are any credentials inaccessible
