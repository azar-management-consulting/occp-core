OCCP v0.3.0: LLM Integration and Production Intelligence

Version 0.3.0 marks the moment OCCP stopped being a task queue with aspirations and became a production-intelligent platform. Real Claude API integration, multi-pattern prompt injection detection, and a dashboard with live pipeline visualization -- this release connected the orchestration engine to actual AI capability.

What Changed in v0.3.0

Prior to v0.3.0, OCCP orchestrated tasks through the Verified Autonomy Pipeline but never made actual LLM calls. The pipeline stages (Plan, Gate, Execute, Validate, Ship) were functional, but the Execute stage used mock responses. The platform could schedule, validate, and audit tasks -- it just could not think.

Version 0.3.0 added three major capabilities:

  • Real Claude API integration via the official Anthropic Python SDK
  • Multi-pattern prompt injection detection upgraded from basic string matching to a comprehensive guard system
  • Dashboard with real-time pipeline visualization using WebSocket connections for live stage updates

Together, these additions transformed OCCP from an orchestration framework into an intelligent orchestration platform.

Claude API Integration

The integration uses the official anthropic Python SDK to make calls to the Claude API. The LLM adapter sits within the Execute stage of the pipeline, meaning every call passes through the Gate stage (policy evaluation) before reaching Claude:

from anthropic import Anthropic

class ClaudeLLMAdapter:
    def __init__(self, api_key: str, model: str = "claude-sonnet-4-20250514"):
        self.client = Anthropic(api_key=api_key)
        self.model = model

    async def execute(self, task_context: TaskContext) -> LLMResponse:
        # The prompt has already passed through the Gate stage
        # Policy engine has verified: no injection, no PII, within limits
        response = self.client.messages.create(
            model=self.model,
            max_tokens=task_context.max_tokens,
            messages=[{
                "role": "user",
                "content": task_context.prompt
            }]
        )
        return LLMResponse(
            content=response.content[0].text,
            model=response.model,
            usage={
                "input_tokens": response.usage.input_tokens,
                "output_tokens": response.usage.output_tokens
            }
        )

The Adapter Pattern

The LLM integration uses an adapter pattern rather than direct SDK calls in the orchestrator. This abstraction serves two purposes. First, it enables testing: the test suite uses a mock adapter that returns deterministic responses, allowing pipeline tests to run without API credentials. Second, it enables provider flexibility: adding support for other LLM providers requires implementing a new adapter class, not modifying the orchestrator.

# Protocol defining the LLM adapter interface
class LLMAdapter(Protocol):
    async def execute(self, task_context: TaskContext) -> LLMResponse: ...

# Mock adapter for testing
class MockLLMAdapter:
    async def execute(self, task_context: TaskContext) -> LLMResponse:
        return LLMResponse(
            content="Mock response for testing",
            model="mock-v1",
            usage={"input_tokens": 10, "output_tokens": 5}
        )
The adapter pattern was not premature abstraction. We tested against two LLM providers during development and the interface stabilized naturally. When the interface emerges from real usage rather than speculation, it tends to be the right one.

Prompt Injection Detection

With real LLM calls flowing through the pipeline, prompt injection defense moved from theoretical to critical. The v0.3.0 Gate stage includes a multi-pattern detection system that evaluates every prompt before it reaches the LLM adapter.

Detection Categories

The guard system organizes detection patterns into categories based on attack vector:

  • Instruction override -- patterns like "ignore previous instructions", "disregard your system prompt", "forget everything above"
  • Role manipulation -- attempts to change the model's behavior through "you are now", "act as", "pretend to be" prefixes
  • System prompt extraction -- queries designed to reveal the system prompt like "repeat your instructions", "what were you told"
  • Encoding evasion -- base64-encoded instructions, Unicode homoglyphs, and character substitution patterns

Each category runs independently. A prompt must pass all categories to proceed. This architecture means a new detection pattern can be added to any category without affecting the others, and the system fails closed -- if any guard raises a flag, the entire request is blocked.

class PromptInjectionGuard:
    def evaluate(self, prompt: str) -> GuardResult:
        results = []
        for pattern_set in self.pattern_categories:
            result = pattern_set.check(prompt)
            results.append(result)
            if result.blocked:
                return GuardResult(
                    allowed=False,
                    reason=result.reason,
                    category=pattern_set.name,
                    pattern_id=result.pattern_id
                )
        return GuardResult(allowed=True)

The initial v0.3.0 release shipped with 8 detection patterns across the four categories. This number would grow to 24 by v0.6.0 as we encountered and catalogued new attack vectors. The guard architecture was designed for this growth -- adding a pattern is a configuration change, not a code change.

Dashboard with Pipeline Visualization

The v0.3.0 dashboard introduced real-time pipeline visualization. Operators can watch tasks move through the five stages of the Verified Autonomy Pipeline as they happen. The visualization uses a horizontal stage indicator with color-coded status:

  • Gray -- pending, not yet reached
  • Amber -- currently active, in progress
  • Green -- completed successfully
  • Red -- failed or blocked by policy

WebSocket Architecture

The live updates are powered by WebSocket connections from the dashboard to the API server. When a task transitions between pipeline stages, the API broadcasts the state change to all connected dashboard clients:

# Server-side WebSocket broadcast
@app.websocket("/ws/pipeline/{task_id}")
async def pipeline_ws(websocket: WebSocket, task_id: str):
    await websocket.accept()
    try:
        async for event in pipeline_events(task_id):
            await websocket.send_json({
                "type": "stage_transition",
                "task_id": task_id,
                "from_stage": event.from_stage,
                "to_stage": event.to_stage,
                "timestamp": event.timestamp.isoformat(),
                "duration_ms": event.duration_ms
            })
    except WebSocketDisconnect:
        pass

The WebSocket connection is scoped to a specific task ID, so the dashboard only receives updates for tasks it is currently displaying. This prevents bandwidth waste when many tasks are running simultaneously.

The Production Intelligence Threshold

Version 0.3.0 represents a qualitative threshold in the OCCP platform's evolution. Before this release, the pipeline processed data through well-defined stages but produced deterministic, predictable outputs. After this release, the pipeline incorporates non-deterministic LLM responses that require validation, sanitization, and audit tracking at a fundamentally different level.

The combination of real LLM integration with injection detection and live monitoring creates a feedback loop: operators see pipeline execution in real time, injection attempts are visible in the dashboard, and the audit trail captures every LLM interaction with its input, output, and token usage. This visibility is what distinguishes an orchestration platform from a script that calls an API.

Metrics from the Release

The v0.3.0 release included 47 new tests, bringing the total to 84. Test coverage for the injection detection module reached 94%. The Claude adapter integration required adding a single new dependency (anthropic) to the API service -- the first external dependency beyond the web framework itself. The WebSocket implementation added zero new dependencies, using FastAPI's built-in WebSocket support from Starlette.
