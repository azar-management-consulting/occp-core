Output Sanitization: Preventing Credential Leakage in AI Responses

Large language models are trained on datasets that include configuration files, code repositories, and documentation — all rich with credential patterns. When these models generate responses, they can reproduce patterns that look exactly like real API keys, database passwords, and authentication tokens. The output sanitization guard is the last line of defense before those patterns reach a user.

The Problem: Uncontrolled LLM Output

Consider a scenario where an AI agent is asked to help configure a database connection. The LLM generates a helpful response that includes an example connection string. That example string might contain a pattern indistinguishable from a real PostgreSQL connection URI with embedded credentials. The model did not "leak" a real credential — it synthesized a plausible one. But the user receiving the response cannot know this, and if the pattern matches a real credential format, automated scanning tools will flag it.

The more dangerous case is when context injection or retrieval-augmented generation feeds the LLM real credential values from internal documents. In this scenario, the LLM may include actual secrets in its response — not because it was asked to, but because the credential appeared in the context window and the model treated it as relevant information.

The LLM does not understand what a credential is. It understands that certain token sequences frequently appear in responses about database configuration. Output sanitization is the mechanism that imposes human understanding of sensitive patterns onto machine-generated text.

The 7-Pattern Guard

The OCCP output sanitization guard detects and redacts seven categories of credential patterns. Each pattern was selected based on empirical analysis of LLM output across thousands of task executions during development, combined with established credential format specifications from the respective service providers.

Pattern 1: AWS Access Keys

AWS access key IDs follow a known format: 20 uppercase alphanumeric characters beginning with AKIA. The guard matches this pattern and its associated secret access key format. Redaction replaces the key with [AWS_KEY_REDACTED] while preserving the surrounding text structure.

Pattern 2: JWT Tokens

JSON Web Tokens have a distinctive three-part structure separated by dots, where each part is a base64url-encoded string. The guard detects strings matching this pattern when they exceed a minimum length threshold (to avoid false positives on short dotted strings). JWT tokens are particularly common in LLM output because they appear frequently in API documentation and tutorial content.

Pattern 3: Database Connection URIs

Connection strings for PostgreSQL, MySQL, MongoDB, and Redis follow predictable URI schemes: postgresql://user:password@host:port/dbname. The guard detects these URI patterns and redacts the credential portion while preserving the scheme and host information. This allows the response to remain useful ("connect to postgresql://[REDACTED]@db.example.com:5432/mydb") while removing the sensitive component.

Pattern 4: Generic API Keys

Many services use API keys with distinctive prefixes: sk- for certain providers, key- for others, and various service-specific prefixes. The guard maintains a list of known prefixes combined with minimum-length requirements to detect generic API key patterns without excessive false positives.

Pattern 5: Private Keys

PEM-encoded private keys are bounded by recognizable header and footer markers: -----BEGIN RSA PRIVATE KEY----- and its variants. The guard detects these boundaries and redacts the entire key block. This pattern has the lowest false positive rate because the markers are unambiguous.

Pattern 6: Connection Strings

Enterprise systems frequently use connection strings with key-value pair formats: Server=host;Database=db;User Id=user;Password=pass;. The guard detects the Password= pattern within connection string contexts and redacts the value while preserving the structure.

Pattern 7: Bearer Tokens

HTTP authorization headers with bearer tokens appear frequently in API documentation examples. The guard detects Bearer followed by a long alphanumeric string and redacts the token value.

Implementation Architecture

The sanitization guard is implemented as a pipeline stage that processes text through all seven patterns sequentially. Each pattern is a compiled regular expression for performance, and the guard maintains match statistics for monitoring.

import re
from dataclasses import dataclass, field

@dataclass
class SanitizationResult:
    text: str
    redactions: int = 0
    patterns_matched: list[str] = field(default_factory=list)

class OutputGuard:
    def __init__(self):
        self._patterns = self._compile_patterns()

    def _compile_patterns(self) -> list[tuple[str, re.Pattern, str]]:
        return [
            ("aws_key", re.compile(r'AKIA[A-Z0-9]{16}'), '[AWS_KEY_REDACTED]'),
            ("jwt_token", re.compile(
                r'eyJ[A-Za-z0-9_-]{10,}\.eyJ[A-Za-z0-9_-]{10,}\.[A-Za-z0-9_-]{10,}'
            ), '[JWT_REDACTED]'),
            ("db_uri", re.compile(
                r'(postgresql|mysql|mongodb|redis)://[^:]+:[^@]+@'
            ), r'\1://[CREDENTIALS_REDACTED]@'),
            # ... patterns 4-7
        ]

    def sanitize(self, text: str) -> SanitizationResult:
        result = SanitizationResult(text=text)
        for name, pattern, replacement in self._patterns:
            new_text, count = pattern.subn(replacement, result.text)
            if count > 0:
                result.redactions += count
                result.patterns_matched.append(name)
                result.text = new_text
        return result

The SanitizationResult object carries both the sanitized text and metadata about what was redacted. This metadata is essential for monitoring — if a particular pattern is matching frequently, it may indicate that the LLM is receiving too many credential examples in its context, which is a separate problem to address.

Handling False Positives

The primary engineering challenge in output sanitization is false positives. A guard that is too aggressive will destroy legitimate content — redacting example API keys in tutorial responses, mangling base64-encoded data that happens to look like a JWT, or stripping connection string examples that the user explicitly requested.

OCCP addresses this through three mechanisms:

  • Pattern specificity — Each regex is crafted to match the specific format of real credentials, not just similar-looking strings. The AWS key pattern requires the AKIA prefix plus exactly 16 uppercase alphanumeric characters, which matches real key IDs with high precision.
  • Minimum length thresholds — Short strings that partially match a pattern are not redacted. A three-character string matching the start of a JWT pattern is ignored; a 100-character string matching the full three-part structure is redacted.
  • Context preservation — Redaction replaces only the sensitive portion of a match, not the surrounding context. A database URI keeps its scheme and host; only the credentials are removed. This preserves the response's informational value.
A sanitization guard that breaks legitimate responses is worse than no guard at all, because operators will disable it. The guard must be precise enough that operators trust it and leave it enabled.

Position in the Pipeline

The output guard sits at a critical position in the Verified Autonomy Pipeline: after the Execute stage (LLM response generated) and before both the Validate stage (output verification) and the audit log write. This positioning has two consequences.

First, the Validate stage sees sanitized output. If validation checks include format verification or content analysis, they operate on the same text that the user will receive. There is no discrepancy between what was validated and what was delivered.

Second, the audit log records sanitized output. This is a deliberate design decision. The audit log is a compliance artifact that may be reviewed by multiple parties. Storing unsanitized LLM output that might contain credential patterns in the audit log would transfer the security risk from the response to the log store. By sanitizing before logging, OCCP ensures that credential patterns do not persist anywhere in the system.

Monitoring and Metrics

The guard exposes three metrics: total redactions per time window, redactions by pattern type, and redaction rate (redactions per 100 responses). A sudden increase in the redaction rate signals that something has changed — perhaps the LLM's context is being populated with more credential-laden documents, or a new integration is exposing sensitive configuration.

These metrics feed into the dashboard's security panel, where operators can monitor sanitization activity in real time and investigate spikes. The pattern breakdown is particularly useful: if JWT redactions spike but other patterns remain stable, it narrows the investigation to token handling in the current context.

// Key Takeaways

  • 7 compiled regex patterns detect AWS keys, JWT tokens, database URIs, API keys, private keys, connection strings, and bearer tokens
  • The guard operates post-LLM, pre-validation, and pre-audit to ensure no credential patterns persist anywhere in the system
  • False positive mitigation through pattern specificity, minimum length thresholds, and context-preserving partial redaction
  • SanitizationResult metadata enables monitoring of redaction frequency and pattern distribution
  • The guard cannot be disabled in the Community Edition — output sanitization is a platform guarantee
