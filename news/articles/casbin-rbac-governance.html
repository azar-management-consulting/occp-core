Casbin RBAC: Role-Based Access Control for AI Governance

In AI orchestration, the principle of least privilege is not just a security best practice — it is an operational necessity. An operator who can run tasks but not modify agent configurations cannot accidentally break the agent fleet. A viewer who can see metrics but not trigger executions cannot accidentally consume LLM tokens. Casbin makes these boundaries enforceable.

Why Casbin

Building a custom RBAC implementation is tempting but inadvisable. Authorization logic is deceptively complex: role hierarchies, resource wildcards, conditional policies, and audit requirements create a combinatorial surface that grows with every endpoint. A custom implementation either starts simple and accrues technical debt, or starts comprehensive and takes months to stabilize.

Casbin is a purpose-built authorization library that separates the access control model from the enforcement logic. The model (who can do what to which resource) is defined in a configuration file. The enforcement logic is a single function call: enforcer.enforce(subject, object, action). The model can be changed without modifying application code. The enforcement can be tested independently of the model. This separation makes the system auditable and maintainable.

OCCP chose Casbin specifically because it supports RBAC with role hierarchies out of the box, has a Python async adapter, integrates cleanly with FastAPI middleware, and stores policies in a format that non-developers can read and review.

The 4-Role Hierarchy

OCCP's role model was designed by analyzing the actual operations that different team members need to perform on an AI orchestration platform. Four distinct operation profiles emerged:

Viewer

The viewer role provides read-only access to the entire platform. Viewers can see task lists, agent registrations, audit logs, dashboard metrics, and pipeline status. They cannot create, modify, or delete any resource. This role is designed for stakeholders who need visibility into AI operations without the ability to affect them — compliance officers, project managers, or executives.

Operator

The operator role inherits all viewer permissions and adds the ability to create tasks and trigger pipeline executions. Operators are the primary users of the platform — they submit work to AI agents and monitor the results. They cannot modify agent configurations, change policies, or manage other users. This boundary ensures that operators cannot accidentally alter the governance framework they operate within.

Admin

The admin role inherits all operator permissions and adds agent management, policy modification, and user administration. Admins configure the platform: they register agents, define capability boundaries, set policy rules, and manage team access. They cannot modify core platform settings or perform destructive operations like data purging.

System Admin

The system_admin role has unrestricted access to every endpoint and every operation. This role exists for platform maintenance: updating configuration, managing RBAC policies themselves, accessing diagnostic endpoints, and performing operations that could be destructive if misused. In most deployments, only one or two individuals hold this role.

Policy Model Configuration

Casbin's policy model is defined in a configuration file that specifies the request format, the policy format, the role hierarchy, and the matching logic.

# model.conf — RBAC with role hierarchy
[request_definition]
r = sub, obj, act

[policy_definition]
p = sub, obj, act

[role_definition]
g = _, _

[policy_effect]
e = some(where (p.eft == allow))

[matchers]
m = g(r.sub, p.sub) && keyMatch2(r.obj, p.obj) && r.act == p.act

The keyMatch2 function in the matcher enables path-based resource matching. A policy for /api/v1/tasks/* matches /api/v1/tasks/abc-123 and /api/v1/tasks/def-456 without needing a separate policy entry for each task ID. This keeps the policy file manageable even as the number of resources grows.

Policy Definitions

The actual permissions are defined in a policy file that maps roles to resources and actions:

# policy.csv — Role-to-permission mappings
# Viewer permissions
p, viewer, /api/v1/tasks,       GET
p, viewer, /api/v1/tasks/*,     GET
p, viewer, /api/v1/agents,      GET
p, viewer, /api/v1/agents/*,    GET
p, viewer, /api/v1/audit,       GET
p, viewer, /api/v1/audit/*,     GET
p, viewer, /api/v1/metrics,     GET

# Operator permissions (additional)
p, operator, /api/v1/tasks,     POST
p, operator, /api/v1/tasks/*/execute, POST
p, operator, /api/v1/tasks/*,   GET

# Admin permissions (additional)
p, admin, /api/v1/agents,       POST
p, admin, /api/v1/agents/*,     PUT
p, admin, /api/v1/agents/*,     DELETE
p, admin, /api/v1/policies,     POST
p, admin, /api/v1/policies/*,   PUT
p, admin, /api/v1/users,        POST
p, admin, /api/v1/users/*,      PUT

# System admin — full access
p, system_admin, /api/v1/*,     GET
p, system_admin, /api/v1/*,     POST
p, system_admin, /api/v1/*,     PUT
p, system_admin, /api/v1/*,     DELETE

# Role hierarchy
g, operator, viewer
g, admin, operator
g, system_admin, admin

The last three lines define the role hierarchy using Casbin's grouping syntax. g, operator, viewer means "operator inherits all permissions that viewer has." This chain means that system_admin inherits from admin, which inherits from operator, which inherits from viewer. Each role only needs to define its additional permissions — inherited permissions are automatic.

JWT Integration

The user's role is embedded in their JWT token as a claim. When the user authenticates, the authentication endpoint reads their role from the database and includes it in the token payload:

{
  "sub": "user-id-abc-123",
  "email": "operator@example.com",
  "role": "operator",
  "iat": 1738000000,
  "exp": 1738086400
}

The RBAC middleware extracts the role claim from the validated JWT token and passes it to the Casbin enforcer as the subject. This means role changes take effect when the user's current token expires and they obtain a new one. For immediate role changes (such as revoking admin access during an incident), the token revocation endpoint invalidates the user's current token, forcing re-authentication with the updated role.

Middleware Integration

The RBAC check is implemented as FastAPI middleware that runs after JWT validation and before the endpoint handler. The middleware maps the incoming request to a Casbin enforcement check:

from fastapi import Request, HTTPException
import casbin

class RBACMiddleware:
    def __init__(self, enforcer: casbin.Enforcer):
        self.enforcer = enforcer

    async def __call__(self, request: Request, call_next):
        # Skip public endpoints
        if request.url.path in PUBLIC_PATHS:
            return await call_next(request)

        # Extract role from JWT (already validated by auth middleware)
        role = request.state.user_role

        # Map request to Casbin check
        resource = request.url.path
        action = request.method

        if not self.enforcer.enforce(role, resource, action):
            raise HTTPException(
                status_code=403,
                detail=f"Role '{role}' does not have {action} access to {resource}"
            )

        return await call_next(request)

The middleware's position in the stack is critical. It runs after JWT validation (so the role is already extracted and trusted) and after rate limiting (so rejected requests are not rate-counted). If the RBAC check fails, the 403 response includes the role and resource in the error detail, making debugging straightforward without exposing the full policy configuration.

Authorization errors should be informative enough for operators to diagnose the problem and specific enough that they know which role would grant access. "Forbidden" is not helpful. "Role 'viewer' does not have POST access to /api/v1/tasks" is actionable.

Least Privilege in AI Orchestration

The principle of least privilege takes on specific meaning in AI orchestration. Consider these scenarios that RBAC prevents:

  • A dashboard viewer cannot accidentally trigger a pipeline execution that consumes LLM tokens and produces unreviewed output.
  • An operator cannot modify agent capability declarations, ensuring that the agent fleet's permission boundaries are managed only by administrators who understand the implications.
  • An admin cannot modify the RBAC policies themselves, preventing privilege escalation where an admin grants themselves system_admin access.
  • No role except system_admin can access the data purge endpoint, ensuring that audit trail deletion requires the highest privilege level.

Each of these boundaries exists because the corresponding action has consequences that extend beyond the immediate request. Running a task costs money (LLM tokens). Modifying agent capabilities changes security boundaries. Altering RBAC policies affects every user. Purging data destroys compliance evidence. The RBAC model ensures that each of these actions is gated by the appropriate level of authority.

Testing RBAC

The RBAC test suite contains 42 tests that systematically verify every role-endpoint-method combination. The tests follow a matrix pattern: for each endpoint, verify that authorized roles receive 200/201 responses and unauthorized roles receive 403 responses.

# Example RBAC test pattern
@pytest.mark.parametrize("role,expected_status", [
    ("viewer",       403),
    ("operator",     201),
    ("admin",        201),
    ("system_admin", 201),
])
async def test_create_task_rbac(role, expected_status, client):
    token = create_test_token(role=role)
    response = await client.post(
        "/api/v1/tasks",
        json={"title": "Test", "description": "Test task"},
        headers={"Authorization": f"Bearer {token}"}
    )
    assert response.status_code == expected_status

This parametrized approach ensures comprehensive coverage. If a new endpoint is added without a corresponding RBAC test, the code review process catches it. The test suite serves as a living documentation of the permission matrix — reading the test parameters tells you exactly which roles can access which endpoints.

Policy Evolution

As OCCP grows, the RBAC model will evolve. The current 4-role hierarchy may expand with team-scoped roles (team_admin who can only manage agents within their team) or resource-scoped permissions (operator who can run tasks only with specific agents). Casbin's model is flexible enough to accommodate these extensions without changing the middleware or the enforcement logic — only the model.conf and policy.csv files need updating.

// Key Takeaways

  • 4-role hierarchy: viewer (read), operator (+execute), admin (+manage), system_admin (full access)
  • Casbin separates the access model from enforcement logic, enabling policy changes without code changes
  • Role inheritance means each role only defines its additional permissions; viewer permissions cascade up
  • JWT claims carry the user's role; middleware checks Casbin policy before every endpoint handler
  • 42 parametrized tests verify every role-endpoint-method combination in the permission matrix
  • Least privilege prevents accidental token spend, capability changes, privilege escalation, and data destruction
