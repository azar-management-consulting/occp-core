Docker Compose in Production: Lessons from Deploying AI Services

Docker Compose is often dismissed as a development-only tool. For OCCP, it became the production deployment target for v0.2.0 -- and the lessons we learned about healthchecks, filesystem security, and service networking shaped every subsequent release.

The OCCP Production Stack

The OCCP platform runs as two primary services: the FastAPI backend (occp-api) on port 8000, and the Next.js dashboard (occp-dash) on port 3000. Both sit behind an Apache2 reverse proxy that handles SSL termination and domain routing. Docker Compose orchestrates the two application containers, their networking, and their shared dependencies.

We evaluated Kubernetes early and rejected it for the current stage. With two services and predictable load patterns, the operational overhead of a Kubernetes cluster was not justified. Docker Compose provides declarative service definitions, automated restarts, and network isolation -- exactly what we needed without the abstraction tax.

The Compose File: Structure and Design Decisions

version: "3.9"

services:
  occp-api:
    build:
      context: .
      dockerfile: Dockerfile.api
    container_name: occp-api
    restart: unless-stopped
    read_only: true
    user: "1000:1000"
    ports:
      - "127.0.0.1:8000:8000"
    volumes:
      - api-data:/app/data
      - /tmp
    environment:
      - OCCP_ENV=production
      - OCCP_LOG_LEVEL=info
    healthcheck:
      test: ["CMD", "python", "-c",
        "import urllib.request; urllib.request.urlopen('http://localhost:8000/health')"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 15s
    tmpfs:
      - /tmp:noexec,nosuid,size=100m

  occp-dash:
    build:
      context: .
      dockerfile: Dockerfile.dash
    container_name: occp-dash
    restart: unless-stopped
    read_only: true
    user: "1000:1000"
    ports:
      - "127.0.0.1:3000:3000"
    environment:
      - NEXT_PUBLIC_API_URL=https://api.occp.ai
    healthcheck:
      test: ["CMD", "node", "-e",
        "require('http').get('http://localhost:3000', r => r.statusCode === 200 ? process.exit(0) : process.exit(1))"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 20s
    depends_on:
      occp-api:
        condition: service_healthy

volumes:
  api-data:

Several design decisions are embedded in this configuration. Each one was learned through trial and error during development.

Healthcheck Configuration

Docker healthchecks are the difference between a service that restarts on failure and one that silently serves errors. The healthcheck for occp-api uses Python's stdlib urllib.request because we cannot assume curl or wget exist in the minimal container image. The dashboard healthcheck uses Node.js's built-in http module for the same reason.

The Four Healthcheck Parameters

  • interval: 30s -- How often Docker checks. We started at 10s but found it generated unnecessary log noise without improving recovery time.
  • timeout: 10s -- Maximum time for the check to complete. AI services can have slow startups during model loading, so 10 seconds provides margin.
  • retries: 3 -- Number of consecutive failures before marking unhealthy. This prevents false positives from transient network issues or garbage collection pauses.
  • start_period: 15s -- Grace period after container start where healthcheck failures do not count. Critical for the API service which needs to initialize database connections and load policy rules.
The depends_on condition with service_healthy ensures the dashboard only starts after the API is confirmed responsive. Without this, the dashboard would boot, attempt to verify its API connection, and fail during startup -- creating confusing error logs.

Read-Only Filesystem Security

The read_only: true directive makes the container's root filesystem immutable. This is a defense-in-depth measure: even if an attacker gains code execution inside the container, they cannot modify application binaries, install persistence mechanisms, or tamper with configuration files.

Read-only mode requires careful handling of directories that applications legitimately need to write to. The API service needs /app/data for the SQLite database and audit logs, and /tmp for temporary file operations. We provide these as explicit exceptions:

volumes:
  - api-data:/app/data     # Named volume for persistent data
tmpfs:
  - /tmp:noexec,nosuid,size=100m  # RAM-backed temp, no executables

The tmpfs mount for /tmp is particularly important. The noexec flag prevents execution of any files written to the temp directory, and nosuid prevents privilege escalation through setuid binaries. The 100MB size limit prevents a runaway process from consuming all available memory through temp file accumulation.

Non-Root Container Users

Both services run as user: "1000:1000" -- a non-root user created in the Dockerfile. Running containers as root is the single most common container security mistake. A root process inside a container is one kernel exploit away from root on the host.

# In Dockerfile.api
RUN groupadd -g 1000 occp && \
    useradd -u 1000 -g occp -s /bin/false occp && \
    chown -R occp:occp /app
USER occp

The user is created with /bin/false as the shell, preventing interactive login even if somehow accessed. File ownership is set during build, so the non-root user has exactly the permissions it needs and nothing more.

Network Isolation

The port bindings use 127.0.0.1:8000:8000 rather than 8000:8000. This subtle difference is critical: binding to localhost means the service is only accessible from the host machine, not from external networks. The Apache2 reverse proxy on the host forwards external traffic to these local ports after SSL termination.

# WRONG: Exposes port to all network interfaces
ports:
  - "8000:8000"

# CORRECT: Binds only to localhost
ports:
  - "127.0.0.1:8000:8000"

Without the localhost binding, the API would be accessible on the server's public IP at port 8000, bypassing the reverse proxy and its SSL encryption. This is the kind of misconfiguration that causes data exposure incidents.

Environment Variable Management

Secrets never appear in the Compose file. The environment section contains only non-sensitive configuration. Actual secrets -- JWT signing keys, API tokens, database credentials -- are injected at runtime through a .env file that is explicitly excluded from version control:

# .env (NOT in git, deployed via secure copy)
OCCP_JWT_SECRET=...
ANTHROPIC_API_KEY=...
OCCP_DB_URL=sqlite+aiosqlite:///app/data/occp.db

The Compose file references these with env_file: .env in production deployments. During development, the same variables are set in a .env.example file with placeholder values, ensuring developers know which variables are required without exposing actual credentials.

Volume Management for Persistent Data

The api-data named volume persists across container restarts and rebuilds. This is where the SQLite database and audit log chain live. Named volumes are managed by Docker and survive docker compose down -- only docker compose down -v removes them.

We implement a backup strategy that snapshots the volume directory before every deployment. The backup script runs as a pre-deploy hook in our CI/CD pipeline:

#!/bin/bash
BACKUP_DIR="/backups/occp/$(date +%Y%m%d_%H%M%S)"
mkdir -p "$BACKUP_DIR"
docker compose stop occp-api
cp -a /var/lib/docker/volumes/occp_api-data/_data/* "$BACKUP_DIR/"
docker compose start occp-api

Lessons Learned

Start Period Matters More Than You Think

Our first deployment had no start_period. The API container took 12 seconds to initialize database connections and load policy rules. Docker marked it unhealthy after 3 failed checks (at 10s intervals), restarted it, and entered a restart loop. Adding a 15-second start period fixed the issue immediately.

Log Driver Configuration

Docker's default json-file log driver has no size limit. In production, AI services generate substantial log output -- especially during pipeline execution with verbose audit logging. We configure log rotation in the Compose file to prevent disk exhaustion:

logging:
  driver: json-file
  options:
    max-size: "50m"
    max-file: "5"

Compose Watch for Development

Docker Compose v2.22+ introduced compose watch, which provides file-syncing and rebuild triggers. During development, code changes sync into the running container without a full rebuild. This reduced our development iteration cycle from 45 seconds (full rebuild) to under 2 seconds (file sync + server restart).

// Key Takeaways

  • Read-only filesystem with explicit writable exceptions prevents container modification attacks
  • Non-root users with /bin/false shells enforce least-privilege at the container level
  • Localhost port binding prevents bypassing the reverse proxy and SSL termination
  • Healthcheck start_period prevents restart loops during slow application initialization
  • Named volumes survive container rebuilds; backup before every deployment
  • Log rotation configuration prevents disk exhaustion from verbose AI service output
