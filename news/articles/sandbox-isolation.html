Sandbox Isolation: nsjail, Bubblewrap, and Defense in Depth

AI-generated code is untrusted code. It may be correct, it may be buggy, or it may be actively malicious — the result of a prompt injection that turned the code generation agent against the platform. The sandbox ensures that regardless of what the code does, it cannot escape its execution boundary.

The Case for Sandboxing AI Code

When an AI agent in OCCP generates Python code as part of task execution, that code needs to run somewhere. The simplest approach is to execute it in the same Python process that runs the platform: call exec() on the generated string and capture the output. This is also the most dangerous approach. The generated code has access to everything the platform process has access to: environment variables (which contain API keys), the filesystem (which contains configuration files), the network (which connects to databases and external services), and the process itself (which it can crash or compromise).

The threat model is not hypothetical. Prompt injection attacks can cause code generation agents to produce code that reads /etc/shadow, exfiltrates environment variables through HTTP requests, or spawns a reverse shell. Even without adversarial intent, a buggy code generation might create an infinite loop that consumes all CPU, or a recursive function that exhausts memory, or a disk operation that fills the filesystem.

The question is not whether AI-generated code will eventually do something dangerous. The question is whether the platform contains the damage when it happens. Sandbox isolation turns a potential security incident into a contained execution failure.

Three-Tier Architecture

OCCP implements three sandbox tiers with decreasing isolation strength. The platform automatically selects the strongest tier available on the host system, with automatic fallback when a tier is not available.

Tier 1: nsjail

nsjail is a process isolation tool developed by Google's security team. It uses Linux kernel namespaces to create a completely isolated execution environment: separate PID namespace (the sandboxed process cannot see other processes), separate network namespace (no network access by default), separate mount namespace (the sandboxed process sees only explicitly mounted paths), and separate user namespace (the process runs as an unprivileged user regardless of the host user).

On top of namespace isolation, nsjail applies seccomp-bpf filters that restrict which system calls the sandboxed process can make. OCCP's seccomp profile allows a minimal set of system calls needed to run Python: read, write, open (restricted to the sandbox directory), close, mmap, brk, and a few others. Dangerous system calls like execve (running other programs), socket (network access), and ptrace (process debugging) are blocked.

# nsjail configuration for OCCP code execution
class NsjailSandbox:
    def __init__(self):
        self.config = {
            "time_limit": 30,          # Max 30 seconds
            "rlimit_as": 256,          # Max 256MB memory
            "rlimit_cpu": 10,          # Max 10 seconds CPU time
            "rlimit_fsize": 10,        # Max 10MB file creation
            "rlimit_nofile": 32,       # Max 32 open file descriptors
            "mount_proc": False,       # No /proc access
            "disable_clone_newnet": False,  # Network namespace isolation
            "cwd": "/sandbox",
            "keep_env": False,         # No environment variable inheritance
        }

    async def execute(self, code: str, timeout: int = 30) -> ExecutionResult:
        # Write code to temporary file in sandbox mount
        code_path = self._write_code_file(code)

        # Execute in nsjail
        proc = await asyncio.create_subprocess_exec(
            "nsjail", *self._build_args(code_path),
            stdout=asyncio.subprocess.PIPE,
            stderr=asyncio.subprocess.PIPE,
        )
        stdout, stderr = await asyncio.wait_for(
            proc.communicate(), timeout=timeout
        )
        return ExecutionResult(
            stdout=stdout.decode(),
            stderr=stderr.decode(),
            exit_code=proc.returncode,
            tier="nsjail"
        )

Key configuration details: keep_env: False ensures the sandboxed process inherits zero environment variables from the host. This prevents exfiltration of API keys, database credentials, or any other secrets stored in environment variables. The mount_proc: False setting blocks access to /proc, which on Linux exposes process information, memory maps, and other sensitive system data.

Tier 2: bubblewrap (bwrap)

bubblewrap is a lightweight sandbox used by Flatpak and other Linux desktop application frameworks. It provides user namespace isolation and filesystem restrictions but does not include seccomp filtering. This makes it weaker than nsjail but significantly stronger than unsandboxed execution.

# bubblewrap sandbox configuration
class BubblewrapSandbox:
    def _build_args(self, code_path: str) -> list[str]:
        return [
            "bwrap",
            "--ro-bind", "/usr", "/usr",           # Read-only system libraries
            "--ro-bind", "/lib", "/lib",
            "--ro-bind", "/lib64", "/lib64",
            "--bind", self.sandbox_dir, "/sandbox", # Read-write sandbox dir
            "--dev", "/dev",                         # Minimal /dev
            "--unshare-all",                         # Unshare all namespaces
            "--die-with-parent",                     # Kill sandbox if parent dies
            "--new-session",                         # New session ID
            "--chdir", "/sandbox",
            "python3", code_path,
        ]

The --unshare-all flag creates new namespaces for PID, network, IPC, UTS, and user. The --ro-bind mounts make system libraries available in read-only mode so Python can run, while the sandbox directory is the only writable location. The --die-with-parent flag ensures the sandbox is terminated if the OCCP process crashes, preventing orphaned sandbox processes.

Tier 3: Restricted Subprocess

When neither nsjail nor bubblewrap is available (macOS development environments, Windows, minimal Linux containers), the sandbox falls back to a standard subprocess with resource limits applied through Python's resource module.

import resource
import subprocess

class SubprocessSandbox:
    def _apply_limits(self):
        """Called in subprocess via preexec_fn."""
        # CPU time limit: 10 seconds
        resource.setrlimit(resource.RLIMIT_CPU, (10, 10))
        # Memory limit: 256 MB
        resource.setrlimit(resource.RLIMIT_AS, (256 * 1024 * 1024, 256 * 1024 * 1024))
        # File size limit: 10 MB
        resource.setrlimit(resource.RLIMIT_FSIZE, (10 * 1024 * 1024, 10 * 1024 * 1024))
        # Max file descriptors: 32
        resource.setrlimit(resource.RLIMIT_NOFILE, (32, 32))

    async def execute(self, code: str, timeout: int = 30) -> ExecutionResult:
        proc = await asyncio.create_subprocess_exec(
            "python3", self._write_code_file(code),
            stdout=asyncio.subprocess.PIPE,
            stderr=asyncio.subprocess.PIPE,
            preexec_fn=self._apply_limits,
        )
        stdout, stderr = await asyncio.wait_for(
            proc.communicate(), timeout=timeout
        )
        return ExecutionResult(
            stdout=stdout.decode(), stderr=stderr.decode(),
            exit_code=proc.returncode, tier="subprocess"
        )

Tier 3 provides resource limits (CPU, memory, file size) but no namespace isolation. The subprocess can access the network, see other processes, and read any file the OCCP process can read. This tier is suitable only for development environments and is explicitly flagged in the dashboard as "limited isolation."

Automatic Fallback

The sandbox manager probes the system at startup to determine which tiers are available:

class SandboxManager:
    def __init__(self):
        self.active_tier = self._detect_tier()

    def _detect_tier(self) -> str:
        if shutil.which("nsjail"):
            if self._test_nsjail():
                return "nsjail"
        if shutil.which("bwrap"):
            if self._test_bubblewrap():
                return "bubblewrap"
        return "subprocess"

    def _test_nsjail(self) -> bool:
        """Run a trivial nsjail command to verify it works."""
        try:
            result = subprocess.run(
                ["nsjail", "--help"],
                capture_output=True, timeout=5
            )
            return result.returncode == 0
        except (FileNotFoundError, subprocess.TimeoutExpired):
            return False

The detection is not just a binary "is the binary present" check. The manager runs a trivial test command to verify the tool actually works. This catches cases where nsjail is installed but the kernel does not support the required namespace features (common in some container environments where user namespaces are disabled).

Resource Limits

All three tiers enforce the same resource limits, using different mechanisms:

  • CPU time — Maximum 10 seconds of CPU time. Prevents infinite loops and computationally expensive operations. Applied via seccomp (Tier 1), rlimit (Tier 2 and 3).
  • Wall clock time — Maximum 30 seconds total execution time. Applied via asyncio.wait_for timeout at the Python level.
  • Memory — Maximum 256 MB virtual memory. Prevents memory-intensive attacks and accidental memory leaks. Applied via nsjail config, rlimit, or subprocess rlimit.
  • Network — Blocked in Tier 1 (namespace isolation) and Tier 2 (namespace unsharing). Not blocked in Tier 3.
  • Filesystem — Write access limited to the sandbox directory in Tier 1 and 2. Maximum 10 MB file creation across all tiers. No access to host system files in Tier 1 and 2.
  • Process creation — Blocked in Tier 1 (seccomp). Limited in Tier 2 (PID namespace). Unrestricted in Tier 3.
The sandbox assumes the worst case for every execution. If the code tries to allocate 1GB of memory, the limit terminates it. If it tries to run for 60 seconds, the timeout kills it. If it tries to open a network socket, the namespace blocks it. Every resource request is bounded, and every bound is enforced by the kernel, not by Python.

Execution Result Handling

The sandbox returns a structured ExecutionResult that includes stdout, stderr, the exit code, and the tier that was used. The tier information is critical for security auditing — if an execution ran in Tier 3 (limited isolation), the audit log reflects this, and operators can investigate whether the execution posed a risk that stronger isolation would have prevented.

Execution results are passed through the same output sanitization guard that processes LLM responses. Even code execution output is scanned for credential patterns before it reaches the user or the audit log. This layered approach means that even if sandboxed code manages to read a credential (possible in Tier 3), the output guard prevents it from being delivered.

Defense in Depth

The sandbox is one layer in a multi-layer defense strategy. Before code reaches the sandbox, the policy engine evaluates whether code execution is permitted for the current task and agent. The injection detection system scans the generated code for known malicious patterns. The agent's capability declaration must include execute permissions. Only after passing all these checks does the code enter the sandbox.

After the sandbox, the output is sanitized, validated against the task's expected output format, and recorded in the audit log with the execution tier and resource consumption metrics. This pre-and-post processing ensures that the sandbox is not the only defense — it is the last defense, backed by multiple earlier layers that reduce the likelihood of malicious code reaching execution in the first place.

// Key Takeaways

  • Three tiers: nsjail (kernel namespaces + seccomp), bubblewrap (user namespaces), subprocess (resource limits only)
  • Automatic detection and fallback at startup with verification that the sandbox tool actually works
  • Resource limits enforced across all tiers: 10s CPU, 30s wall clock, 256MB memory, 10MB disk, no network (Tier 1/2)
  • Environment variables are never inherited by the sandbox (Tier 1/2), preventing credential exfiltration
  • Execution tier is recorded in audit logs for security review of isolation level
  • The sandbox is defense-in-depth: policy, injection detection, and capability checks filter code before it reaches execution
