Skills System Architecture: Modular Capabilities for AI Agents

Raw tool access gives agents power without structure. Skills give agents power with purpose. OCCP's skills system wraps tools, permissions, prompts, and validation rules into reusable, composable units that transform generic AI agents into domain-specific specialists — all within the governance framework.

The Problem with Raw Tool Access

Consider an agent with access to the GitHub MCP server. It can list repositories, read files, create branches, open pull requests, merge code, delete branches, modify repository settings, and manage team permissions. These are all valid operations. But should a code review agent have the ability to merge code? Should a documentation agent be able to modify repository settings? Raw tool access makes no distinction between the tools an agent needs for its purpose and the tools that happen to be available on the same server.

The permissions model from RBAC (Casbin) controls which roles can access which endpoints. But RBAC operates at the API level, not the task level. A user with the "operator" role can create tasks and execute them, but RBAC cannot express "this agent should only use GitHub for reading code, not writing it." That constraint operates at a finer granularity than roles and endpoints — it operates at the tool-and-parameter level. This is the gap that skills fill.

Skill Definition

A skill is a JSON document that declares five properties: the tools the skill requires, the permissions it grants, the validation rules it enforces, the prompt template it provides to the agent, and the output schema it expects from the result.

{
  "id": "skill-code-review-v1",
  "name": "code-review",
  "version": "1.0.0",
  "description": "Review pull requests for code quality, security issues, and best practices",
  "required_tools": [
    "github.get_pull_request",
    "github.get_pull_request_diff",
    "github.create_review",
    "github.list_pull_request_files"
  ],
  "forbidden_tools": [
    "github.merge_pull_request",
    "github.delete_branch",
    "github.update_repository"
  ],
  "permissions": [
    "read:repository",
    "write:review"
  ],
  "validation_rules": {
    "output_must_contain": ["severity", "recommendation"],
    "output_must_not_contain_patterns": ["password", "token", "secret"],
    "max_output_length": 10000
  },
  "prompt_template": "You are a code reviewer. Analyze the following diff...",
  "output_schema": {
    "type": "object",
    "properties": {
      "summary": { "type": "string" },
      "issues": { "type": "array" },
      "severity": { "type": "string", "enum": ["low", "medium", "high", "critical"] },
      "recommendation": { "type": "string", "enum": ["approve", "request-changes", "comment"] }
    },
    "required": ["summary", "severity", "recommendation"]
  }
}

The forbidden_tools field is as important as required_tools. It creates an explicit deny list that prevents the skill from accessing tools that are available on the same MCP server but outside the skill's intended scope. Even if the agent's underlying model attempts to call github.merge_pull_request during a code review, the skills system blocks the call before it reaches the MCP server.

Skill Acquisition

Agents do not start with skills. They acquire them through a request-and-validation flow that ensures the agent has the prerequisites to use the skill effectively.

# occp/skills/acquisition.py — skill acquisition flow
class SkillAcquisition:
    async def request_skill(self, agent_id: int, skill_id: str) -> AcquisitionResult:
        agent = await self.agent_repo.get(agent_id)
        skill = await self.skill_repo.get(skill_id)

        # 1. Check tool availability
        for tool in skill.required_tools:
            if not self.tool_registry.is_available(tool):
                return AcquisitionResult(
                    granted=False,
                    reason=f"Required tool '{tool}' is not connected"
                )

        # 2. Check agent's registered capabilities
        for permission in skill.permissions:
            if not agent.has_capability(permission):
                return AcquisitionResult(
                    granted=False,
                    reason=f"Agent lacks capability '{permission}'"
                )

        # 3. Check concurrent skill limit
        active_skills = await self.get_active_skills(agent_id)
        if len(active_skills) >= agent.max_concurrent_skills:
            return AcquisitionResult(
                granted=False,
                reason=f"Agent at max concurrent skills ({agent.max_concurrent_skills})"
            )

        # 4. Grant skill
        await self.skill_repo.assign(agent_id, skill_id)
        await self.audit.log("skill_acquired", agent_id=agent_id, skill_id=skill_id)

        return AcquisitionResult(granted=True)

The acquisition flow validates three conditions. First, every tool required by the skill must be available in OCCP's tool registry — which means the corresponding MCP server must be connected and healthy. Second, the agent must have the capability permissions declared by the skill. An agent registered with read:repository capability can acquire a code-review skill, but an agent without that capability cannot. Third, the agent must not exceed its concurrent skill limit, which prevents a single agent from accumulating excessive tool access.

Skill Revocation

Skills can be revoked by platform administrators or by automated policy enforcement. If an MCP server goes offline, all skills that depend on that server's tools are automatically suspended until the server reconnects. If an agent's capabilities are modified (demoted from operator to viewer, for example), skills that require the revoked capability are automatically removed from the agent's profile. Revocation is logged in the audit trail with the reason and the revoking authority.

Skill Execution

When an agent executes a task that requires a skill, the skills system provides three things to the execution context: the prompt template (which structures how the agent approaches the task), the tool configuration (which tools are available and which are forbidden), and the validation rules (which the output must satisfy before the task completes).

# Skill execution context injected into the pipeline
class SkillExecutionContext:
    skill: Skill
    allowed_tools: list[str]     # required_tools only
    forbidden_tools: list[str]   # explicitly denied
    prompt_template: str         # structured prompt for the agent
    output_schema: dict          # expected output format
    validation_rules: dict       # output validation constraints

    def validate_output(self, output: dict) -> ValidationResult:
        """Validate agent output against skill's rules."""
        # Schema validation
        try:
            jsonschema.validate(output, self.output_schema)
        except ValidationError as e:
            return ValidationResult(valid=False, reason=str(e))

        # Content validation
        for required_field in self.validation_rules.get("output_must_contain", []):
            if required_field not in str(output):
                return ValidationResult(
                    valid=False,
                    reason=f"Output missing required field: {required_field}"
                )

        return ValidationResult(valid=True)

The execution context is injected into the pipeline's Execute stage. The agent receives the prompt template with the task-specific data interpolated, calls only the tools listed in allowed_tools, and produces output that conforms to the output_schema. Any attempt to call a tool not in the allowed list — or explicitly in the forbidden list — is rejected by the MCP executor before the call leaves OCCP.

Composability: Building Complex Agents from Simple Skills

Skills are designed to be composable. A "deployment automation" workflow might require three skills: code-review (analyze the PR), test-runner (execute the test suite), and deployment (push to staging). Each skill provides its own tools, permissions, and validation rules. The agent acquires all three skills and the pipeline orchestrates their execution in sequence, with each skill's output feeding into the next skill's input.

Skills are to agents what functions are to programs. A function encapsulates logic, declares its inputs and outputs, and can be composed with other functions. A skill encapsulates capability, declares its tools and permissions, and can be composed with other skills. Both create reusable units of verified behavior.

This composability model has a clear benefit for governance. Instead of auditing "what can this agent do?" by examining every tool on every MCP server it can access, an administrator audits the agent's skill set. Each skill has an explicit scope, explicit permissions, and explicit validation rules. The audit surface is the skill definition, not the entire MCP server catalog.

The Skills Marketplace Vision

The current skill definitions are bundled with the OCCP distribution. The long-term vision is a skills marketplace where organizations can publish and share skill definitions. A security consulting firm could publish a vulnerability-assessment skill. A DevOps team could publish a kubernetes-deployment skill. A data science team could publish a dataset-analysis skill. Each published skill would include its tool requirements, permission scope, and validation rules, allowing any OCCP deployment to evaluate and adopt skills that match their needs.

The marketplace model introduces its own governance challenges: skill verification, version compatibility, and trust delegation. These are engineering problems for future releases. The current skills system establishes the architectural foundation — the skill definition format, the acquisition flow, the execution context, and the validation pipeline — that the marketplace will build on.

Skills and the Policy Engine

Skills interact with the policy engine at two levels. At acquisition time, the policy engine evaluates whether the skill's permission requirements comply with organizational policies. An organization that prohibits agents from having write access to production databases would have a policy rule that blocks acquisition of any skill with write:database permission. At execution time, every tool call made through a skill passes through the standard policy gate, ensuring that skill-level permissions do not override organization-level restrictions.

This two-level enforcement means that skills cannot escalate privileges. A skill can only grant access to tools that the agent already has capability for, and every tool call is still subject to the same policy evaluation that governs direct tool access. Skills are a convenience layer for organizing tool access, not a bypass mechanism for the governance framework.

// Key Takeaways

  • Skills bundle tools, permissions, prompts, and validation rules into reusable, governed capability units
  • The forbidden_tools field explicitly denies access to tools outside the skill's scope, even if available on the same MCP server
  • Skill acquisition validates tool availability, agent capabilities, and concurrent skill limits before granting
  • Skills are composable: complex workflows combine multiple skills, each with independent validation
  • Two-level policy enforcement: organizational policies at acquisition time, tool-level policies at execution time
  • Skills cannot escalate privileges — they organize existing capabilities, they do not create new ones
