Audit Retention Engineering: Automated Compliance at Startup

Log retention is one of the most overlooked aspects of AI compliance. The EU AI Act requires that operational logs be maintained for an appropriate period, but most systems treat retention as a background cron job that nobody monitors. OCCP takes a different approach: retention compliance is verified at application startup, before the platform serves a single request.

The Retention Problem

Audit log retention seems simple: keep logs for N days, delete (or archive) anything older. In practice, retention is fraught with engineering challenges. Logs grow continuously, creating storage pressure. Archive operations can fail silently, leaving gaps in the retention chain. Hash chain integrity must be preserved across active and archived segments. And the retention policy itself must be configurable per deployment, because different jurisdictions and risk levels require different retention periods.

The most common failure mode is not data loss — it is undetected non-compliance. A cron job that archives old logs might fail for weeks without anyone noticing. The logs appear to be retained (the retention job is "running"), but the archive destination is full, the network mount is disconnected, or a permission change broke the write path. The organization discovers the gap during an audit, which is the worst possible time to discover it.

The difference between "we have a retention policy" and "we are compliant with our retention policy" is the difference between intent and evidence. OCCP's retention system produces evidence at every startup.

Startup-Time Enforcement

The Startup Sequence

When OCCP starts, the application initialization sequence includes a mandatory retention verification step. This step runs after database migration checks and before the HTTP server begins accepting requests. The sequence is: verify database connectivity, run pending migrations, execute retention verification, start the HTTP server. If retention verification fails, the server does not start.

# Application startup sequence
async def startup():
    """OCCP application startup with retention enforcement."""
    # Step 1: Database
    await verify_database_connection()
    await run_pending_migrations()

    # Step 2: Retention enforcement (blocking)
    retention = await enforce_audit_retention()
    if not retention.compliant:
        logger.critical(
            "Retention verification failed: %s",
            retention.failure_reason
        )
        raise StartupError(
            "Cannot start: audit retention non-compliant. "
            f"Reason: {retention.failure_reason}"
        )

    logger.info(
        "Retention verified: %d active entries, "
        "%d archived, chain integrity: %s",
        retention.active_count,
        retention.archived_count,
        retention.chain_valid
    )

    # Step 3: Start serving (only if retention passes)
    await start_http_server()

Why at Startup

Startup-time enforcement guarantees that the platform is never in a non-compliant state while serving requests. A cron-based approach has a window between the last successful run and the current time where compliance is assumed but not verified. A startup-based approach eliminates this window: every time the application starts (after a deploy, a restart, a crash recovery), retention compliance is re-verified from scratch.

This is particularly important for containerized deployments where containers are frequently recreated. Each new container instance verifies retention before joining the load balancer. There is no accumulated state or stale assumptions from previous runs.

Configurable Retention Periods

Default and Override

OCCP's default retention period is 365 days, aligned with the most common interpretation of EU AI Act Article 19 requirements. The retention period is configurable through environment variables and the configuration file, allowing organizations to set longer periods for regulated industries or shorter periods where permitted.

# Configuration options
OCCP_AUDIT_RETENTION_DAYS=365       # Default: 1 year
OCCP_AUDIT_ARCHIVE_BACKEND=s3       # s3, filesystem, database
OCCP_AUDIT_ARCHIVE_PATH=/archive    # Backend-specific path
OCCP_AUDIT_VERIFY_CHAIN=true        # Verify hash chain integrity
OCCP_AUDIT_STARTUP_BLOCK=true       # Block startup on failure

Per-Category Retention

Different audit entry categories may have different retention requirements. Security-related entries (authentication failures, policy denials, boundary violations) may require longer retention than operational entries (task completion, performance metrics). OCCP supports per-category retention periods, with a global minimum that cannot be reduced below the platform-wide default.

# Per-category retention configuration
retention:
  default_days: 365
  categories:
    security_events: 730      # 2 years for security logs
    policy_denials: 730       # 2 years for compliance evidence
    task_completions: 365     # 1 year for operational logs
    performance_metrics: 180  # 6 months for metrics
  minimum_days: 180           # Global floor

The Archival Process

Archive, Never Delete

OCCP's retention system moves expired entries to archive storage; it never deletes them. This design decision reflects a regulatory reality: an audit investigation may request logs from any period, and "we deleted them per our retention policy" is a weaker defense than "here are the archived logs." Archive storage is typically cheaper than primary storage, so the cost of indefinite archival is manageable.

Archive Backends

OCCP supports three archive backends. The filesystem backend writes archived entries as JSON files to a configured directory, suitable for development and small deployments. The S3-compatible backend writes to any S3-compatible object store (AWS S3, MinIO, Cloudflare R2), suitable for production deployments with durability requirements. The database backend writes to a separate database table or schema, suitable for organizations that prefer to keep all data in a single database system.

# Archive entry format (filesystem backend)
# /archive/2025/10/aud_20251015_001.json
{
    "entry_id": "aud_20251015_001",
    "original_hash": "sha256:9f86d0...",
    "prev_hash": "sha256:7c211a...",
    "archived_at": "2026-02-25T00:00:00Z",
    "retention_category": "task_completions",
    "original_timestamp": "2025-10-15T14:30:00Z",
    "data": { ... }  # Complete original entry
}

Hash Chain Continuity

When entries are moved to the archive, the hash chain must remain verifiable. The archived entries retain their original hashes and chain links. A "chain bridge" entry is written at the transition point, recording the hash of the last archived entry and the hash of the first active entry. This bridge allows the complete chain to be reconstructed from archive storage for audit investigations, while the active chain remains independently verifiable for routine operations.

Retention Verification in Health Checks

Continuous Monitoring

Startup-time verification ensures compliance at boot. But what about retention compliance during long-running operation? OCCP's health check endpoint includes a retention status component that reports the current state of retention compliance without performing a full re-verification (which would be too expensive for a health check).

# Health check response including retention status
{
    "status": "healthy",
    "components": {
        "database": "connected",
        "retention": {
            "status": "compliant",
            "last_verified": "2026-02-25T00:00:00Z",
            "active_entries": 45230,
            "oldest_active": "2025-02-26T08:15:00Z",
            "retention_days": 365,
            "chain_integrity": "verified",
            "next_archive_due": "2026-02-26T00:00:00Z"
        }
    }
}

Degradation Alerts

If the retention status degrades (archive backend unreachable, chain integrity check failed, retention period exceeded without archival), the health check reports a degraded state. Monitoring systems (Prometheus, Grafana, or any system polling the health endpoint) can alert on this degradation. The platform continues serving requests during retention degradation — stopping service because archival is delayed would be disproportionate — but the degradation is visible and actionable.

Operational Considerations

Storage Growth

Audit logs grow proportionally to the platform's activity. A moderately active deployment generating 1,000 tasks per day, with an average of 5 audit entries per task (one per pipeline stage), produces approximately 5,000 entries per day. At an average of 2 KB per entry, this is 10 MB per day or 3.6 GB per year. For most deployments, storage growth is not a concern. For high-volume deployments, the S3 archive backend provides cost-effective long-term storage.

Recovery from Archive Failures

If the archive backend is unavailable during a scheduled archival run, OCCP retains the expired entries in the active store and retries at the next startup or scheduled interval. Entries are never discarded because the archive is unreachable. This fail-safe behavior means the active store may temporarily hold more entries than expected, but compliance is maintained: the entries exist, they are just in the wrong storage tier.

Performance Impact

Startup-time retention verification adds latency to the application boot process. For a deployment with 50,000 active audit entries, the verification (count entries, verify chain integrity, identify entries for archival) takes approximately 2-5 seconds on a standard database. For most deployments, this is acceptable. For deployments where sub-second startup is critical (serverless, aggressive autoscaling), the chain integrity verification can be configured to run asynchronously after the server starts, with the startup only checking entry counts and retention period compliance.

// Key Takeaways

  • Retention verification runs at application startup, before serving requests, eliminating undetected non-compliance windows
  • Default retention period is 365 days (EU AI Act Article 19), configurable per deployment and per audit category
  • Expired entries are archived, never deleted, preserving the complete audit history for regulatory investigations
  • Hash chain continuity is maintained across active and archived segments through bridge entries
  • Health check endpoints report retention compliance status for continuous monitoring
  • Archive failures trigger retry, not data loss: entries remain in active storage until successfully archived
