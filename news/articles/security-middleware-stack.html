Security Middleware Stack: HSTS, Rate Limiting, and Request Logging

Security in a web application is not a single feature — it is a composition of independent layers, each addressing a specific threat vector. OCCP's FastAPI middleware stack implements HSTS enforcement, sliding-window rate limiting, structured request logging with correlation IDs, and strict CORS policies. Together, these layers create defense-in-depth for an AI orchestration API.

Middleware Composition in FastAPI

FastAPI inherits Starlette's middleware architecture: each middleware wraps the next in a chain, receiving the request before the route handler and the response after it. The order of middleware registration determines the execution order. In OCCP, the stack is registered from outermost to innermost: CORS first, then security headers, then rate limiting, then request logging, and finally the route handler itself.

# occp/middleware/__init__.py — middleware registration order
from fastapi import FastAPI
from .cors import configure_cors
from .security_headers import SecurityHeadersMiddleware
from .rate_limiter import RateLimitMiddleware
from .request_logger import RequestLoggerMiddleware

def register_middleware(app: FastAPI):
    """Register middleware in correct order (outermost first)."""
    configure_cors(app)
    app.add_middleware(SecurityHeadersMiddleware)
    app.add_middleware(RateLimitMiddleware, max_requests=100, window_seconds=60)
    app.add_middleware(RequestLoggerMiddleware)

This ordering is deliberate. CORS headers must be set before any other processing to ensure preflight requests receive correct headers. Security headers apply to every response regardless of whether the request is rate-limited or rejected. Rate limiting runs before request logging to prevent denial-of-service attacks from flooding the log pipeline.

HSTS Enforcement

HTTP Strict Transport Security (HSTS) instructs browsers to only communicate with the server over HTTPS. Without HSTS, a user who types http://api.occp.ai into their browser sends the first request in plaintext, vulnerable to man-in-the-middle interception. The HSTS header eliminates this window by telling the browser to upgrade all future requests to HTTPS automatically.

class SecurityHeadersMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request: Request, call_next):
        response = await call_next(request)

        # HSTS: enforce HTTPS for 1 year, include subdomains
        response.headers["Strict-Transport-Security"] = (
            "max-age=31536000; includeSubDomains; preload"
        )
        # Prevent MIME type sniffing
        response.headers["X-Content-Type-Options"] = "nosniff"
        # Prevent clickjacking
        response.headers["X-Frame-Options"] = "DENY"
        # Restrict referrer information
        response.headers["Referrer-Policy"] = "strict-origin-when-cross-origin"
        # Content Security Policy
        response.headers["Content-Security-Policy"] = (
            "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'"
        )
        # Disable browser features we don't use
        response.headers["Permissions-Policy"] = (
            "camera=(), microphone=(), geolocation=(), payment=()"
        )

        return response

The includeSubDomains directive is critical for OCCP because the platform operates across three subdomains: api.occp.ai for the FastAPI backend, dash.occp.ai for the Next.js dashboard, and occp.ai for the landing page. Without subdomain inclusion, an attacker could intercept dashboard traffic even if the API enforces HTTPS.

Additional Security Headers

Beyond HSTS, the middleware applies six additional headers to every response. X-Content-Type-Options: nosniff prevents browsers from interpreting responses as a different MIME type than declared. X-Frame-Options: DENY blocks the API from being embedded in iframes, preventing clickjacking attacks. Referrer-Policy limits how much URL information is leaked in the Referer header when navigating away from the application. Permissions-Policy explicitly disables browser APIs that the platform does not use, reducing the attack surface available to injected scripts.

Rate Limiting: Sliding Window per IP

Rate limiting protects the API from both malicious abuse and accidental overuse. OCCP implements a sliding window algorithm that tracks request counts per client IP address. Unlike fixed windows that reset on minute boundaries, sliding windows provide smoother rate enforcement without the thundering herd problem at window boundaries.

class RateLimitMiddleware(BaseHTTPMiddleware):
    def __init__(self, app, max_requests: int = 100, window_seconds: int = 60):
        super().__init__(app)
        self.max_requests = max_requests
        self.window_seconds = window_seconds
        self.requests: dict[str, list[float]] = {}

    async def dispatch(self, request: Request, call_next):
        client_ip = request.client.host
        now = time.time()
        window_start = now - self.window_seconds

        # Clean expired entries and count current window
        self.requests.setdefault(client_ip, [])
        self.requests[client_ip] = [
            t for t in self.requests[client_ip] if t > window_start
        ]

        if len(self.requests[client_ip]) >= self.max_requests:
            return JSONResponse(
                status_code=429,
                content={"detail": "Rate limit exceeded"},
                headers={"Retry-After": str(self.window_seconds)},
            )

        self.requests[client_ip].append(now)
        response = await call_next(request)
        response.headers["X-RateLimit-Limit"] = str(self.max_requests)
        response.headers["X-RateLimit-Remaining"] = str(
            self.max_requests - len(self.requests[client_ip])
        )
        return response

The implementation stores timestamps in an in-memory dictionary. For a single-instance deployment, this is efficient and requires no external dependencies. For multi-instance production deployments behind a load balancer, the rate limiter would be backed by Redis — a planned enhancement for the multi-tenancy release. The rate limit headers (X-RateLimit-Limit, X-RateLimit-Remaining) are included in every response so clients can implement backoff logic proactively.

Structured Request Logging

Every API request generates a structured JSON log entry with a correlation ID. The correlation ID is a UUID that follows the request through every layer of the system — from middleware through the policy engine to the database and back. When debugging a failed pipeline execution, the correlation ID links the initial HTTP request to the specific policy evaluation, LLM call, and audit entry that participated in that execution.

class RequestLoggerMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request: Request, call_next):
        correlation_id = request.headers.get(
            "X-Correlation-ID", str(uuid.uuid4())
        )
        start_time = time.perf_counter()

        # Inject correlation ID into request state
        request.state.correlation_id = correlation_id

        response = await call_next(request)

        duration_ms = (time.perf_counter() - start_time) * 1000
        log_entry = {
            "timestamp": datetime.utcnow().isoformat(),
            "correlation_id": correlation_id,
            "method": request.method,
            "path": request.url.path,
            "status_code": response.status_code,
            "duration_ms": round(duration_ms, 2),
            "client_ip": request.client.host,
            "user_agent": request.headers.get("user-agent", ""),
        }
        logger.info(json.dumps(log_entry))

        response.headers["X-Correlation-ID"] = correlation_id
        return response

The log format is JSON for machine parseability. Each entry includes the HTTP method, path, status code, response time in milliseconds, client IP, and user agent. Sensitive data — request bodies, authentication tokens, query parameters — is explicitly excluded from the log. The correlation ID is returned in the response header so the client can reference it in support requests or debugging sessions.

CORS Configuration for Dashboard Communication

The OCCP dashboard at dash.occp.ai communicates with the API at api.occp.ai via cross-origin requests. CORS must be configured to allow this specific origin while rejecting all others. The configuration is not a wildcard — only the dashboard origin is permitted to make authenticated cross-origin requests.

def configure_cors(app: FastAPI):
    app.add_middleware(
        CORSMiddleware,
        allow_origins=[
            "https://dash.occp.ai",
            "http://localhost:3000",  # development
        ],
        allow_credentials=True,
        allow_methods=["GET", "POST", "PUT", "DELETE", "PATCH"],
        allow_headers=["Authorization", "Content-Type", "X-Correlation-ID"],
        expose_headers=["X-Correlation-ID", "X-RateLimit-Remaining"],
    )

The allow_credentials=True setting enables the dashboard to send JWT tokens in the Authorization header. The expose_headers list makes the correlation ID and rate limit headers visible to the dashboard's JavaScript code, enabling client-side monitoring of API usage and request tracing.

CORS is not a security feature for the server. It is a browser enforcement mechanism. The real security is the JWT validation that happens after CORS allows the request through. CORS without authentication is decoration; authentication without CORS breaks the dashboard.

How Middleware Composition Creates Defense-in-Depth

No single middleware layer protects the application. HSTS prevents transport interception but does not stop application-layer attacks. Rate limiting prevents abuse but does not authenticate users. Request logging provides forensic capability but does not prevent incidents. The security value emerges from the composition: each layer addresses a specific threat, and the combination covers the OWASP Top 10 categories relevant to API security.

The middleware stack is also the first code that executes for every request, before routing, before authentication, and before any business logic. This means security controls cannot be bypassed by misconfigured routes or missing decorators. A new endpoint added by any developer automatically inherits HSTS, rate limiting, CORS enforcement, and structured logging without any additional configuration. This is the core principle of security as infrastructure: it is not a feature you add to endpoints, it is a property of the platform itself.

// Key Takeaways

  • Middleware registration order matters: CORS first, security headers, rate limiting, then logging
  • HSTS with includeSubDomains protects all three OCCP subdomains from transport interception
  • Sliding-window rate limiting avoids the thundering herd problem of fixed-window approaches
  • Correlation IDs link HTTP requests to policy evaluations, LLM calls, and audit entries
  • CORS is restricted to the dashboard origin only — no wildcard origins in production
  • Security middleware runs before routing, ensuring every endpoint inherits protections automatically
