Agent Boundary Enforcement: Three Levels of Capability Control

In a multi-agent system, every agent is a potential privilege escalation vector. An agent that can exceed its declared scope, bypass registration requirements, or consume unlimited resources is not governed — it is merely monitored. OCCP enforces agent boundaries at three distinct levels, each addressing a different category of boundary violation.

Why Boundaries Are Non-Negotiable

Multi-agent AI systems amplify the consequences of boundary failures. In a single-agent system, an agent that exceeds its scope produces incorrect output. In a multi-agent system, an agent that exceeds its scope can influence other agents, corrupt shared state, consume resources allocated to other tasks, and create cascading failures across the entire orchestration pipeline. The blast radius of a boundary violation scales with the number of agents in the system.

Traditional access control systems (RBAC, ABAC) operate at the user level: they control what a human user can do. Agent boundary enforcement operates at the task level: it controls what an AI agent can do within the context of a specific task execution. The difference is critical. A human user has a persistent identity with stable permissions. An agent's effective permissions should be scoped to its current task, not to a global role.

Access control governs who can act. Boundary enforcement governs what each actor can do within a specific context. For AI agents, context-scoped boundaries are more important than identity-scoped roles.

Level 1: Task Scope Validation

Declared Task Types

Every agent in OCCP declares its supported task types at registration time. A text-analysis agent declares that it handles text_analysis and summarization tasks. A code-generation agent declares code_generation and code_review. These declarations are not advisory — they are enforced at the Gate stage of the Verified Autonomy Pipeline.

# Agent registration with declared task types
{
    "agent_id": "agent_text_01",
    "name": "Text Analysis Agent",
    "task_types": ["text_analysis", "summarization"],
    "capabilities": ["llm_call", "text_processing"],
    "max_concurrent_tasks": 5,
    "token_budget_per_task": 8192
}

Gate-Stage Enforcement

When a task is submitted to the orchestrator, the Plan stage determines the task type and assigns it to an appropriate agent. Before the task reaches the Execute stage, the Gate evaluates whether the assigned agent is authorized to handle that task type. If a text_analysis task is somehow routed to a code-generation agent, the Gate rejects the assignment. This prevents both misconfiguration (an admin accidentally assigns the wrong agent) and manipulation (a compromised routing layer attempts to redirect tasks to a more privileged agent).

def validate_task_scope(agent: Agent, task: Task) -> GateResult:
    """Level 1: Verify agent is authorized for this task type."""
    if task.task_type not in agent.task_types:
        audit_log(
            "BOUNDARY_VIOLATION",
            level="task_scope",
            agent_id=agent.agent_id,
            task_type=task.task_type,
            allowed_types=agent.task_types
        )
        return GateResult(
            allowed=False,
            reason=f"Agent {agent.agent_id} not authorized "
                   f"for task type '{task.task_type}'"
        )
    return GateResult(allowed=True)

Preventing Scope Creep

Task scope validation prevents a common failure mode in multi-agent systems: scope creep. Without enforcement, an agent designed for text analysis might gradually be assigned code generation tasks "because it works well enough." Over time, the agent accumulates de facto capabilities far beyond its original design. With scope validation, each new task type must be explicitly declared and approved, making scope expansion a deliberate, audited decision rather than an accidental drift.

Level 2: Registration Checks

The Registration Requirement

Every agent that executes tasks in OCCP must be registered in the agent registry. Registration requires a unique agent identifier, a name, the declared task types (as described above), the declared capabilities, and resource quotas. Unregistered agents cannot execute tasks. There is no anonymous execution path.

Registration as a Trust Boundary

Registration is not just an inventory mechanism. It is a trust boundary. The registration process validates the agent's configuration, verifies that its declared capabilities are consistent with its task types (a text analysis agent should not need filesystem_write capability), and records the registrar's identity in the audit chain. Every registered agent has a provenance trail: who registered it, when, and with what configuration.

def validate_registration(agent_id: str) -> GateResult:
    """Level 2: Verify agent is registered and active."""
    agent = agent_registry.get(agent_id)
    if agent is None:
        audit_log(
            "BOUNDARY_VIOLATION",
            level="registration",
            agent_id=agent_id,
            reason="unregistered_agent"
        )
        return GateResult(
            allowed=False,
            reason=f"Agent {agent_id} is not registered"
        )
    if agent.status != "active":
        audit_log(
            "BOUNDARY_VIOLATION",
            level="registration",
            agent_id=agent_id,
            status=agent.status
        )
        return GateResult(
            allowed=False,
            reason=f"Agent {agent_id} status is "
                   f"'{agent.status}', expected 'active'"
        )
    return GateResult(allowed=True)

Lifecycle Management

Registered agents have lifecycle states: active, suspended, retired. Only active agents can execute tasks. An agent can be suspended immediately in response to a security incident, without deleting its configuration or audit history. Retired agents remain in the registry for audit purposes but cannot be reactivated without a new registration. The lifecycle model ensures that decommissioning an agent does not destroy its audit trail.

Level 3: Capacity Limits

Resource Quotas Per Agent

Each agent has configurable resource quotas that are enforced at runtime. The quotas cover three dimensions: token budget (maximum tokens per task and per time window), execution time (maximum wall-clock time per task), and concurrency (maximum number of simultaneous tasks).

# Capacity limit configuration
{
    "agent_id": "agent_text_01",
    "limits": {
        "max_tokens_per_task": 8192,
        "max_tokens_per_hour": 100000,
        "max_execution_seconds": 60,
        "max_concurrent_tasks": 5,
        "max_tasks_per_hour": 200
    }
}

Runtime Enforcement

Capacity limits are enforced during execution, not just at submission time. If an agent's LLM call approaches its token budget, the orchestrator truncates the request or terminates the call before the budget is exceeded. If an agent's execution time approaches its timeout, the task is cancelled with a timeout error. If an agent attempts to start a new task while at its concurrency limit, the new task is queued until a slot becomes available.

Preventing Resource Exhaustion

Without capacity limits, a single misbehaving agent can exhaust the platform's resources. A runaway agent making unbounded LLM calls can consume the entire token budget in minutes, at costs that scale linearly with the model's per-token pricing. A task stuck in an infinite loop can hold execution resources indefinitely, starving other agents. Capacity limits ensure that resource exhaustion is bounded per agent: one agent's failure cannot cascade into a platform-wide outage.

The Enforcement Chain in the Gate Stage

The three enforcement levels execute as a chain in the Gate stage. Level 2 (registration) runs first: if the agent is not registered, there is no point checking scope or capacity. Level 1 (task scope) runs second: if the agent is not authorized for the task type, capacity limits are irrelevant. Level 3 (capacity) runs last: if the agent is registered and authorized, check that it has available capacity.

def enforce_agent_boundaries(
    agent_id: str, task: Task
) -> GateResult:
    """Three-level boundary enforcement chain."""
    # Level 2: Registration (must run first)
    reg_result = validate_registration(agent_id)
    if not reg_result.allowed:
        return reg_result

    agent = agent_registry.get(agent_id)

    # Level 1: Task scope
    scope_result = validate_task_scope(agent, task)
    if not scope_result.allowed:
        return scope_result

    # Level 3: Capacity limits
    capacity_result = validate_capacity(agent, task)
    if not capacity_result.allowed:
        return capacity_result

    return GateResult(allowed=True)

The chain is fail-fast: the first failing check short-circuits the remaining checks. This is both an optimization (avoid unnecessary computation) and a security property (do not leak information about subsequent check results to an unauthorized agent). An unregistered agent learns only that it is unregistered, not whether it would have passed scope or capacity checks.

Audit Trail for Boundary Violations

Every boundary violation is recorded in the SHA-256 audit chain with the violation level, the agent identity, the attempted action, and the enforcement result. This audit trail serves three purposes. Operational: it helps administrators identify misconfigured agents. Security: it detects potential attacks where an agent repeatedly probes boundaries. Compliance: it provides evidence that boundary enforcement is active and effective for regulatory audits.

The audit entries for boundary violations are immutable and hash-chained, just like all other audit entries. An attacker who compromises an agent cannot retroactively erase the evidence of its boundary violations. The audit trail persists even if the agent is subsequently deregistered and purged from the active registry.

// Key Takeaways

  • Level 1 (Task Scope): agents can only execute task types they declared at registration, preventing scope creep
  • Level 2 (Registration): unregistered agents cannot execute any tasks, enforcing a mandatory trust boundary
  • Level 3 (Capacity): per-agent quotas on tokens, time, and concurrency prevent resource exhaustion
  • The enforcement chain runs in the Gate stage with fail-fast short-circuiting
  • All boundary violations are recorded in the immutable SHA-256 audit chain
  • Agent lifecycle states (active, suspended, retired) enable immediate incident response without losing audit history
