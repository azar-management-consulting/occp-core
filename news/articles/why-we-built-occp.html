Why We Built OCCP: The Case for a New AI Orchestration Layer

The AI ecosystem has powerful models — GPT-4, Claude, Gemini — but no governance layer between the model and production. OCCP exists to fill that gap with a verified control plane that treats every AI action as an auditable event.

The Missing Layer in the AI Stack

Consider the modern AI stack. At the bottom, you have foundation models trained by Anthropic, OpenAI, Google, and others. Above them, you have orchestration frameworks like LangChain, CrewAI, and AutoGen that wire models together into multi-step workflows. At the top, you have applications — chatbots, code generators, data analysts — that end users interact with.

Between the orchestration layer and production, there is a conspicuous void. No governance layer ensures that every LLM call is policy-checked before execution. No audit trail records what the AI did, why it did it, and what data it touched. No sandbox isolates AI-generated code from the host system. Every framework assumes that the developer will handle governance as an afterthought — a few lines of validation here, a logging statement there.

The most dangerous assumption in the AI ecosystem is that governance can be bolted on later. By the time you realize you need it, your agent has already executed thousands of unverified actions in production.

This is the gap OCCP was designed to fill. Not another orchestration framework, but a control plane — the governance infrastructure that sits between your orchestrator and your production environment.

The Risk Landscape: What Happens Without Governance

In late 2024 and early 2025, several high-profile incidents demonstrated what happens when AI agents operate without governance infrastructure. Autonomous coding agents committed credentials to public repositories. Customer-facing chatbots leaked personally identifiable information from training context. Research agents made API calls that exceeded rate limits and accumulated unexpected costs. In every case, the underlying model worked exactly as designed. The failure was in the absence of a control layer.

The Five Categories of Ungoverned Risk

  • Data Leakage — LLMs can emit PII, credentials, or proprietary data that was present in context. Without output scanning, this data reaches the end user.
  • Prompt Injection — Adversarial inputs can override system prompts. Without input validation, every user-facing AI application is a potential attack vector.
  • Unaudited Actions — When an AI agent modifies a database, sends an email, or writes a file, the action is indistinguishable from a human action unless explicitly logged.
  • Resource Exhaustion — Unconstrained agents can consume unlimited tokens, make unbounded API calls, and run indefinitely. Without resource limits, cost control is impossible.
  • Compliance Violations — The EU AI Act, GDPR, and sector-specific regulations require record-keeping, human oversight, and explainability. Without infrastructure support, compliance is a manual process.

Why Existing Frameworks Are Insufficient

LangChain is an excellent library for composing LLM calls into chains and agents. CrewAI provides elegant multi-agent collaboration primitives. AutoGen enables conversational agent patterns. These tools solve the orchestration problem, and they solve it well.

But orchestration and governance are fundamentally different concerns. Consider a concrete example:

# LangChain orchestration — what happens
chain = prompt | llm | output_parser

# What SHOULD happen
chain = prompt | policy_gate | llm | output_validator | audit_logger

The first line is how most AI applications work today. The second line is what enterprise production requires. The difference is three stages that existing frameworks delegate entirely to the developer: policy evaluation before execution, output validation after execution, and audit logging for every action.

The Configuration vs. Infrastructure Problem

Some frameworks offer callback hooks or middleware patterns that could theoretically be used for governance. The problem is that callbacks are configuration, not infrastructure. A callback can be removed with a single line change. Infrastructure cannot.

OCCP treats governance as infrastructure: it is not optional, not configurable, and not bypassable. Every task that enters the system passes through the policy gate. Every output is validated before delivery. Every action is recorded in a tamper-evident audit chain. This is the fundamental architectural difference between a governance layer and a callback hook.

The Vision: A Control Plane for AI Agents

The term "control plane" comes from networking. In a network, the data plane handles packet forwarding while the control plane handles routing decisions, access control, and policy enforcement. The data plane is fast. The control plane is smart.

OCCP applies this pattern to AI systems. The "data plane" is the LLM call itself — fast, powerful, but fundamentally unaware of organizational context. The "control plane" is OCCP — the layer that enforces policy, validates output, maintains audit trails, and ensures every AI action aligns with organizational requirements.

# The OCCP architecture concept
User Request
    |
    v
[Plan] ---> Task decomposition, risk assessment
    |
    v
[Gate] ---> Policy evaluation, PII scan, injection check
    |
    v
[Execute] ---> Sandboxed LLM call
    |
    v
[Validate] ---> Output verification, constraint check
    |
    v
[Ship] ---> Audit log, result delivery

This is the Verified Autonomy Pipeline (VAP), and it is the core of what we set out to build. Every AI action passes through five stages: Plan, Gate, Execute, Validate, Ship. Each stage has clear inputs, clear outputs, and clear failure modes. No action reaches the user without passing through all five.

Design Principles

Deny by Default

In security engineering, the principle of deny-by-default means that everything is prohibited unless explicitly allowed. OCCP applies this to AI governance: every LLM action is denied at the Gate stage unless it passes all active policies. This is the opposite of how most AI frameworks work, where everything is permitted unless the developer adds explicit checks.

Evidence Over Trust

Every claim about what an AI agent did must be backed by evidence: a hash-chained audit entry, a policy evaluation result, a validated output. OCCP never trusts that an action was safe; it verifies that it was safe, and records the proof.

Infrastructure, Not Configuration

Governance behavior is enforced by the platform, not configured by the developer. You cannot skip the policy gate. You cannot bypass output validation. You cannot disable audit logging. These are not features; they are the platform itself.

What We Set Out to Build

The initial OCCP design targeted four core components: an orchestrator engine implementing the Verified Autonomy Pipeline, a policy engine providing code-level governance with PII detection and injection guards, a CLI tool for developer-first interaction, and a Python SDK for programmatic integration.

The goal was not to replace LangChain or CrewAI. The goal was to provide the missing layer they assume someone else will build. OCCP is designed to wrap any orchestrator, any model provider, and any execution environment with a governance layer that verifies every action before it executes and audits every result after it completes.

The articles that follow in this series will detail how each component was designed and built — from the pipeline architecture to the policy engine, from the audit chain to the authentication system. This is the engineering log of building a new kind of infrastructure for a new kind of software.
