OCCP v0.6.0: Security Hardening and GDPR Compliance

Version 0.6.0 represents a philosophical shift in how OCCP treats security. Instead of patching vulnerabilities as they surface, this release embeds security into every data path — from API input validation to LLM output sanitization to regulatory compliance infrastructure.

Release Overview

OCCP v0.6.0 ships four interconnected security improvements that together create a comprehensive defensive posture. Structured Pydantic validation for all API inputs eliminates an entire class of injection and type confusion attacks. A 7-pattern output sanitization guard strips credentials and tokens from LLM responses before they reach the user. The prompt injection detection system expands from 12 to 24 patterns, covering multi-language attacks and indirect injection vectors. And GDPR-compliant legal infrastructure provides the regulatory foundation that European enterprise deployments require.

These are not independent features. They form a security pipeline where every data path — input, processing, and output — passes through at least one defensive layer.

Structured Input Validation

Prior to v0.6.0, API input validation was handled by a mix of manual type checks and FastAPI's basic query parameter parsing. This worked for the simple cases but left gaps. A malformed JSON body with extra fields would be silently accepted. A string field containing embedded control characters would pass through unexamined. Every gap was a potential attack surface.

The v0.6.0 approach replaces all manual validation with Pydantic v2 models that define the exact shape of every API request.

from pydantic import BaseModel, Field, field_validator

class TaskCreateRequest(BaseModel):
    title: str = Field(..., min_length=1, max_length=200)
    description: str = Field(..., max_length=5000)
    agent_id: str = Field(..., pattern=r'^[a-zA-Z0-9_-]+$')
    priority: int = Field(default=5, ge=1, le=10)

    @field_validator('title')
    @classmethod
    def sanitize_title(cls, v: str) -> str:
        # Strip control characters and normalize whitespace
        return ' '.join(v.split())

    model_config = ConfigDict(extra='forbid')

The extra='forbid' setting is critical. It means any request containing fields not defined in the model is rejected outright. This prevents parameter pollution attacks where an attacker adds unexpected fields hoping they will be processed by downstream code. Combined with field_validator decorators that sanitize individual fields, the validation layer ensures that only clean, well-typed data enters the application.

Validation Coverage

Every API endpoint in OCCP v0.6.0 has a corresponding Pydantic request model. This includes task creation, agent registration, policy updates, authentication requests, and audit queries. The total coverage is 100% of mutable endpoints — no POST, PUT, or PATCH request bypasses structured validation.

Output Sanitization Guard

LLMs are trained on vast datasets that include code repositories, documentation, and configuration files. When an LLM generates a response, it may inadvertently include patterns that resemble real credentials. An API key format, a database connection string, a JWT token — these patterns appear in the LLM's training data and can surface in its output.

The output sanitization guard scans every LLM response for 7 credential patterns and replaces matches with redaction markers before the response is delivered to the user. The detailed implementation is covered in a dedicated article, but the key design principle is that the guard operates on the response text after the LLM has generated it and before it enters the audit log or reaches the user.

# Output guard in the pipeline
async def execute_with_guard(task: Task) -> TaskResult:
    raw_output = await llm_provider.complete(task.prompt)

    # Sanitize before anything else sees the output
    sanitized_output = output_guard.sanitize(raw_output)

    # Audit log receives sanitized version
    await audit_log.record(task_id=task.id, output=sanitized_output)

    return TaskResult(output=sanitized_output)

The guard never modifies the original prompt or the LLM's behavior. It operates purely as a post-processing filter. This separation of concerns means the guard can be updated with new patterns without any changes to the LLM integration layer.

24-Pattern Injection Detection

The prompt injection detection system in OCCP has evolved through three phases. Version 0.1.0 shipped with 5 basic patterns that caught obvious attacks like "ignore all previous instructions." Version 0.3.0 expanded to 12 patterns, adding detection for role-play manipulation and encoding tricks. Version 0.6.0 reaches 24 patterns, covering a significantly broader attack surface.

The new patterns in v0.6.0 address three categories of attack that previous versions did not cover:

  • Multi-language attacks — Injection attempts in languages other than English that exploit models' multilingual capabilities to bypass English-only pattern matching
  • Indirect injection — Attempts to embed instructions in data that the LLM will process, rather than in the direct prompt (e.g., hiding instructions in document metadata or URL parameters)
  • Data exfiltration prompts — Instructions that attempt to make the LLM include system information, environment variables, or internal configuration in its response
Prompt injection detection is an arms race. The goal is not to catch every possible attack — that is impossible with pattern matching alone. The goal is to raise the cost of attack high enough that unsophisticated attempts fail consistently, while logging sophisticated attempts for human review.

GDPR Compliance Infrastructure

The European Union's General Data Protection Regulation imposes specific requirements on any system that processes personal data. AI orchestration platforms process user prompts, which frequently contain personal data. This makes GDPR compliance not optional but mandatory for any European deployment.

OCCP v0.6.0 introduces three GDPR-focused components:

Privacy Policy and Terms of Service

GDPR Articles 13 and 14 require that data subjects be informed about how their data is processed. OCCP now ships with comprehensive legal pages that describe data collection, processing purposes, retention periods, and data subject rights. These pages are served directly from the platform, ensuring they are always accessible and always current.

Data Processing Transparency

The platform now logs exactly what data is processed at each pipeline stage. When a task enters the Verified Autonomy Pipeline, the audit entry records which data fields were examined by the policy engine, which fields were passed to the LLM, and which fields were included in the output. This creates the processing transparency that GDPR Article 30 requires.

Consent Management Foundation

While full consent management is an ongoing effort, v0.6.0 establishes the data model and API endpoints for consent records. Each user can have multiple consent entries with specific scopes, timestamps, and revocation capabilities. This foundation enables deployments to implement consent flows appropriate to their use case.

The Security Mindset Shift

The most significant change in v0.6.0 is not any single feature but the architectural principle it establishes: security is not a feature to be added; it is a property of the architecture. Every data path has a defensive layer. Every input is validated. Every output is sanitized. Every action is logged.

This principle will guide all future OCCP development. Features that cannot be built securely will not be built at all. Security review is a prerequisite for merging, not a follow-up task. And the test suite now includes dedicated security test cases that verify each defensive layer independently.

Migration Notes

For users upgrading from v0.5.0, the structured validation may reject requests that were previously accepted. Any API client sending extra fields in request bodies will need to be updated. The output sanitization guard is enabled by default and cannot be disabled in the Community Edition. The injection detection expansion is transparent — it may flag prompts that were not flagged in previous versions, but the detection behavior is additive and non-breaking.
