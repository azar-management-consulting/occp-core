OCCP v0.7.0: ORM Migration, RBAC, and Sandbox Isolation

Version 0.7.0 is the release where OCCP stops being a prototype and starts being enterprise infrastructure. Three foundational changes — a proper ORM, role-based access control, and sandboxed code execution — establish the patterns that every subsequent feature will build upon.

The Enterprise Readiness Threshold

There is a moment in every platform's evolution where the gap between "it works" and "it is production-ready" becomes the primary engineering constraint. For OCCP, that moment arrived at v0.6.0. The platform had a working pipeline, solid security guards, and a functional dashboard. But the data layer was raw SQL strings, access control was binary (authenticated or not), and code execution ran in the same process as the platform itself.

Enterprise deployments require three properties that the pre-v0.7.0 architecture could not provide. First, schema governance — the ability to evolve the database schema safely across versions without manual migration scripts. Second, granular authorization — different users need different access levels, and "admin or not" is insufficient for organizations with multiple teams. Third, execution isolation — AI-generated code must run in an environment where it cannot affect the host system, regardless of what the code attempts.

Version 0.7.0 addresses all three with SQLAlchemy 2.0, Casbin RBAC, and a three-tier sandbox architecture.

Pillar 1: SQLAlchemy 2.0 Async ORM

The migration from raw SQL to SQLAlchemy 2.0 mapped models was the largest single refactoring in OCCP's history. Every database interaction in the codebase — task CRUD, audit logging, user management, agent registration — was rewritten to use ORM models and async sessions.

# Before (v0.6.0): Raw SQL
async def get_task(db, task_id: str):
    row = await db.execute(
        "SELECT * FROM tasks WHERE id = ?", (task_id,)
    )
    return dict(row) if row else None

# After (v0.7.0): SQLAlchemy 2.0 mapped model
async def get_task(session: AsyncSession, task_id: str) -> Task | None:
    result = await session.execute(
        select(Task).where(Task.id == task_id)
    )
    return result.scalar_one_or_none()

The ORM migration provides three capabilities that raw SQL could not. First, type safety: every database column maps to a typed Python attribute, and the IDE catches type mismatches at development time rather than at runtime. Second, relationship management: eager and lazy loading patterns replace manual JOIN queries, reducing boilerplate and preventing N+1 query problems. Third, backend portability: the same ORM models work with SQLite (development) and PostgreSQL (production), with the ORM handling dialect differences automatically.

The migration also introduced Alembic for schema migrations. Every schema change is now a versioned migration file that can be applied forward or rolled back. This eliminates the "apply this SQL script manually" deployment step that was error-prone and irreversible.

Pillar 2: Casbin RBAC

Before v0.7.0, OCCP's access control was binary: a valid JWT token granted full access to every endpoint. This was acceptable during early development but inadequate for any deployment with more than one person. A junior operator should not have the same access as a system administrator. A read-only dashboard viewer should not be able to modify agent configurations.

OCCP v0.7.0 implements role-based access control using Casbin, a policy engine that evaluates access decisions based on (subject, object, action) triples. Four roles form a hierarchical permission model:

  • viewer — Read-only access. Can view tasks, agents, audit logs, and dashboard metrics. Cannot create, modify, or delete any resource.
  • operator — Inherits viewer permissions. Can create and run tasks, trigger pipeline executions, and view execution results. Cannot modify agent configurations or platform settings.
  • admin — Inherits operator permissions. Can register and configure agents, modify policies, manage user accounts, and access advanced settings.
  • system_admin — Full access. Can modify platform configuration, manage RBAC policies, access security-sensitive endpoints, and perform destructive operations like data purging.
# Casbin policy model (model.conf)
[request_definition]
r = sub, obj, act

[policy_definition]
p = sub, obj, act

[role_definition]
g = _, _

[policy_effect]
e = some(where (p.eft == allow))

[matchers]
m = g(r.sub, p.sub) && r.obj == p.obj && r.act == p.act

The Casbin integration sits in the middleware stack, evaluating permissions before every API endpoint handler executes. The user's role is extracted from their JWT token claims, and the requested endpoint is mapped to an (object, action) pair. If the Casbin policy does not grant the user's role access to that (object, action), the request is rejected with a 403 Forbidden response.

Pillar 3: Sandbox Isolation

When an AI agent generates code and requests execution, that code runs on the platform's host system. Without isolation, the generated code has the same access as the OCCP process itself: filesystem, network, environment variables, other running processes. A malicious or buggy code generation could read secrets, modify files, or consume unbounded resources.

OCCP v0.7.0 introduces a three-tier sandbox architecture that provides the strongest available isolation on the host system:

  • Tier 1: nsjail — Linux namespace isolation with seccomp-bpf system call filtering. The strongest option, providing kernel-level isolation of the execution environment. Available on Linux systems with nsjail installed.
  • Tier 2: bubblewrap — User-namespace-based sandboxing with filesystem and network restrictions. A lighter-weight alternative when nsjail is not available. Common on Linux desktop distributions.
  • Tier 3: subprocess with restrictions — A standard subprocess with resource limits (CPU time, memory) applied via resource.setrlimit. The weakest option, used as a fallback when neither nsjail nor bubblewrap is available.

The sandbox manager probes the host system at startup to determine which tier is available. It configures the highest-tier sandbox that the system supports and logs the result. Operators can verify the isolation level through the /api/v1/system/sandbox endpoint and the dashboard's system status panel.

The sandbox does not trust the code it executes. It assumes every code execution attempt is potentially hostile. Resource limits, network blocking, and filesystem restrictions are not optional safeguards — they are the execution environment's default state.

Test Suite Expansion: 313 Tests

The three pillars of v0.7.0 required extensive test coverage. The ORM migration needed tests for every model, every relationship, and every query. The RBAC implementation needed tests for every role-endpoint combination. The sandbox needed tests for each isolation tier and the fallback logic between them.

The v0.7.0 test suite contains 313 tests across multiple categories:

Security Middleware Stack

The v0.7.0 middleware stack processes every HTTP request through a defined sequence of security checks:

  1. HSTS enforcement — Strict-Transport-Security headers ensure HTTPS-only access.
  2. Rate limiting — Per-IP and per-user rate limits prevent abuse.
  3. JWT validation — Token signature and expiry verification.
  4. RBAC evaluation — Casbin policy check against the user's role and the requested endpoint.
  5. Request logging — Structured log entry for the request, excluding sensitive headers.

Each middleware layer can reject the request independently. A rate-limited request never reaches JWT validation. An unauthenticated request never reaches RBAC evaluation. This layered approach ensures that the most common rejection reasons are handled at the lowest cost.

Migration Path

Upgrading from v0.6.0 to v0.7.0 requires running Alembic migrations to update the database schema. The migration adds columns for RBAC role assignments, creates the Casbin policy tables, and restructures the task table to match the new ORM model. Existing data is preserved and transformed in place. A pre-migration backup is strongly recommended but the migrations are designed to be non-destructive.

All existing users are assigned the admin role during migration, preserving their current access level. Operators should review role assignments after migration and downgrade users who do not require full administrative access.

  • Casbin RBAC implements a 4-role hierarchy (viewer, operator, admin, system_admin) with per-endpoint authorization
  • Three-tier sandbox (nsjail, bubblewrap, subprocess) provides the strongest available isolation with automatic fallback
  • 313 tests across 10 categories ensure comprehensive coverage of all new systems
  • Security middleware stack processes HSTS, rate limiting, JWT, RBAC, and logging in sequence for every request
  • Migration from v0.6.0 is non-destructive with automatic role assignment for existing users
