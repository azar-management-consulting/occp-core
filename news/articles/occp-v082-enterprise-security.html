OCCP v0.8.2: Enterprise-Grade Token Security

v0.8.0 made OCCP accessible. v0.8.2 makes it enterprise-ready. This release adds AES-256-GCM envelope encryption for LLM API tokens, supply chain verification for MCP packages and agent extensions, three-level agent boundary enforcement, and concrete implementations of EU AI Act Articles 12, 14, and 19. Security is no longer a feature — it is the release.

AES-256-GCM Envelope Encryption

OCCP stores LLM API tokens — Anthropic API keys, OpenAI keys, Google API keys — that represent direct access to expensive cloud resources. Before v0.8.2, these tokens were stored in the database with application-level obfuscation. v0.8.2 replaces this with proper cryptographic protection: AES-256-GCM authenticated encryption with HKDF key derivation.

The Encryption Architecture

The system uses envelope encryption, a pattern where each secret is encrypted with a unique data encryption key (DEK), and the DEK itself is encrypted with a key encryption key (KEK). The KEK is derived from a master secret using HKDF (HMAC-based Key Derivation Function) with a per-deployment salt.

# Envelope encryption flow
Master Secret (environment variable)
    |
    v
HKDF-SHA256 (salt=deployment_id, info="occp-token-encryption")
    |
    v
Key Encryption Key (KEK)
    |
    v
AES-256-GCM encrypt(DEK) -> encrypted_dek
    |
    v
DEK + AES-256-GCM encrypt(token, nonce) -> encrypted_token

# Stored in database:
# { encrypted_dek, encrypted_token, nonce, salt, version }

The two-layer approach provides a critical operational benefit: key rotation. When the master secret is rotated, only the DEK wrappers need to be re-encrypted. The actual tokens, encrypted under their individual DEKs, remain untouched. This means key rotation does not require decrypting and re-encrypting every stored secret — it only requires re-wrapping the DEKs, which is a lightweight operation that can execute without downtime.

GCM Authentication

AES-256-GCM provides authenticated encryption, meaning it produces both ciphertext and an authentication tag. The authentication tag detects any modification to the ciphertext, including truncation, bit-flipping, and reordering attacks. If the tag verification fails during decryption, the operation is rejected entirely — no partial plaintext is ever returned. This prevents a category of attacks where an adversary modifies the encrypted token to produce a valid but different plaintext.

Each encryption operation uses a unique 96-bit nonce generated from a cryptographically secure random source. The nonce is stored alongside the ciphertext. Nonce reuse with the same key would compromise GCM's security guarantees, which is why each token gets its own DEK — even if a nonce were accidentally reused across two tokens, they would be encrypted under different keys.

The difference between encryption and authenticated encryption is the difference between "nobody can read this" and "nobody can read or tamper with this." In a system that stores API tokens, tamper detection is as important as confidentiality.

Supply Chain Verification

v0.8.0 introduced MCP server connections and agent extensions. v0.8.2 adds verification that these external components are safe to load. The supply chain verification system operates at three levels: package allowlisting, static analysis, and fail-closed enforcement.

Package Allowlisting

MCP server packages must appear on a curated allowlist before they can be connected to the platform. The allowlist contains package names, version ranges, and cryptographic hashes of the approved packages. When an administrator attempts to connect a new MCP server, the system verifies the server's package identity against the allowlist. Packages not on the allowlist are rejected with a clear message indicating that the package requires explicit approval.

Static Analysis

Agent extensions — custom Python modules that add capabilities to agents — undergo static analysis before loading. The analyzer checks for known-dangerous patterns: dynamic code execution (eval, exec, __import__), network access outside approved domains, filesystem access outside approved directories, and attempts to modify the OCCP runtime. Extensions that fail static analysis are quarantined and cannot be loaded until the issues are resolved.

# Supply chain verification — three levels
class SupplyChainVerifier:
    async def verify_mcp_package(self, package: MCPPackage) -> VerifyResult:
        # Level 1: Allowlist check
        if not self.allowlist.contains(package.name, package.version):
            return VerifyResult(
                approved=False,
                reason=f"Package '{package.name}@{package.version}' not in allowlist"
            )

        # Level 2: Hash verification
        computed_hash = hashlib.sha256(package.binary).hexdigest()
        if computed_hash != self.allowlist.get_hash(package.name, package.version):
            return VerifyResult(
                approved=False,
                reason="Package hash mismatch — possible tampering"
            )

        # Level 3: Static analysis
        analysis = await self.analyzer.scan(package)
        if analysis.has_violations:
            return VerifyResult(
                approved=False,
                reason=f"Static analysis violations: {analysis.violations}"
            )

        return VerifyResult(approved=True)

Fail-Closed Enforcement

The verification system is fail-closed: if the allowlist is unavailable, if the hash check encounters an error, or if the static analyzer crashes, the package is rejected. This is the opposite of fail-open behavior where errors result in approval. Fail-closed means a verification infrastructure outage prevents new packages from being installed, which is an acceptable trade-off: it is better to delay a new MCP server connection than to load an unverified package.

Agent Boundary Enforcement

v0.8.2 introduces three-level agent boundary enforcement that prevents agents from exceeding their declared capabilities:

  1. Task Scope Validation — When a task is assigned to an agent, the system validates that the task's requirements fall within the agent's registered capabilities. An agent registered for "code analysis" cannot be assigned a "database administration" task. The scope validation runs at task creation time, before any pipeline execution begins.
  2. Registration Checks — Agent capabilities are declared at registration time and can only be modified by platform administrators. An agent cannot self-modify its capability set. If an agent's LLM attempts to request additional tool access or modify its own configuration, the request is rejected at the API level with an authorization error.
  3. Capacity Limits — Each agent has configurable limits: maximum concurrent tasks, maximum tokens per task, maximum tool calls per execution, and maximum execution time. These limits are enforced at the pipeline level — exceeding any limit terminates the current execution and records the violation in the audit trail.

The three levels create a defense-in-depth model. Task scope prevents misassignment. Registration checks prevent self-escalation. Capacity limits prevent resource exhaustion. All three must be satisfied for every execution, and a failure at any level is treated as a security event with full audit logging.

EU AI Act Compliance

v0.8.2 implements three specific articles from the EU AI Act that are relevant to AI orchestration platforms:

Article 12: Record-Keeping

Article 12 requires that high-risk AI systems maintain logs that enable tracing of the system's operation. OCCP's SHA-256 hash-chained audit trail satisfies this requirement by recording every pipeline stage execution with inputs, outputs, timestamps, durations, and the identity of the agent and user involved. The hash chain ensures that records cannot be modified after creation — any alteration breaks the chain and is detectable.

Article 14: Human Oversight

Article 14 requires that high-risk AI systems be designed to allow effective human oversight. OCCP implements this through the Gate stage of the Verified Autonomy Pipeline: every plan is evaluated by the policy engine before execution, and configurable approval workflows require human sign-off for high-risk tasks. The dashboard provides real-time visibility into pipeline execution, and the audit trail provides post-execution review capability.

Article 19: Log Retention

Article 19 requires automatic log generation and retention for a period appropriate to the system's purpose. OCCP v0.8.2 implements automated audit retention enforcement at application startup. When the server starts, it calculates the retention period (default: 2555 days / ~7 years) and marks expired entries for archival. The retention period is configurable through the settings system but enforces a minimum that satisfies the EU AI Act's requirements.

# Startup audit retention enforcement
class AuditRetentionEnforcer:
    async def enforce_at_startup(self):
        retention_days = await self.settings.resolve("audit.retention_days")
        cutoff_date = datetime.utcnow() - timedelta(days=retention_days)

        # Archive entries older than retention period
        archived = await self.audit_repo.archive_before(cutoff_date)
        logger.info(f"Audit retention enforced: {archived} entries archived, "
                     f"retention={retention_days}d, cutoff={cutoff_date.isoformat()}")

Upgrade Path from v0.8.0

Upgrading to v0.8.2 requires three steps. First, set the OCCP_MASTER_SECRET environment variable — this is the master key for envelope encryption. Second, run alembic upgrade head to apply the new migration that adds encryption metadata columns to the settings table. Third, run occp migrate-tokens — a one-time command that encrypts existing plaintext tokens with the new envelope encryption scheme. After migration, the plaintext tokens are securely overwritten in the database.

The migration is backward-compatible: if OCCP_MASTER_SECRET is not set, the system falls back to the v0.8.0 obfuscation behavior with a warning log. This ensures that development environments and test suites are not broken by the upgrade, while production deployments get the full encryption protection.
