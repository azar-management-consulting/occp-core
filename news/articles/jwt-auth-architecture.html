JWT Authentication Architecture for AI Control Planes

OCCP's authentication layer uses JSON Web Tokens to identify users and embed role claims in every API request. This article details the JWT design decisions: why tokens over sessions, how the token structure supports RBAC, bcrypt password hashing configuration, token expiration strategy, and how the auth middleware validates requests at the gateway level.

Why JWT Over Sessions

Session-based authentication stores user state on the server: a session ID in a cookie maps to a server-side record containing user identity and permissions. JWT authentication stores user state in the token itself: the token payload contains the user identity and role, signed by the server's secret key.

For traditional web applications, sessions work well. But AI control planes have characteristics that favor JWT:

SDK-Native Authentication

OCCP is accessed through three channels: the CLI, the Python SDK, and the TypeScript SDK. Sessions require cookies, which are a browser mechanism. The CLI and Python SDK do not run in browsers. They can set cookie headers manually, but this feels unnatural and requires cookie jar management (persistence, domain scoping, expiration tracking). JWTs are bearer tokens: set the Authorization header and the request is authenticated. Every HTTP client supports this pattern natively.

# Python SDK — JWT authentication is natural
import urllib.request

token = "eyJhbGciOiJIUzI1NiIs..."
request = urllib.request.Request(
    url="https://api.occp.ai/api/v1/tasks",
    headers={"Authorization": f"Bearer {token}"},
    data=json.dumps(task_data).encode(),
    method="POST"
)

Stateless Verification

Session validation requires a server-side lookup: the session store (database, Redis, in-memory cache) must be queried on every request. For an AI platform processing long-running pipeline executions, the session lookup adds a database round-trip to every API call. JWT validation is a local operation: the server verifies the token's signature against its secret key and checks the expiration claim. No database query is required. This makes JWT validation both faster and more resilient — a database outage does not prevent authentication.

Horizontal Scalability

When multiple API server instances handle requests behind a load balancer, session-based authentication requires a shared session store. Without it, a request authenticated on instance A will be unauthenticated on instance B. JWT tokens are verified independently by each instance using the shared secret key, requiring no inter-instance communication. This simplifies horizontal scaling for production deployments.

Token Structure

A JWT consists of three parts: a header specifying the algorithm, a payload containing claims, and a signature verifying integrity. OCCP's token structure includes the following claims:

// JWT Header
{
  "alg": "HS256",
  "typ": "JWT"
}

// JWT Payload
{
  "sub": "usr_7f3a2e1d",          // Subject: user ID
  "role": "operator",              // User role for RBAC
  "exp": 1733443200,               // Expiration: Unix timestamp
  "iat": 1733356800,               // Issued at: Unix timestamp
  "iss": "occp-api"                // Issuer: OCCP API server
}

// JWT Signature
HMACSHA256(
  base64UrlEncode(header) + "." + base64UrlEncode(payload),
  secret_key
)

The sub Claim

The sub (subject) claim contains the user's unique identifier, prefixed with usr_ for type disambiguation. This ID maps to the user record in the database and is used for audit trail attribution. When an AI task is submitted through the API, the audit entry records the sub claim as the actor, creating a direct link between the authenticated user and the actions they initiated.

The role Claim

The role claim embeds the user's authorization level directly in the token. In v0.2.0, OCCP defines two roles: admin (full platform access including user management) and operator (task submission, status queries, and audit trail access). The role claim is the foundation for the Casbin RBAC system introduced in v0.7.0, which expands the role hierarchy to four levels: admin, operator, viewer, and auditor.

Embedding the role in the token means that authorization checks do not require a database lookup. The middleware extracts the role from the token payload and compares it against the endpoint's required role. This is particularly important for the audit endpoint, which may be called frequently by monitoring systems — each call requires authentication and authorization, and database lookups for every call would create unnecessary load.

The exp and iat Claims

Token expiration (exp) is set to 24 hours after issuance (iat). This balance reflects the operational reality of AI platform usage: developers and operators interact with the platform throughout the workday, and requiring re-authentication every hour would create friction. The 24-hour window allows a full working day of uninterrupted access while limiting the exposure window if a token is compromised.

Token expiry is a trade-off between security and usability. Shorter expiry reduces the window of compromise but increases authentication friction. For an AI control plane, where operators need sustained access during long-running tasks, 24 hours provides the right balance.

bcrypt Password Hashing

User passwords are stored as bcrypt hashes with a work factor of 12 (2^12 = 4,096 iterations). bcrypt was chosen over alternatives for specific cryptographic properties:

  • Adaptive cost — The work factor can be increased as hardware improves, maintaining resistance against brute-force attacks without changing the algorithm
  • Built-in salt — bcrypt generates and stores a random salt with each hash, preventing rainbow table attacks and ensuring identical passwords produce different hashes
  • Deliberate slowness — At 12 rounds, bcrypt takes approximately 250ms per hash on modern hardware. This is imperceptible during login but makes brute-force attacks (millions of attempts) computationally prohibitive
import bcrypt

class PasswordService:
    ROUNDS = 12

    @staticmethod
    def hash(password: str) -> str:
        """Hash a password with bcrypt.
        Returns the hash as a string for database storage."""
        salt = bcrypt.gensalt(rounds=PasswordService.ROUNDS)
        return bcrypt.hashpw(password.encode("utf-8"), salt).decode("utf-8")

    @staticmethod
    def verify(password: str, stored_hash: str) -> bool:
        """Verify a password against a stored bcrypt hash."""
        return bcrypt.checkpw(
            password.encode("utf-8"),
            stored_hash.encode("utf-8")
        )

The password is never stored, logged, or transmitted in plaintext after initial receipt. The login endpoint receives the password over HTTPS, verifies it against the stored hash, and discards it. The JWT token returned to the client does not contain the password or hash.

Auth Middleware Architecture

The auth middleware intercepts every API request before it reaches the endpoint handler. The middleware performs four steps in sequence:

  1. Header extraction — Extract the Authorization header and verify it follows the Bearer {token} format. Missing or malformed headers produce a 401 response.
  2. Signature verification — Decode the JWT and verify its HMAC-SHA256 signature against the server's secret key. Invalid signatures (indicating a forged or corrupted token) produce a 401 response.
  3. Expiration check — Verify that the current time is before the token's exp claim. Expired tokens produce a 401 response with a message indicating token expiry.
  4. Context injection — Extract the sub and role claims and inject them into the request context, making them available to endpoint handlers for authorization checks and audit attribution.
from fastapi import Request, HTTPException
from fastapi.security import HTTPBearer, HTTPAuthorizationCredentials
import jwt

class JWTMiddleware:
    def __init__(self, secret_key: str):
        self.secret_key = secret_key
        self.scheme = HTTPBearer()

    async def authenticate(self, request: Request) -> dict:
        credentials: HTTPAuthorizationCredentials = await self.scheme(request)
        token = credentials.credentials

        try:
            payload = jwt.decode(
                token,
                self.secret_key,
                algorithms=["HS256"],
                options={"require": ["sub", "role", "exp", "iat"]}
            )
        except jwt.ExpiredSignatureError:
            raise HTTPException(status_code=401, detail="Token has expired")
        except jwt.InvalidTokenError:
            raise HTTPException(status_code=401, detail="Invalid token")

        return {
            "user_id": payload["sub"],
            "role": payload["role"],
        }

Public Endpoints

Two endpoints are exempt from authentication: the healthcheck endpoint (/api/v1/health), which must be accessible to Docker healthcheck probes and load balancer health monitors, and the login endpoint (/api/v1/auth/login), which is how unauthenticated users obtain a token. All other endpoints require a valid JWT.

Role-Based Access Control Foundation

The role claim in the JWT payload provides the foundation for role-based access control. In v0.2.0, the implementation is simple: endpoint handlers check the role claim against a required role. In later versions, this evolves into a full Casbin RBAC system with policy definitions, but the JWT role claim remains the source of identity.

# Role check in endpoint handler
@app.post("/api/v1/users")
async def create_user(
    user_data: UserCreate,
    auth: dict = Depends(jwt_middleware.authenticate)
):
    if auth["role"] != "admin":
        raise HTTPException(status_code=403, detail="Admin role required")
    # ... create user logic

This pattern — extract role from token, check against required role, deny if insufficient — is repeated at every protected endpoint. The Casbin integration in v0.7.0 replaces these inline checks with policy-based evaluation, but the JWT token remains the identity source. The token tells OCCP who is making the request and what role they hold; the authorization system decides what that role is allowed to do.

Security Considerations

JWT authentication has known limitations that OCCP addresses through complementary measures. Token revocation is not natively supported by JWT (the token is valid until it expires, even if the user's account is disabled). OCCP mitigates this with short expiration windows and a server-side token blocklist for emergency revocations. Token theft (if an attacker obtains a valid token) is mitigated by HTTPS enforcement (tokens are never transmitted over unencrypted connections) and the 24-hour expiration window. Secret key compromise (if the signing key is leaked) would allow token forgery; this is mitigated by the key rotation architecture introduced in v0.8.2.

// Key Takeaways

  • JWT chosen over sessions for SDK compatibility, stateless verification, and horizontal scalability
  • Token payload includes sub (user ID), role (RBAC), exp (24h expiry), and iat (issuance time)
  • bcrypt with 12 rounds for password hashing — ~250ms per hash, resistant to brute-force attacks
  • Auth middleware performs 4-step validation: header extraction, signature check, expiry check, context injection
  • Role claim in JWT is the foundation for RBAC — no database lookup needed for authorization
  • Known JWT limitations addressed with token blocklist, HTTPS enforcement, and key rotation support
